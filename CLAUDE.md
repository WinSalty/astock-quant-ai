# astock-quant-ai 项目约束

A 股打板量化项目（信号侧 `stock-ah-premium-ai` + 执行侧 `astock-quant-ai/qmt_strategy`）。本文是本项目级约束，与工作区根 `CLAUDE.md`/`AGENTS.md` 叠加生效。

## 文档归类规则（活文档在项目 `doc/`、归档在工作区 `doc/`，必须遵守）

项目文档分五类，各有固定位置与生命周期；**新增/整理文档一律按此归位**。①~④ 为活文档，常驻**项目** `doc/`（即 `coding/astock-quant-ai/doc/`）；⑤ 归档在**工作区级** `doc/`（即 `/Users/salty/codeProject/ai/doc/已归档-完成不再维护/`，在本项目 git 仓库之外）：

1. **`doc/README.md` — 项目现状口径（单一入口）。** 只记**当前**的交易逻辑、架构、部署模块设计、资金/风控口径。常驻 `doc/` 根，随代码演进更新；**不堆历史、不混评审/修复状态、不混待办**。
2. **`doc/待办与上线验证清单.md` — 待办（活文档）。** 记「还需处理的事」+「只能真实上生产/目标机实测才能验证的事」+「上线部署步骤/注意事项」。常驻 `doc/` 根，处理完一项划掉一项。
3. **`doc/评审与修复状态概要.md` — 评审与修复状态概要（持久状态记录）。** 只记**高层概要**：做了哪几轮评审、修了什么、当前状态，并**指向** ④ 的详细报告/台账。常驻 `doc/` 根、**不随处理文档归档**（处理文档归档后它仍作历史状态留存、随后续评审轮次续写）。
4. **评审/修复处理文档（如 `doc/08-*评审报告`、`doc/09-*修复台账`、`doc/10-*分析建议`）。** 处理**进行中**时放**项目** `doc/` 根（与 README 同级）；**全部处理完后整体移入工作区级归档目录 `/Users/salty/codeProject/ai/doc/已归档-完成不再维护/`**（高层状态由 ③ 承接，不丢）。
5. **`/Users/salty/codeProject/ai/doc/已归档-完成不再维护/` — 归档（完成存档·不再看）。** 此目录在**工作区根** `doc/` 下、**不在本项目 `doc/`、也不在本项目 git 仓库内**。放「已经处理完、不再维护」的历史件：过去的开发设计、开发计划、可行性分析、方案说明、被新版取代的旧评审稿等。**前提是对应工作都已落地/处理过**；放进去即视为**丢弃不再管、日常无需再看**（目录名已体现）。

> 生产部署/运维手册不在本仓库 `doc/` 下，在工作区 `/Users/salty/codeProject/ai/doc/Qmt生产服务器.txt`（含服务器密码/token，**严禁 commit 进任一仓库、严禁明文外泄**）；README 与待办清单只以**路径指针**引用它，不把其内容复制进仓库。

### 操作口径

- **归档 = 丢弃不再管。** 只有确认「都处理过了」才往归档目录放；归档件不随代码更新、可能与现状不一致，仅作历史追溯。
- 新写的设计/计划/评审类**处理文档**（④）处理完后**归档**，不要长期堆在 `doc/` 根；其**高层结论**同步进 ③ 状态概要（避免归档后状态信息丢失）。
- README 只写**现状口径**；「还没做的」「上线注意事项」进待办清单，「评审/修复状态」进状态概要，三者不互相混。
- **项目 `doc/` 内部**的移动/重命名用 `git mv` **保留历史**；**归档到工作区级 `已归档-完成不再维护/`** 属于移出本项目仓库（工作区根非 git 仓库，无法跨仓 `git mv`），改用普通 `mv` 搬运，并在项目仓库内提交该文件的删除（其历史仍可在项目仓库的提交记录中追溯）。
- 移动/重排章节后**同步修好 README/待办/概要 等处指向被移动文件的链接与「§N」章节号引用**；归档件的链接要改为指向工作区级 `/Users/salty/codeProject/ai/doc/已归档-完成不再维护/…`（绝对路径或对应相对路径均可，不留断链、不留错号）。
- 归档目录名要**始终体现「不再看」**（当前为工作区根 `doc/` 下的 `已归档-完成不再维护/`）；新建归档目录沿用此命名取向。

## 缺陷修复与数据缺测处理口径（评审后修复必须遵守）

评审发现的问题进入修复时，按问题性质分流处理，不得一刀切；尤其下列口径是本项目的硬约定：

- **跨两仓协同 / 涉及契约的问题**：先排查信号侧能否经 Tushare 正常取到所需数据，再据下列口径定处置，不要凭空假设取数能力。
- **数据偶尔不全（Tushare 能取到、仅偶发缺）**：一律约定一个**特殊值 / null sentinel**；信号侧产出该 sentinel 后，执行侧消费到即对该票**放弃买入**；若该票**已在持仓则一律按卖出处理**（受 A 股 T+1 物理约束，仅对已过 T+1 的可卖单元生效）。
- **信号侧经 Tushare 根本拿不到的数据**：先仔细研究是否有其它可用取数渠道，整理候选后**与负责人商量再定如何处理**，不得擅自默认处理或跳过。
- **契约 / 跨两仓口径类问题**：思考并给出**最合理方案（含取舍）交负责人决策**，**绝不给「默认处理 / 跳过」**——这是真实要部署上线的代码，错配代价是真金资金。
- **回测类问题**：本项目基于 LLM 选股、回测样本有限，回测口径问题**单独整理进回测方向问题记录文档**（如 `doc/28-回测方向问题记录.md`）；代码层面修复**优先级低、可暂不处理**，不挤占主链修复排期。
