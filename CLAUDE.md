# astock-quant-ai 项目约束

A 股打板量化项目（信号侧 `stock-ah-premium-ai` + 执行侧 `astock-quant-ai/qmt_strategy`）。本文是本项目级约束，与工作区根 `CLAUDE.md`/`AGENTS.md` 叠加生效。

## 文档归类规则（`doc/` 下，必须遵守）

项目文档分四类，各有固定位置与生命周期；**新增/整理文档一律按此归位**：

1. **`doc/README.md` — 项目现状（单一入口）。** 只记**当前**的交易逻辑、架构、部署模块设计。常驻 `doc/` 根，随代码演进更新；**不堆历史**。
2. **`doc/待办与上线验证清单.md` — 待办（活文档）。** 记「还需处理的事」+「只能真实上生产/目标机实测才能验证的事」。常驻 `doc/` 根，处理完一项划掉一项。
3. **最新评审/修复处理文档（如 `doc/08-*评审报告`、`doc/09-*修复台账`）。** 处理**进行中**时放 `doc/` 根（与 README 同级）；**全部处理完后整体移入归档目录**。
4. **`doc/已归档-完成不再维护/` — 归档（完成存档·不再看）。** 放「已经处理完、不再维护」的历史件：过去的开发设计、开发计划、可行性分析、方案说明、被新版取代的旧评审稿等。**前提是对应工作都已落地/处理过**；放进去即视为**丢弃不再管、日常无需再看**（目录名已体现）。

### 操作口径

- **归档 = 丢弃不再管。** 只有确认「都处理过了」才往归档目录放；归档件不随代码更新、可能与现状不一致，仅作历史追溯。
- 新写的设计/计划/评审类文档，处理完后**归档**，不要长期堆在 `doc/` 根。
- README 只写**现状**；「还没做的」一律进待办清单，不要混进 README。
- 移动文档用 `git mv` **保留历史**；移动后**同步修好 README 等处的相对链接**（不留断链）。
- 归档目录名要**始终体现「不再看」**（当前为 `已归档-完成不再维护/`）；新建归档目录沿用此命名取向。

## 其它

- 文档撰写不含开发周期/工时预估（沿用工作区根约束）；按阶段+验收+依赖顺序组织。
- 触碰代码须补中文注释（业务意图/边界/重跑口径），沿用工作区根 `AGENTS.md`。
- 提交信息末尾带 model 后缀；正式交付须 commit + push（两仓：执行侧 main、信号侧 feature 分支）。
