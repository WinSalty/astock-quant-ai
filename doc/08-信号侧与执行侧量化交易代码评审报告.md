# 信号侧 + 执行侧 量化交易代码评审报告

> 评审对象：A 股打板量化闭环的两侧代码
> - **信号侧** `stock-ah-premium-ai/backend`（Linux / FastAPI / MySQL）—— 多阶段 LLM 选股、龙头增强、回测、供数契约
> - **执行侧** `astock-quant-ai/qmt_strategy`（Windows / miniQMT）—— 连接守护、集合竞价、建仓五策略、持仓/卖出、风控、回流、对账
>
> 评审日期：2026-06-14　|　评审方式：15 路模块/横切智能体逐行读源码 + 对每条 critical/high/medium 发现做双视角对抗式验证（重读源码核验），共产出 166 条发现；其中 **12 条 critical、24 条 high、63 条 medium 经验证确认**。最关键的 3 条"闭环不可用"发现已由评审者再次手工 grep/读码复核。
>
> 性质：本项目**真实投入资金**，本报告按"宁可多报关键资金风险、不放过"的口径撰写。结论可能与各模块设计文档/README 的"已落地、单测通过"自评存在出入——单测通过不等于端到端接线正确，本报告以**当前源码事实**为准。

## 0. 一句话结论

**当前两侧代码工程质量很高（强类型契约、Decimal 价位、三层幂等设计、异步写不阻塞交易、大量中文注释与防御注释），但端到端闭环存在多处相互独立的致命断裂：执行侧在当前接线下既不会按设计开仓（所有计划票被路由到 SKIP），也不会按设计卖出（持仓状态机从未被买入成交回写）；项目宣称的多层资金护栏（账户熔断、单票止损、情绪退潮禁开仓、go/no-go 上线闸门）大量是"声明已落地但实为死代码/未接线"。在 P0 问题全部修复并补端到端联调前，严禁接入真实资金。**

良性的一面：这些问题**绝大多数是"接线/口径"层面，而非架构层面**。锁定的契约层、唯一下单点、台账幂等、风控纯函数、回测撮合铁律等底层设计都是对的，缺的是把它们正确串起来并做端到端验证。

## 3. P0 致命缺陷（上线前必须修复）

> 以下为去重后的 P0 主题。每条标注 [模块]、验证共识、关键 `文件:行`。

### P0-A　闭环断裂：执行侧既不会按设计开仓、也不会按设计卖出

**A1. `strategy_family` 取值域 EN/CN 失配 → 所有计划票一律路由到 SKIP，永不开新仓**　[x-contract，confirmed×2，已手工复核]
- 信号侧 `limit_up_leader_scoring_service.py:369/371/373` 产出英文枚举 `"DABAN"/"DIXI"/"BANLU"`，`limit_up_push_service.py:630` 原样落表、原样下发。
- 执行侧 `entry/entry_router.py:200/204/208` 用**中文关键词包含匹配**（`"低吸"/"打板"/"连板"/"板"/"龙"`），对 `"DABAN"/"BANLU"/"DIXI"` 全部不命中 → `:215 return EntryAction.SKIP`。映射层 `mappers`/`remote_watchlist` 全程不做 EN→CN 翻译。
- **后果**：盘前名单装载、竞价四因子、建仓五策略全部空转，系统看似运行实则永不下买单。反之若有人把 family 改成中文，又会突然全量放开——是高度脆弱的隐性开关。
- **修复**：在执行侧唯一映射点建立 `DABAN→打板族 / BANLU→半路族 / DIXI→低吸族` 的规范化表，或让 `_select_action` 按英文枚举精确匹配 `EntryAction`；补两侧共享的 `strategy_family` 取值域契约清单与"至少一条非 SKIP 路由"的联调用例。

**A2. 买入成交从未回写持仓状态机 → 可卖集合恒空，永不卖出（只买不卖裸奔扛单）**　[x-time，confirmed×2，已手工复核]
- `data_writer/callbacks.py:92 on_stock_trade` 只调 `_dw.upsert_trade` 与 `_ledger.add_fill`，**从不调** `PositionManager.mark_position_on_fill`；`ExecCallback` 构造时根本没有 `PositionManager` 句柄。
- 全仓 `mark_position_on_fill` / `apply_position_snapshot` **零生产调用方**（只在 `tests/` 出现）。`PositionManager._units` 永远是空集 → `sellable_units(today)` 恒返回 `[]` → `run_sell_pass` 永远找不到可卖单元。
- **后果**：买入后无论次日是否触发止盈/止损/风控清仓，都不会对真实持仓发任何卖单。**与 A1 叠加**：A1 让它不买、A2 让它不卖；一旦只修 A1 不修 A2，就变成"只买不卖"，是最危险的形态。
- **修复**：`on_stock_trade` 对 BUY 成交调用 `mark_position_on_fill(...)`（`today` 取该成交的东八区自然日，见 P0-E/F 时间口径）；盘前/收盘用 `query_stock_positions` 经 `apply_position_snapshot` 以 QMT 权威 `can_use_volume` 校准；补"买入成交→次日 `refresh_state`→`sellable_units` 非空→`run_sell_pass` 出卖单"的端到端集成测试。

> A1+A2 决定了：**当前执行侧不具备端到端交易能力，任何"小仓实盘"在此前都没有意义。**

### P0-B　资金风控护栏整体为死代码 / 未接线

**B1. 买入路径完全绕过 `risk.gate`**　[exe-risk-order，confirmed]
- 全仓 `risk.gate` 仅在 `app/main.py:265`（卖出路径 `run_sell_pass`）被调用一次（已手工复核）。买入入口 `_router_sink→run_open`（`main.py:238 self._order.place(decision)`）前置只有 `open_new_position_allowed`（空仓）与 `auction_timing_enabled` 两道闸，从不调用 `risk.gate`。`is_open_blocked(market_state)` 也无任何调用方。
- **后果**：账户已触发回撤/亏损红线（本应全账户冻结停止开仓）时，系统仍会继续按信号开新仓，风控形同虚设。

**B2. 账户回撤 / 已实现亏损 / 单票浮亏 三类阈值从未喂入 gate → 三条 FREEZE 分支永不触发**　[exe-risk-order，confirmed×2，corrected→high]
- `risk.py:50-52` 三参默认全 `None`；唯一调用点 `main.py:265` 只传 `market_state/market_feed_ok/trade_conn_ok`，三个亏损量一律缺省。`_breached`（`risk.py:179-181`）在 `observed is None` 时返回 False → `risk.py:94/100/107` 三条 FREEZE 分支永远不命中。
- **后果**：`account_drawdown_limit`/`account_loss_limit`/`stock_float_loss_limit` 即便配了阈值也毫无作用，账户日内回撤熔断与单票止损冻结完全未运行。

**B3. `max_orders_per_day` / `max_total_exposure` / `price_deviation_guard_pct` / 黑名单 四项声明但零实现**　[exe-risk-order，confirmed]
- 三个配置仅在 `settings.py` 定义/解析，无任何业务消费；黑名单全仓零命中。`order_executor._plan_volume`（`order_executor.py:264-287`）只用 `per_order_max_amount`/`max_position_per_stock` 卡单票金额。
- **后果**：单日下单次数无上限（信号异常放量/重复推送可无限开仓）、组合总敞口无刹车、限价偏离无护栏。

**B4. 情绪闸门失效（两个独立 bug 叠加）**　[sig-gate-factors critical + x-contract critical→high，confirmed]
- (a) **`highest_chain_change` 恒为 0**：生产者 `limit_up_push_service.py:2233` 返回 `dict{today,previous,change}`，但 `sentiment_gate.py:318` 直接透传整个 dict（未取 `["change"]`），下游 `_num(dict)` 抛异常被吞→返回 0.0。导致六档分类里"高潮 `hcc>=0`""发酵 `hcc>=0`""分歧 `hcc<=0`"的高度约束全部恒真/恒成立，最高板掉头时仍可能判成发酵/高潮→放行开仓。
- (b) **`market_state` 取值域两侧不一致**：信号侧 `market_state` 只有三档（空仓/谨慎参与/参与），退潮/冰点在另一列 `sentiment_cycle`；执行侧 `settings.py:169 market_state_block=["退潮","冰点","空仓"]` 却拿它匹配 `market_state`，而 `remote_watchlist` 根本不读 `sentiment_cycle`。结果：退潮/冰点在 `market_state` 列永不出现（禁开仓死代码），**唯一能挡的是"空仓"一档，"谨慎参与"日照常激进开仓**。`watchlist_loader.py:30` 注释还误写"与信号侧 market_state 统一口径（启动/高潮/…）"。
- **修复**：执行侧改为消费 `sentiment_cycle`（六档）做 block、`market_state`（三档）做开仓总闸；或信号侧把禁开仓语义显式编码为可机读枚举（如 `open_gate=ABANDON/CAUTION/OK`）由执行侧直接消费。`flatten_gate_inputs` 取 `ec.get("highest_chain_change",{}).get("change")` 标量并补正负两侧分支单测。

### P0-C　下单 / 台账 / 幂等的资金安全

**C1. 卖出直连 `trader.order_stock`，绕过唯一下单点与本地台账**　[exe-risk-order，confirmed，已手工复核]
- `order_executor.py:3` 明文"OrderExecutor 是整条链路中【唯一】调用 `order_stock`/`cancel_order_stock` 的地方"。但 `app/main.py:307 _place_sell` 直接 `self._deps.trader.order_stock(...)` 发卖单（已确认这是除买入 `order_executor.py:212` 外第二个 `order_stock` 调用点）。
- **后果**：卖单无 `biz_order_no`、不落本地台账、无 TTL 撤单、无业务幂等；三层幂等第一层对卖出完全失效；对账事实源缺失卖单记录。
- **修复**：卖出走同一 `OrderExecutor`（新增 `place_sell`，生成 biz 单号、落台账、纳入幂等与 TTL）；禁止编排层直连 trader。

**C2. SELLING 在途单元被重复决策 → 二次下卖单 + 抛错中断本轮其余卖出**　[exe-position，confirmed]
- `position_manager.py:245 sellable_states` 含 `SELLING`，`run_sell_pass` 每个 INTRADAY tick 都对在途 SELLING 单元再次 `decide_intraday`；命中 CLEAR/REDUCE 时 `_place_sell` 先 `order_stock` 再 `mark_selling`，而 `mark_selling` 对非 HOLDING/PART_SOLD 态 `raise ValueError`——即"已发第二张卖单之后才抛错"。异常被 scheduler 吞掉，但本轮其余持仓的应卖决策全被跳过。注释自称 SELLING 留集合里"供上层判定重复下单幂等"，但全链路没有任何 `state==SELLING 跳过` 判断。
- **后果**：同一标的可能被卖两次（超卖/废单）+ 漏卖其余持仓。**叠加 A2 修复后该问题立即暴露**，必须同批修。
- **修复**：`run_sell_pass` 入口对 `unit.state==SELLING` 显式 `continue`；`_place_sell` 在 `order_stock` 之前先校验状态可发卖单。

**C3. write-behind 台账未落盘即发单 → 崩溃/重启窗口重复下单**　[exe-storage-sync + x-idem，confirmed×2/×2]
- `order_executor.py:208 insert(state=PLANNED)` 紧接 `:212 order_stock(...)` 真实发单；但 `PersistentLocalLedger.insert→_mirror` 只 `write_queue.submit`（`put_nowait` 立即返回），真正写盘在后台单写线程异步执行，无 fsync/同步屏障。注释"先落盘再发单"与实现不符。
- **后果**：`order_stock` 已被券商接受、但该行尚未 drain 到磁盘的窗口内进程崩溃/断电，重启 `load_from_db` 读不到 → `find_active` 返回 None → 同一计划被再次下单（资金重复建仓、突破单票上限）。打板瞬间下单密集，恰是崩溃高发时段。
- **修复**：`place()` 在 `order_stock` 之前对该 PLANNED 行做一次**同步落盘**（单独同步写连接或等待该写任务 `task_done`），保证"先有持久台账、再发委托"；启动时除 `load_from_db` 外强制用 QMT `query_orders/query_trades` 当日全量对账兜底，对 remark 命中但台账缺失的委托补建台账、绝不再下。

**C4. `biz_order_no` 序号计数器纯内存、重启不重建 → 同号覆盖失败单 + 潜在重复下单**　[x-idem，confirmed]
- `order_executor.py:72 _seq_counter` 纯内存，`build_biz_order_no` 自增产 `..._001`；`load_from_db` 只重建台账不回填计数器。若某 (date,ts_code,family) 当日唯一一笔是终态失败（CANCELLED/REJECTED/ERROR，不在 `active()` 集），重启后 `find_active` 返回 None→继续 place，新单产出同号 → `INSERT OR REPLACE` 整行覆盖原失败留痕；若原失败单券商侧有残留委托则可能重复建仓。
- **修复**：`load_from_db` 后用台账已有行重建 `_seq_counter`（取已存在最大 seq+1），或改用含时间戳/UUID 段的全局唯一单号。

### P0-D　go/no-go 与回测可信度

**D1. 所声称的 go/no-go 闸门根本不存在**　[sig-backtest，confirmed×2]
- `_evaluate()`（`limit_up_backtest_service.py:608-658`）只产 `distribution/portfolio/by_role/control_excess_mean/leader_strength_ic` 描述统计，`summary_json` 落库即结束。全仓 `go_no_go/GoNoGo` 零命中，summary 字段无任何下游消费者。
- **后果**：README"上线硬次序"的核心门槛（信号侧回测 go/no-go ✅）在代码里是空的——go/no-go 退化为人工肉眼看数字，极易在样本不足/偶然为正时被主观判为"可上实盘"。
- **修复**：把判据代码化（最小样本量、可成交样本胜率/均值下限、对照组超额显著为正、IC 显著为正、最大回撤/p10 不破限……），输出结构化 verdict 落 run 表，导出/上线流程强制读取；阈值文档固化。

**D2. 回测无任何统计显著性检验**　[sig-backtest，confirmed→medium]
- `distribution`/`control_excess_mean`/`leader_strength_ic` 均为裸均值/裸相关，无 CI/t/bootstrap/p 值，`_pearson` 仅 `n<3` 才拒算（3 个点就出 IC）。短线隔日收益方差极大，小样本极易把噪声当 alpha。
- **修复**：均值/超额/IC 加置信区间与 bootstrap/t 显著性，设最小可成交样本数与最小交易日数门槛；不达标标 `INSUFFICIENT` 并阻断 go/no-go。

**D3. 回测默认零费零滑点、仅剔一字/秒封 → 系统性高估收益**　[sig-backtest/x-quant，confirmed→low/opt，但影响 go/no-go]
- `BacktestConfig.include_fees=False` 默认毛收益；买入价 = B 日开盘价、无滑点/冲击成本；可成交性只剔一字/秒封。打板/竞价强开追的真实开盘撮合远比"开盘价精确成交"难。
- **修复**：go/no-go 口径默认 `include_fees=True` 并加滑点/冲击成本（按封单强度/换手分档），summary 并列"理想成交 vs 含成本"，放行只看含成本口径。

**D4. LLM 概率先验从未做校准回测即被消费**　[sig-backtest，confirmed]
- `continuation_prob`/`next_day_premium_prob` 落库并被执行侧消费，但回测全文无 `Brier/可靠性曲线/分桶命中率`，只对 `leader_strength_score` 算了 IC。等于把未验证的 LLM 概率当可信先验。
- **修复**：回测加概率校准（Brier 分、可靠性曲线、分桶命中率/AUC），纳入 go/no-go；校准明显偏离时阻断概率先验在实盘的使用。

### P0-E　时间 / 交易日历

**E1. 生产装配硬编码 `WeekdayTradeCalendar`（仅排周末、不含法定节假日）**　[exe-infra + x-time，confirmed，两条同源]

## 6. 量化逻辑优化空间（用户第二问 · 一）

> 即便修完上述缺陷，以下是把"能跑"提升到"有可持续 edge 且可证伪"的方向。

1. **go/no-go 必须含成本、含显著性、含校准三件套**（对应 D1–D4）：默认净收益 + 滑点/冲击成本；均值/超额/IC 配置信区间与 bootstrap；对 LLM 概率档位做后验校准（统计"高/中/低"档真实续板率/隔日溢价率反推经验概率），替换 `_PROB_BAND_TO_DECIMAL` 的人工定值并随 `scoring_version` 固化。这是真实资金放行的最低门槛。
2. **打分权重与全部阈值当前是未校准占位**（`WEIGHTS_V1`、换手倒 U 5%–25%、续板 0.3/0.6、量能比 0.1 等）。建立校准流水线：用回测 `leader_strength_ic`/分档单调性/分组胜率，在单调约束下搜权重；换手倒 U、action 切点用分组隔日收益定位最优断点；闸门阈值用"空仓日反事实回放"标定。校准前应在字段口径标注"强度分为实验态，执行侧仅作只读归因维度"。
3. **分位/角色用全涨停池而非入选小样本**（P1#5）：分位基准集与落表集解耦，避免 role 随入选数量漂移、`theme_max_board` 只反映入选股。
4. **高度与位置因子正交化**：`board_level` 同时进 height（+0.15）与 position（≤0.03 扣减），高位风险几乎没被真正惩罚，净效应仍是连板越高分越高。应做净"高度-风险"曲线的倒 U / 封顶并回测验证。
5. **缺测降级应"偏保守"而非"中性"**：封板质量/资金/位置缺测回退 0.5 实际把弱票往上抬；资金安全相关维度宜回退到偏保守值（如 0.35）并统计单票 imputed 维度数，超阈时整体下调或标 `low_confidence` 供下游降级。
6. **续板/隔日溢价先验应概率化加权进卖出决策**：当前续板继续持有完全依赖盘口 bool，未用 `continuation_prob/next_day_premium_prob/leader_strength_score` 做分档减仓比例；`next_day_premium_prob` 全链路透传但执行侧零消费（可用于 T+1 早盘兑现 vs 持有择时）。
7. **回测的反事实能力**：现样本=最终通过所有过滤的票，无法回答"universe 过滤/空仓闸门本身是否带正贡献"。新增"反事实回放"（空仓日同样撮合不强制记 0）量化闸门净贡献；落选票落影子表参与对照组，量化过滤机会成本。
8. **回测撮合贴近实盘**：现回测是机械 T+1 open→open/close、固定 hold_window=1，与执行侧动态 `sell_decider`（止盈/止损/炸板/连板）不一致 → go/no-go 难反映实盘 P&L。可让回测复用同一套退出规则，或显式标注"回测=收益上界、实盘 gap 由执行侧承担"。

## 7. 供数逻辑优化空间（用户第二问 · 二）

1. **价位/因子口径统一到信号侧单一来源**：`limit_up_price`/`reasonable_open_*`/`first_board_vol`/`float_mktcap` 由信号侧（有复权/板块/ST 元数据，口径更权威）随 watchlist 下发，执行侧只读不重算；确需本地兜底时 `board_rules` 涨跌幅表必须与信号侧同源 + 单测交叉校验，消除"两套涨停价口径并存"（对应 F1/F3）。
2. **可成交性降级标识结构化下发**：契约增 `tradable_flag`(TRADABLE/WATCH/BLOCKED) + 结构化 `reason`（停牌/ST/新股/一字板/流动性），由信号侧统一判定，执行侧只消费不重判；空仓日 BLOCKED 行应在导出层置可机读否决位（如 `gate_blocked=true`）或清空开仓字段，而非仅软标志（对应 P0-B4 与"空仓行仍导出"问题）。
3. **未复权日线指标的除权失真**：MA5/10/20 与短期涨幅用未复权日线，窗口内除权除息会扭曲均线/收益产生伪破位。窗口内检测到 `adj_factor` 变化时改前复权或打降级标记；新股/次新短窗口均线置 None 或附 `sample_count`。
4. **筹码/市值标度自检**：`cyq` 百分比与 `circ_mv`(万元)/`free_float`(元) 等口径靠约定，缺标度校验。入口加自检（如 `sum(percent)` 落在 [80,120] 视为百分数），不符打质量项置缺，避免换源单位漂移静默失真（对应 F2）。
5. **市场宽度统计口径与选股过滤解耦**：情绪/宽度指标用未剔 ST 的全量涨停池统计，仅在候选入选/落表阶段做 ST 排除，并文档固化"情绪指标含/不含 ST"（对应 P2 ST 一刀切）。
6. **ST/退市/缺数据降级要可观测**：ST 判定依赖 `a_stock_st` 当日同步完整性，名单为空与"当日无 ST 变动"不可区分 → 当日 st_codes 为空/异常偏少时记 WARN 并可选 fail-safe；universe 增 `list_status!='L'` 或 `delist_date<=as_of_date` 落选。
7. **回流四表映射实测固化 + 自检**：`normalize` 全部数值映射与下单常量（BUY=23/SELL=24/FIX=11）在目标机用 `vars()/dir()` 实测固化、统一引用同一常量源，并补"映射命中率/未知值占比"自检日志（大量兜底 BUY / 未知 status 即强告警）。
8. **跨侧封板强度不同口径需显式区分**：信号侧 `seal_ratio_pct`（KPL 盘后封单）与执行侧虚拟封单（竞价）不同源，不可直接比对；字段命名/文档区分 `signal_seal_ratio_eod` 与 `exec_virtual_seal_ratio_auction`，归因只在同口径内比较。
9. **关于 Tushare 历史分钟权限**（承接此前讨论）：当前日线闭环 + QMT 自采 tick 不依赖 Tushare 分钟数据；补 `stk_mins` 仅用于"回测保真度"（把一字/秒封日线保守近似升级为盘中精判 `v_exec.2`、支持 VWAP 成本），且**无法**验证集合竞价择时（竞价微观结构需 QMT `get_full_tick` 实测）。属精度提升项，非上线必需。
