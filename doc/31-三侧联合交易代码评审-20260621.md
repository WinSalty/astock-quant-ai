**信号-4. FOCUS 把 stock_priors 返回成对象（而非数组）时，整张当日选股表写入崩溃、全天 watchlist 落空**

**信号-7. 龙头打分首封时间分位用 bisect_right，遇大量一字 / 竞价封（秒数并列为 0）时把「最早最强」票算成中等分位、封板质量对最强票偏低**（待确认）
`[信号侧] backend/app/services/limit_up_leader_scoring_service.py:215-223,299-305` · 时区与精度
封板质量分里「首封时间越早越强」一项，先把首封时间换成距开盘秒数，一字板和竞价封死都被钳成 0 秒。强势涨停日往往一大批票首封秒数全是 0、扎堆并列；分位用 `bisect_right` 会把所有「和自己一样早」的等值项都计入「下方」。10 只里 6 只 0 秒时，某只 0 秒票分位算成 6/10=0.6、再取 `1-0.6=0.4`——本该接近满分的最早封板最强票只拿 0.4。换算到最终 0-100 分，极端扎堆时把最该高分的一字 / 最早封板票压低约 4 分多。「越小越好」的首封时间本应用 bisect_left。强度分用于执行侧资金加权，方向正好坑最强票。

**信号-8. go/no-go 的「净均值闸门」与「显著性 t 检验」用的是两个不同的收益均值**
`[信号侧] backend/app/services/limit_up_backtest_service.py:860-887` · 取数口径
第一道关「净收益均值为正」看的是 buyable 口径（只统计买得进的成交、逐笔算收益再平均，扣费后）；第二道关「净均值要显著（t≥1.64）」算 t 用的却是 portfolio 口径（每信号日聚成组合日收益、空仓日和全员买不进日记 0、再对所有交易日平均，扣费后）。后者被大量记 0 的日子稀释、通常远小于前者、甚至异号。第一关考核 A、第二关却对 B 算显著性，「两关都过」并不等于「被第一关考核的均值是显著的」，据回测放行实盘的口径自相矛盾（还留着一个 buyable 标准差被读出却没用的死代码，佐证 t 口径漂移）。

**信号-9. 空仓日被当成「收益为 0 的真实交易日」塞进显著性检验，既稀释均值又撑大 n、t 值口径失真**
`[信号侧] backend/app/services/limit_up_backtest_service.py:679-684,772-795,876-877` · 边界
策略判定某天「空仓」时回测给这天记一行收益 0。这个 0 进组合累计收益（代表踏空成本，合理）的同时，也被喂进「判断策略是否显著赚钱」的 t 检验。但空仓不是「下注后收益为 0」、而是策略根本没下注，却被当成独立观测样本：一方面把样本量 n 撑大（t 随 √n 上升、易显得更显著），另一方面把平均日收益拖向 0、波动也改了。这个决定「能不能上实盘」的 t 既不是「真正出手那些天的显著性」也不是干净的「按日历天算的显著性」。空仓多的区间可能 t 虚高误放行、也可能均值被一堆 0 拖低误杀。

**信号-11. 信号腿卖出日遇封死会顺延，对照组卖出日不做任何顺延，两边收益口径不对齐、超额被系统性偏移**

**信号-6. 先验档位文字与四档映射表（高 / 中 / 低 / 极低）稍有出入即落空、触发 DATA_MISSING 误伤（漏买）**（待确认·原标 P1，B3 下线后降为 P2）

**信号-14. _extract_json_payload 用最外层花括号贪婪截取，LLM 在 JSON 前后夹带含括号文本时截到错误片段或解析失败静默走兜底**（待确认）
`[信号侧] backend/app/services/limit_up_push_service.py:4213-4231` · 异常处理
提取 LLM 返回的 JSON 是「找到第一个 `{`、找到最后一个 `}`，中间整段抠出来解析」。当模型在真正 JSON 前后又多说带花括号的话时，这个一刀切会把无关文字和真 JSON 黏在一起、或变成两个并排对象、解析直接失败静默返回空，整阶段被判「解析失败走兜底」。SELECTION 阶段后果是该层（首板 / 连板 / 高位）所有候选股全标不可交易、动作改放弃观察。兜底方向是「不交易」而非错误下单。

**信号-15. 题材成员数把 board_level=0（未识别连板高度）的杂票也算一个成员，独苗真龙头因一只噪声票被误判「板块龙头」高置信**（待确认）

`[信号侧] backend/app/schemas/limit_up_watchlist.py:80-85` · 风控门控

回流落库服务对每条记录算「要更新哪些列」时会排掉唯一键列、DB 自管列（id/created_at/updated_at）、再排掉本次没送来的列。对 qmt_position_snapshot 和 qmt_decision_log 这两张表，程序不会像另三张表那样自动补一个「数据质量留痕」列。于是若执行侧某条回流恰好只带了唯一键那几列、别的字段一个没带，要更新的列就成空列表，拼 upsert 这步直接报 ValueError（更新字典不能为空）。它不是数据校验类错误（不走 422）、落兜底返回 500，执行侧没收 2xx 保 synced=0、下轮拿同一条只带键的数据再传还是 500、这一批一直卡住。边界场景：正常回流会带很多字段，只有执行侧某分支只回键时才踩到。

**两系-9. 盘后回流遇「数据本身非法」被信号侧判 422，执行侧把它当可重试失败、每天重试都失败、永远清不掉**
`[执行侧] qmt_strategy/qmt_strategy/storage/http_ingest_repository.py:64-76` · 幂等
盘后把本机当天的成交 / 委托 / 持仓 / 账户四张表逐行 POST 回信号侧入库。绝大多数脏字段会在本地序列化或信号侧「放宽为可空」的兜底里被洗成空值正常入库。但若某行「确定性非法」——唯一键列为空、或 data 里的账户号 / 交易日和外层信封对不上、或某个非空但格式畸形的日期 / 数值落进对应列——信号侧判「来料校验失败」返回 422。执行侧不区分这种「内容本身错、重传永远过不了」的失败和「网络 / 服务器临时故障」这种可重试失败，一律把该行保留为 synced=0。运维每次手动补同步这行都被原样重发、又被判 422、永远卡未同步：当天同步报告永远 ok=False、错误日志反复刷，却没有任何「放弃 / 死信」机制把救不回来的行剔出去。影响是同步通道被这行长期堵住、运维噪音、不直接造成下单或资金错误。

**两系-10. 竞价量能比的分子（实时撮合量·股）与分母（信号侧 first_board_vol·手）单位可能不一致，比值被缩小约 100 倍**（待确认）
`[执行侧] qmt_strategy/qmt_strategy/auction/auction_factors.py:76-98,259-266` · 时区与精度
竞价量能比 = 开盘集合竞价那 10 分钟成交量 ÷ 这票信号当天全天成交量。分母（全天量）来自信号侧 Tushare、单位「手」（1 手=100 股）；分子（竞价段量）直接取自实时行情 tick 的 volume 字段、一般是「股」。一个用股一个用手，相除时比值被白白缩小约 100 倍、这个「竞价放量」指标整体偏低、和分母不在一个量纲上。目前这个偏低比值不会害它买错卖错：执行侧只有一个策略用到它、且代码已把它从「硬性放弃条件」降级成「只打个弱量标记写进下单理由、不真弃单」，买不买实际由高开幅度和竞价重心趋势（单位正确）决定；项目上线待办也把「volume 到底是手还是股、要和分母对齐」列为真机上线前必核项。风险在将来一旦把这个指标重新当硬门槛用、却忘了先做单位对齐。

**两系-11. 复盘 / 闭环回挂的「生效信号版本」与对外导出口径方向相反，会归因到旧版本信号**
`[信号侧] backend/app/services/qmt_review_service.py:464-487` · 契约一致性

**两系-12. 历史净值曲线 cumulative / NAV 用未剔出入金口径，胜率 / 夏普却用剔出入金口径，两者自相矛盾**

**两系-13. 下单成功率把部分成交的委托算作「成功」，与「买不进只数」口径不互补**（待确认）

# 四、贯穿性的高危集中点（建议优先看）

1. **先验链路对 LLM 输出零防御 + DATA_MISSING 的连锁影响**：信号侧 1 / 4 / 5 / 6（及执行侧 2 的 fail_conditions 关键词误判）是同一条主线——LLM 的 FOCUS 先验只要解析抖动、漏写、近义词、类型不对、跨档串号，就能让好票被静默判缺测或错配。**因 B3 已下线，缺测后果现状是「整档 / 个别强势龙头被静默漏买」而非「真实持仓被强卖」**（这点务必以 §⚠ 修正为准，勿沿用旧契约注释里的强卖口径）。仍属后果较重、触发面广的一类，且全程无告警。

2. **联合接口大量「脏一行 → 整批 / 整页 500 或静默 0」**：两系-1 / 2 / 3 / 5 / 6 / 7 / 8 集中暴露——回流缺 NOT NULL 列、NaN 强度分、坏信封、整批映射失败、空总资产、脏成交、空可更新列，都会放大成 500 死循环重传、盘前装载卡死、或被误当「正常空仓日」吞掉。「入库侧放宽可空」与「展示 / 消费侧仍要求非空、仍会崩」之间没对齐，是系统性缺口。

3. **真机魔数未钉死**：执行侧 5 / 6 的下单方向 / 状态码占位值（23 / 24 / 11）分散三处、回读侧是独立副本，是开交易通道前必须先用 `vars(xtconstant)` 实测钉死并收敛到单一来源的硬前置。

4. **回测 go/no-go 闸门口径不自洽**：信号侧 8 / 9 / 10 / 11 让「能否上实盘」这道总闸的统计口径在多处自相矛盾或被残缺 / 不对齐样本带偏，会直接影响放行实盘决策的可信度（按项目约定，回测类口径修复优先级可后置，但需如实记录）。

5. **注释 / 文档与实现已多处漂移**：两系-14（缺测强卖）、两系-11（复盘版本口径注释称一致实则相反）、执行-16（守卫注释把非乐观锁当乐观锁）、信号-8（t 检验口径漂移留死代码）等，反映多处注释陈述与实际代码相反，易误导后续改动。
