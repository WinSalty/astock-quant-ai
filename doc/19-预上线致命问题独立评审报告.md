## 一、致命问题（CRITICAL）

### C-1 默认生产配置下，买入后无任何自动卖出出口（裸奔扛单）

- **文件:行号**：`app/run.py:414-425`（`sell_pass_live=False → sell_books_provider=None`）；
  `app/scheduler.py:142-145`（provider 为 None 即 `return`，`run_sell_pass` 不执行）；
  `config/settings.py:171`（`QMT_SELL_PASS_LIVE` 默认 `False`）。
- **触发条件**：默认配置（`QMT_SELL_PASS_LIVE` 未显式置真）。当前生产 `run_qmt.bat`（见待办 §G）
  亦未配该项。
- **后果**：一旦买入成交，止损 / 炸板 / 破位 / 连板了结 / T+1 常规卖出**全部不触发**——盘中只跑
  `sweep_ttl`（仅撤未成买单）、收盘只做快照对账，**没有任何路径对已建仓位下卖单**。即买入即裸奔扛单，
  无止损出口。
- **复核结论**：**真实**。这是文档（README §6、状态概要「第四轮」、待办 §0）已登记的头号上线硬阻塞；
  本报告仅作完整性确认——它是当前代码状态下最高优先级的致命事实，未解除前严禁接实盘。

### C-2 卖出决策器 `decide_auction` 全程不读 `is_sealed`，竞价封板的弱先验隔夜票被误判「弱开」清仓

- **文件:行号**：`position/sell_decider.py:114-164`（`decide_auction` 各分支从不引用 `book.is_sealed`）；
  对比 `position/sell_decider.py:212`（`decide_intraday` 有 `is_sealed` 续持判定）。
- **触发条件**：开启 `QMT_SELL_PASS_LIVE` 后；持仓为可卖的隔夜票（非 `LOCKED_T1`），先验弱或缺失
  （`prior=None`/`continuation_prob` 缺失或偏低 → `prior_strong=False`），今日竞价一字 / 秒封涨停
  （`open_pct>0`、`is_sealed=True`、未背离）。
- **后果**：逐分支走至兜底 `if not prior_strong:`（`:157-163`）→ `_reduce_or_clear(prior_strong=False)`
  → **CLEAR**，reason 记为「弱开」。即对一只**正封涨停**的持仓在竞价段判定清仓，`place_sell` 以现价
  （≈涨停价）挂卖单——砸开自家封板或在涨停高点主动放弃仍在续板的强势仓，与分时段「秒板稳封 → HOLD」
  口径自相矛盾。
- **复核结论**：**真实**。与文档已登记的 B1（plan 缺失致 `is_sealed=False`）是**不同**的洞：本洞即便
  `is_sealed=True` 也会清。属卖出链开门（`QMT_SELL_PASS_LIVE=true`）前必须先消除的逻辑缺陷，
  叠加 C-3 共同决定「卖出链现状不可开门」。

### C-3 生产卖出盘口构造的跨帧字段恒为默认值，开门后炸板 / 破位 / 烂板 / 尾盘了结四类硬扳机永不触发

- **文件:行号**：`position/sell_book_builder.py:63-67`（`broke_board`/`below_support`/`volume_surge`/
  `near_close_weak`/`price_volume_diverge` 恒 `False`、`open_times` 恒 `0`）；消费侧
  `position/sell_decider.py:201-231`。
- **触发条件**：开启 `QMT_SELL_PASS_LIVE` 且仍用当前 `build_order_book`（生产盘口唯一来源
  `app/main.py` 的 `build_sell_books`）。
- **后果**：`decide_intraday` 的炸板出（`broke_board`）、走弱破位（`below_support && volume_surge`）、
  烂板出（`open_times>=2`）、冲高止盈 / 尾盘了结（`price_volume_diverge`/`near_close_weak`）**全部恒不命中**；
  强先验单元在分时只会落 HOLD，唯一价位安全网退化为 `_unit_stop_loss_breached` 的单帧浮亏止损（止损线通常
  远低于炸板 / 破位位），等同破位扛单。
- **复核结论**：**真实**（代码注释自述「最小版保守默认，待 T1.2 真机实现」）。该状态下若误开
  `QMT_SELL_PASS_LIVE`，C-2 误清仓 + C-3 止损失效会同时暴露在真金持仓上。属强阻塞，列为致命以提示
  「跨帧盘口保真未落地前绝不可开卖出门控」。

## 二、严重问题（HIGH）

### H-1 `sync_status` 缺「不回退已成交态」守卫，乱序回报会把已 TRADED 单降级 → 对已全成单发撤单 + 台账态错乱

- **文件:行号**：`order/local_ledger.py:145-148`（`sync_status` 对非 CANCELLED/REJECTED 态
  无条件 `e.state = state`，无防降级判断）；对比同文件 `add_fill` 的 `:196`（`if e.state != OrderState.TRADED`
  防降级守卫）；下游后果链 `order/order_executor.py:1014-1021`（state ∈ {SUBMITTED,REPORTED,PART_TRADED}
  即发 `cancel_order_stock`）、`:1059-1060/1066-1077`（CANCELLING 每宽限期重发撤单）。
- **触发条件**：QMT 的 `on_stock_trade`（成交，驱动 `add_fill` 置 TRADED）与 `on_stock_order`（委托态，
  驱动 `sync_status`）异步乱序——成交回报先到把单置 TRADED 后，迟到的 `on_stock_order(部成/已报)` 再到。
- **后果**：已全额成交单被降级为 PART_TRADED/REPORTED；其 TTL 截止仍在，到期 `on_ttl_expired` 判「在途」
  → **对一只已全成的委托发撤单**（`filled>0` 不走 miss/转次优，但仍 cancel），置 CANCELLING；该撤单不会有
  CANCELLED 回报（券商无活动委托），导致每个宽限期（默认 30s）反复重发撤单至收盘，且收盘对账时该单台账态
  非 TRADED，污染滑点 / 成交勾稽口径。
- **复核结论**：**真实**（`add_fill` 与 `sync_status` 对「不回退已成态」的口径不对称是确定的代码缺口）。
  触发依赖 QMT 回调投递次序（属 §A 真机待实测项），但单帧浮亏 / 高频秒成场景下乱序回报现实可达；
  `OrderState` 无序数，靠枚举大小无法拦截。资金不直接损失（成交量由持仓侧独立追踪），属「台账状态错乱 +
  向券商发无效撤单」级硬伤。

### H-2 `reconnect` / `connect_and_subscribe` 无「重连进行中」互斥，多线程并发重连导致 trader 实例与后台线程泄漏 + 句柄漂移

- **文件:行号**：`connection/connection_guard.py:142-185`（`_lock` 只包状态读写的短临界区，
  `trader_factory`/`register_callback`/`start`/`connect`/`subscribe` 这些 I/O 全在锁外）、
  `:205-254`（`reconnect` 无重入保护）；并发触达点：`app/run.py:347`
  （`on_disconnected → guard.reconnect`，xtquant 回调线程）、`app/run.py:375`
  （`_supervise_once → guard.reconnect`，主线程）、`app/scheduler.py:171`
  （PREWARM `connect_and_subscribe`，调度线程）。
- **触发条件**：盘中断线（QMT 不自动重连，断线为常态运营事件）。断线后回调线程经 `on_disconnected`
  立即重连；主线程 `run_forever` 返回后经 `_supervise_once` 也重连；二者无串行化。
- **后果**：两路并发各自 `trader_factory(session_id)` 造两个新 trader、各自 `start()` 起后台线程，
  `_current_trader` 与 `holder` 被后写者覆盖——**前一个 trader 的后台线程无人 stop/join 泄漏**，且就绪窗口内
  主线程可能对「已被覆盖、即将泄漏」的半就绪句柄 `run_forever`；多个 trader 并存还可能重复进回调。
  反复断线下线程数与连接句柄持续泄漏。
- **复核结论**：**真实**。`snapshot()` 的一致快照只保证读到「某一个一致对」，挡不住两线程各建一个 trader；
  代码注释亦自承「run_forever / 断线重连交互依 QMT 运行期行为须实测」「与 PREWARM 建连叠加 TODO 实测口径」
  ——即并发安全尚未被验证关闭。

### H-3 账户回撤闸与限价偏离闸默认关闭且无装配期强校验，「风控五道闸」中两道默认形同虚设

- **文件:行号**：`config/settings.py:140`（`price_deviation_guard_pct=None`）、`:151`
  （`account_drawdown_limit=None`）；`config/settings.py:365-389`（`assert_safe_to_trade` 仅强校验
  `auction_timing_enabled→verified` 与 `sell_pass_live→stock_float_loss_limit`，**不校验**
  `account_drawdown_limit` / `price_deviation_guard_pct`）。
- **触发条件**：运维未显式配 `QMT_ACCOUNT_DRAWDOWN_LIMIT` / `QMT_PRICE_DEVIATION_GUARD_PCT`
  （二者均无默认、无启动断言；当前生产 `run_qmt.bat` 未列入）。
- **后果**：README 宣称的「风控五道闸」中——账户级当日回撤熔断对开仓**零作用**（爆亏当天仍可继续开新仓
  至现金耗尽）；限价偏离护栏不工作（价位口径错位产生的追高 / 废单只剩「超法定涨停价」一道兜底）。两道
  资金护栏退化为「按需启用」，且无任何启动期 fail-closed 提醒。
- **复核结论**：**真实**。`risk` 闸对 `limit=None` 直接放行、`_limit_price_sane` 对 guard=None 整段跳过；
  与单票浮亏止损（已对 `sell_pass_live` 做强校验）相比，回撤 / 偏离两闸缺同等启动强校验，属确定的 fail-open
  配置面缺口。

### H-4 调度相位单点：PREWARM 过 9:15 即定稿（不论连接 / 装载是否成功）、RUN_AUCTION 先标 fired 再长阻塞，异常 / 未就绪下当日竞价相位不可恢复

- **文件:行号**：`app/scheduler.py:196-201`（`should_retry = before_auction and (...)`，`before_auction`
  即 `now < 9:15`；过 9:15 一律 `_mark(PREWARM)` 定稿）；`app/scheduler.py:211-218`
  （`RUN_AUCTION` 分支**第一行**即 `_mark(RUN_AUCTION)`，随后才 `run_auction()` 阻塞至 9:30）；
  `decide` 复触发判据 `app/scheduler.py:110-124`（已 fired 即不再返回该相位）。
- **触发条件**：① 进程在 9:15–9:30 之间启动 / 重启且建连失败 → PREWARM 仍被定稿，`engine.prewarm`
  基于未就绪 trader 重建持仓 / 抓日初权益基线（可能失败 / 不完整），且永不重做；② `run_auction()` 入口
  附近抛任何异常（被 `run` 顶层 try/except 吞），但 RUN_AUCTION 已 fired → 当日竞价**整段放弃且不重试**。
- **后果**：①「连接未就绪下定稿的 PREWARM」使持仓重建 / 权益基线停留在不完整态、当日不再修正；
  ② 竞价段一次进入失败即当日不竞价建仓——与 `AuctionPoller` 内「单轮异常不放弃整段」的健壮性设计相矛盾
  （保护只覆盖 poll 循环内部，不覆盖进入 `run_auction` 前 / 框架级失败）。
- **复核结论**：**真实**（相位「先标 fired 再执行」+「过 9:15 强制定稿」的口径确定）。实际影响取决于启动
  时序与 `run_auction` 入口的抛异常面，属调度层不可恢复单点；竞价择时开启（真金竞价下单）后风险显著放大。

## 三、中等问题（MEDIUM，仍影响资金 / 数据正确性）

### M-1 持久化非原子：台账行镜像与成交明细 INSERT 分两次入队、purge 两条 DELETE 分两次入队，叠加「明细为重启权威」可把成交量纠成错值或留孤儿明细

- **文件:行号**：`storage/sqlite_ledger.py:245-259`（`add_fill` 先 `_mirror`（整行 replace 含累计
  `filled_volume`）再单独 `submit` 明细 INSERT，两次独立任务）；`:122-134`（`purge_before` 删明细、删台账
  分两条独立 `submit`）；`:86-88`（`load_from_db` **全表无窗口**读回 `local_order_fill`）；重算覆盖口径
  `order/local_ledger.py:227-229`（`reconcile_fills_from_detail` 用明细之和**覆盖** `filled_volume`）。
- **触发条件**：写线程对单条任务的异常被「只吞不传播」（`storage/write_queue.py:194-213`），或队列溢出
  丢弃单条（`:116-119`）——出现「整行镜像成功、明细 INSERT 丢失」或「删台账成功、删明细失败」的非原子半成。
- **后果**：① 明细丢失时，重启 `load_from_db` 以明细为权威**反向把正确的快照 `filled_volume` 纠成偏小值**
  → 已成交量被低估（可能误判未成 / 漏卖 / 对账不平）；② 台账行已删而明细未删 → 孤儿明细，后续 purge 子查询
  再也定位不到，且 `load_from_db` 每次启动全表读回，规模随运行无界增长。
- **复核结论**：**真实但有前提**（依赖单条写任务失败 / 溢出的非原子半成；正常运行下 purge 已在收盘
  `app/main.py:1267` 调用、明细随台账行一并清理而有界）。属持久化原子性 / 重启可恢复性的确定缺口，非
  常态必现。`flush_confirm` 的批级确认经复核**不构成漏报**（仅由调度单线程串行调用，满足其「单一调用方」前提）。

### M-2 单票持仓查询失败时 `_held_value` 返回 0，单票上限净额校验漏计隔夜持仓，跨日同票可超单票上限加仓

- **文件:行号**：`order/order_executor.py:791-793`（`query_stock_positions` 异常 → `warn` 后 `return Decimal("0")`）；
  净额校验链 `:850-856`（`exposure_for_code = _held_value + _committed_remaining_for_code`）、`:896-908`
  （`effective_cap = cap_stock - held`）。
- **触发条件**：配置了 `QMT_MAX_POSITION_PER_STOCK`；下单测算时刻 `query_stock_positions` 抖动失败；
  且该票为跨日连续上榜的同一标的（打板核心标的复买）。
- **后果**：`exposure` 退化为只含当日未成在途计划额、**不含隔夜已有持仓市值** → 单票上限净额闸失效，
  在已有持仓之上再满额买入，单票敞口可累计逼近 2×cap。注释自承「查询失败保守返 0、由其它闸兜底」，但
  总敞口 / 单笔上限均不针对单票累计，无法真正兜住。
- **复核结论**：**真实但有前提**（需「查询失败 + 同票跨日复买 + 配了单票上限」三者同时）。后果方向是
  单票超配（资金集中度风险），非全局超买。

## 五、结论

执行侧主资金路径（下单幂等、强度加权分配、买入前置过滤、台账三层去重、跨日反查、写队列 fail-closed）
经多轮评审后已相对扎实；本轮新确认的问题集中在三处：**(a) 卖出链整体（C-1/C-2/C-3）——默认无出口、
开门后逻辑误清仓且止损失效**；**(b) 异步回报 / 重连的并发与乱序口径（H-1/H-2）**；
**(c) 风控与调度的 fail-open / 不可恢复单点（H-3/H-4）**。其中 C-1/C-2/C-3 与 H-1/H-2 直接关系资金安全，
应视为放量上线前的硬阻塞。
