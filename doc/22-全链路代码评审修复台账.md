## B1 · 部成单被撤后无限重撤 + 死量预算超额承诺（✅ 已修并测）

**根因**：`PART_TRADED` 同属 `OrderState.terminal()` 与 `active()`，且 `local_ledger.sync_status` 把「`CANCELLED/REJECTED` 且 `filled>0`」的部成撤单收口回 `PART_TRADED`——它仍在 `on_ttl_expired` 可撤集合 `{SUBMITTED,REPORTED,PART_TRADED}` 内，每个 grace 周期被反复撤单；其死量 `remaining` 仍被 `committed_amount` 计入预算承诺。

**设计**：新增独立终态 `OrderState.PART_CANCELLED`（部成单未成部分被撤/废后的终态收口），与「部成仍在途」的 `PART_TRADED` 严格区分。它：① 不在 `on_ttl_expired` 可撤集合 / `rebuild_runtime_state` in_flight 集合内 → 退出 TTL 盯防、不再重撤；② `committed_*` 对它置 `remaining=0` → 死量不占承诺；③ 仍在 `active()`（保 find_active 幂等防重复下单）、`filled>0` 仍占建仓只数名额；④ 三处「不回退」守卫（`sync_status`/`add_fill`/`reconcile_fills_from_detail`）粘滞终态，杜绝迟到回报把它降级回在途态复活重撤循环。

**改动**：
- `contracts/enums.py`：新增 `PART_CANCELLED`，纳入 `terminal()` 与 `active()`。
- `order/local_ledger.py`：`sync_status` 收口改 `PART_CANCELLED` + 不回退守卫扩到 `(TRADED, PART_CANCELLED)`；`add_fill` 状态推进守卫同扩（迟到成交仍累计 filled 但不改终态）；`reconcile_fills_from_detail` 跳过重算守卫 + 收口同步改。
- `order/order_executor.py`：`committed_amount`/`_committed_for_code`/`_committed_remaining_for_code` 对 `PART_CANCELLED` 置 `remaining=0`。
- `data_writer/callbacks.py`：同步过时注释（PART_TRADED→PART_CANCELLED；该函数判据走券商回报状态，逻辑无变化）。

**测试**（新增 4 个，更新 1 个）：
- `test_local_ledger.py`：`test_sync_status_fill_aware_cancel_settles_part_cancelled`（收口 PART_CANCELLED）、`test_part_cancelled_is_sticky_terminal_no_downgrade`（迟到委托态/成交不降级、filled 仍累计）。
- `test_order_executor.py`：`test_part_cancelled_no_recancel_loop`（撤后再 sweep 不复发撤单、退出 TTL 盯防）、`test_committed_amount_excludes_part_cancelled_dead_remaining`（死量 remaining 不计承诺）。

**review**：独立对抗式审查通过——逐处核对所有读 `OrderState.PART_TRADED` 的分支（`on_ttl_expired`/`rebuild_runtime_state`/`_is_position_committed`/对账/持仓机/存储/回调），确认无漏处理、无回归；SELL 方向（`committed_*` 已过滤 `side==BUY`，持仓机不读 OrderState、on_road 回扣按券商回报状态）不受影响；`state TEXT` 无 CHECK 约束、`OrderState(value)` 可还原新值，无需迁移。审查唯一发现 = `callbacks.py` 两处过时注释，已修。

**结果**：全量 **705 passed**（净增 3 用例）。

**SQLite 兼容**：`local_order_ledger.state` 为 `TEXT NOT NULL` 无 CHECK 约束，`mappers.row_to_ledger` 用 `OrderState(value)` 还原，新值跨重启/落盘正常，无需 Alembic 迁移。

## E1/E2/E3 · 建仓路由/竞价采集的当日状态与幂等锁（✅ 已修并测，同批处理）

三者共属「竞价/建仓路由的当日/锁状态」根因，一次处理。

### E1 · 建仓幂等锁先于编排层风控/限价闸 → 单帧瞬时拒单致该票全天该买不买

- **根因**：`EntryRouter.on_auction_snapshot` 在产出非 SKIP 决策的那一帧即把 ts_code 锁进 `_decided`，而 `_router_sink` 的 `_open_blocked_by_risk`/`_limit_price_sane` 晚于它执行；两条拒单分支只 `return`、不 `release` → 单帧瞬时拒因（行情/下单中断、账户回撤查询瞬断 fail-closed、限价偏离护栏缺盘口现价）把该票永久锁死、当日再不评估。
- **修复**：`_router_sink` 的 risk_block 与 limit_price_guard 拒单分支在 `return` 前各加 `self._entry.release(decision.ts_code)`，使瞬时拒因恢复后的后续帧能重评下单（与既有「竞价 collect-only 分支每帧 release」同口径）。`open_new_position_allowed=False`（空仓/降级 day-level 持久态）分支刻意不 release（保持锁定、避免无谓重评，且该分支不 release 故天然每票只留痕一次）。
- **复审修改（防洪泛）**：独立 review 指出「解锁重评在持久拒因下会每帧×每票重复 emit SKIP_ORCHESTRATION 到 decision_log（有界队列 2000 会 drop 真实行）+ 刷 engine 日志」。据其建议补去抖：新增 `self._orch_block_logged` 集合（`__init__` 建、prewarm 清），按 `(ts_code, reason_code)` 每票每拒因每日只留痕一次；`_open_blocked_by_risk` 增 `quiet` 参数（仅 gate 内部 fail-closed 日志、不改裁决、不跳过 `_account_drawdown`/`gate` 计算）。release 仍每帧无条件执行，恢复能力不受去抖影响。

### E2 · `EntryRouter._decided` 跨交易日永不重置 → 昨日成交标的次日被永久锁死不建仓

- **修复**：新增 `EntryRouter.reset_day()` 清 `_decided` + `_last_recorded_action`，`Engine.prewarm` 每日调用。破跨日「该买不买」、与 `_strength_budget_volume` 预期的「跨日复买连板龙头」自洽。

### E3 · `AuctionPoller._history` 跨交易日永不重置 → 内存无界增长 + tick_seq 留痕失真

- **修复**：新增 `AuctionPoller.reset_history()` 清 `_history`，`Engine.prewarm` 每日调用。跨日从空历史起算、tick_seq 不被前日帧数抬高。

### 测试（新增 5）
- `test_entry_router.py::test_reset_day_clears_idempotent_lock_allows_next_day_buy`（E2 跨日可再买）。
- `test_auction.py::test_reset_history_clears_accumulated_frames`（E3 清空 + tick_seq 复位）。
- `test_app_engine.py::test_transient_risk_block_releases_lock_allows_later_buy`（E1 瞬时拒单解锁、恢复后下单）。
- `test_app_engine.py::test_persistent_risk_block_dedups_留痕_no_flood`（E1 复审去抖：5 帧重评只留痕一次）。

### review
- 第一轮独立对抗 review：核心修复正确（不重复下单——拒单分支无 place、place 另有 find_active 幂等；空仓语义保持；跨日重置完整；scheduler 单线程顺序执行无并发清空风险）；发现持久拒因洪泛回归 → 已按建议补 `_orch_block_logged` 去抖。
- 第二轮 review（针对去抖修改）：确认 release 仍每帧执行（恢复不削弱）、quiet 只 gate 日志不改裁决、`(ts_code,reason)` 键允许不同拒因各留痕一次、首帧留痕后续静默无 off-by-one、集合每日清空、BUY 成交留痕走 order_executor 独立不被波及。无新问题。

**结果**：全量 **709 passed**（净增 5 用例）。

## C1 · 交易日历末日越界 fail-open → fail-closed（✅ 已修并测）

- **根因**：`PositionManager.mark_position_on_fill`（买入回写）与 `rebuild_from_broker_positions`（券商持仓重建）直接调 `calendar.next_open/prev_open(today)`，`StaticTradeCalendar` 在日历末日==today 时抛 `ValueError`：① 买入回写异常被 `_apply_trade_to_position` 宽 except 吞掉 → 已成交持仓静默丢出状态机、隔夜裸奔无止损；② rebuild 无 per-item 兜底 → 整批失败穿透 prewarm → 当日 RUN_AUCTION/INTRADAY 被门控掐断、全仓卖出管理瘫痪。本应 fail-closed（拒开新仓+强告警）被偷换成 fail-open。
- **修复**：
  - `position_manager.py`：新增 `_safe_next_open`/`_safe_prev_open`（catch 越界 → 强告警 + 退化为「下一/上一非周末自然日」占位），买入回写与 rebuild 改用之，**保证已成交持仓仍进状态机被管理**；rebuild 循环加 per-item try/except（单条坏记录跳过+告警、不拖垮整批）。T+1 安全由「买入当日锁硬编码（`can_use_volume=0`/`LOCKED_T1` 不读占位日）+ 券商 `can_use_volume` 在 rebuild/snapshot 中权威兜底」双重保障，占位日不精确不致 T+1 违规。
  - `trade_calendar.py`：新增 `trading_days_left(d)`（`StaticTradeCalendar` 返回晚于 d 的交易日数、`WeekdayTradeCalendar` 返回大数）。
  - `app/main.py::prewarm`：duck-typed 覆盖度预警，剩余交易日 <5 即强告警提示外延日历。
  - **review 同源补漏**：独立 review 发现 `reconcile/reconcile.py::backfill_signal_trade_date` 的裸 `prev_open` 同属 C1 越界 fail-open（历史补采老成交 trade_date 早于日历首日时 ValueError 中断对账）→ 同口径加 try + 占位反推（仅影响 signal_trade_date 归因标签、不涉 T+1/下单）。
- **测试**（新增 6）：买入回写越界仍建单元（不丢仓、占位可卖日）、rebuild 越界仍建多单元且单条隔离、`trading_days_left` 边界（Static/Weekday）、reconcile prev_open 越界占位。
- **review**：独立对抗 review 确认 T+1 安全确凿（占位日只影响状态机解锁时点、券商 can_use_volume 独立把守 T+1 量闸）、per-item try 语义/锁/计数/作用域正确、买入回写丢仓漏洞已堵；唯一发现 reconcile 同源遗漏 → 已补并测。
- **结果**：全量 **714 passed**（净增 6 用例，含 reconcile 同源）。

## B2 · 负哨兵 order_id 写入反查索引（✅ 已修并测）

- **根因**：`order_executor.place`/`place_sell` 在 `order_stock` 同步失败（返回 <0 哨兵）时把负值写进台账 order_id 字段 → 被 `local_ledger._index_order_id`（只跳过 None、不拦负值）注册进两级反查索引 → 多笔失败在 `(-1,today)` 互相覆盖（留痕串改）、任何携负 order_id 的回报经 `_resolve_biz(-1)` 误改写该 ERROR 行（add_fill 对 ERROR 无守卫会误累计成持仓）。
- **修复**：buy/sell 同步失败分支改为【不传 order_id】（保持 PLANNED insert 的 None），失败码改写进 `error_msg=f"...(返回{order_id})"`。order_id=None → `_index_order_id` 跳过、`_resolve_biz(-1)` 永不命中、负值回报永不串到该失败单；log/emit 留痕仍用 order_id 变量（不入台账）。
- **测试**（新增 1）：`test_sync_failure_keeps_order_id_none_no_index_pollution`——同步失败后台账 order_id=None、失败码入 error_msg、`get_by_order_id(-1)`/`add_fill(-1,...)` 均落空不串单。

## P1 · 盘中重连重建对 SELLING 单元不复位 → 整日卡死漏卖（✅ 已修并测）

- **根因**：盘中断线重连后 `on_reconnect_backfill` 只调 `_rebuild_positions_from_broker`（校准 volume/can_use，但对 SELLING 单元不复位/不清 on_road/不查券商委托终态），而能复位卡死 SELLING 的 `_reconcile_stuck_selling` 仅在 `prewarm` 调用。故断线期挂出的卖单被券商撤/废且撤单回执在断线期丢失时，单元跨重连仍卡 SELLING、`sellable_remaining=can_use-on_road=0`、`_evaluate_and_sell_unit` 对 SELLING 一律跳过 → 该票当日永久漏卖，止损/破位/炸板失效，需等次日盘前才解。门控前提：`QMT_SELL_PASS_LIVE=true`。
- **修复**：`on_reconnect_backfill` 在 `_rebuild_positions_from_broker(today)` 之后补调 `self._reconcile_stuck_selling(today)`（复用盘前同口径的券商委托终态对账：零成交终态 CANCELLED/REJECTED/EXPIRED→revert_selling 复位 HOLDING+清 on_road 重挂；已成/部成/仍在途→不动；查不到→保守不动+告警）。`today` 提到 try 顶部统一取。
- **测试**（新增 1）：`test_on_reconnect_backfill_reconciles_stuck_selling_after_rebuild`——验证重连路径在 rebuild 之后补调了卡死 SELLING 对账（reconcile 逻辑本身由 `test_position_manager` 的 reconcile_stuck_selling 用例覆盖）。
- **review**：独立对抗 review 确认——重连上下文查询安全、终态分类两侧一致（不误 revert 活跃卖单）、rebuild→reconcile 顺序正确（校准 can_use 被保留、复位后可正常重挂，且优于反序）、异常包裹为尽力而为不阻断、收盘 close_batch 路径不需补（次日 prewarm 兜底）。无回归。
- **结果**：全量 **716 passed**（净增 1 用例）。
