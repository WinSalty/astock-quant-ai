# 两侧代码修复计划与执行清单

> 依据：[`08-信号侧与执行侧量化交易代码评审报告`](08-信号侧与执行侧量化交易代码评审报告.md)（12 critical + 24 high + 63 medium）。
> 原则：按依赖顺序分批、每批可独立验收、全程不破坏既有绿测、所有触碰代码补中文注释、不含工时。
> 测试环境：执行侧 `astock-quant-ai/.venv/bin/python -m pytest`（qmt_strategy 下，fake 适配器，macOS 可跑全量）；信号侧 `backend/.venv/bin/python -m pytest`。
> 实测口径：标 **[实测]** 的子项依赖目标 Windows/QMT 机，本地只落代码结构 + 安全默认 + 单测，真实数值在目标机用 `vars()/dir()` 固化。

## 批次总览与依赖

```
批次1 闭环可用+防重复(P0-A/C) ─┐
批次2 风控护栏生效(P0-B)        ├─► 批次5 P1 高危
批次3 日历+供数口径(P0-E/F) ────┘            │
批次4 go/no-go+回测可信(P0-D) ───────────────┴─► 批次6 P2 中危 + 优化项
```
- 批次 1 是一切前提：闭环不能交易，其它修复无法端到端验收。
- 批次 2 依赖批次 1（风控要接到真实买/卖路径上）。
- 批次 3、4 可与 1/2 并行推进，但上线门槛要求四批全过。
