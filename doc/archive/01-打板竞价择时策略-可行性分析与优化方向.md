# 打板量化方案设计：LLM 选股 → 龙头战法增强 → QMT 侧独立交易

> 方案定位：复用 `stock-ah-premium-ai` 的 LLM 打板报告，把它产出的「重点股票」作为**待观测股票池（watchlist）**；在信号侧加一层确定性的龙头战法增强与空仓闸门、结构化落表；交易（买入/持仓/卖出）在 **QMT 侧独立开发**。本文结论与设计结合了对生产库 `stock_ah_ai` 的只读查询、对项目代码的走查、对 Tushare 能力与 A 股交易规则的外部核查，并经对抗式核验修正。
>
> 边界约定（贯穿全文）：信号侧只产「候选 + 特征 + 标注」，**绝不产仓位/买卖/止盈止损指令**；仓位与满仓在 QMT 隔离账户内由用户自决、风险自担，本文不做仓位说教。

---

## 一、总体结论

本方案工程上可行，评估口径是「**LLM 每日选出的 watchlist 作为信号是否具备可执行的增量收益（alpha），以及如何结构化落表、解耦给 QMT 侧独立执行**」。三条主结论：

1. **结构化产出是确定能拿到的价值。** 在现有多阶段 LLM pipeline（约 08:31 跑批、基于上一交易日 T 涨停数据）末尾追加一层确定性的「可计算特征 + 龙头强度打分 + 角色/战法/情绪标注 + 可成交性标注」，把已存在但嵌在 `context_json.pipeline` longtext blob 里的选股，展开为一股一行的可查询落表。即便打板信号最终被证明没有 alpha，这一步也让你第一次能用自己的数据回答「LLM 选股到底有没有增量」。
2. **选股与交易解耦是明确工程边界。** 信号侧（Linux，现有项目）只产 watchlist；执行侧（Windows，QMT 独立模块）只消费 watchlist + 自采实时行情，自行决定择时、下单、持仓、止盈止损。两侧只通过一份契约（落表 + 只读接口）通信。
3. **「打板信号有 alpha」本身是待证伪的假设。** 推进的成败闸门是：watchlist 相对全市场涨停池是否有显著、稳定、可执行的增量与分组区分度。**先证明增量存在，再谈精细化。**

## 二、前置条件与假设

> 三项外部资源统一列为**将到位的前置条件**（不是「能不能做」的不确定性），并说明各自解锁什么能力。客观接口限制只作为实现注意事项写明。

### 1. 三项前置条件与各自解锁的能力

| 前置条件 | 状态 | 解锁的能力 |
|---|---|---|
| (a) 国金证券 QMT 账户与程序化交易权限 | 将开通 | `xtdata`（行情）+ `xttrader`（下单）做**实时竞价轮询、盘中下单、持仓管理与卖出**；QMT 侧消费信号契约并真正「扣扳机」 |
| (b) Tushare 历史/实时分钟数据权限 | 将搞定 | `stk_mins` 做**分钟级回测的成交判定**（把日线对一字/秒封的保守近似升级为按盘中是否出现成交、封单是否被砸开做精判），及实时分钟辅助盘中校验 |
| (c) 独立 Windows Server VPS | 将购置 | 专跑 QMT（`xtdata`/`xttrader` 为 Windows-only），承载竞价轮询器与卖出决策器等实时模块，与信号侧（Linux）物理隔离 |

### 2. 能力本身的客观限制（实现注意事项，非不确定性）

- **竞价段 tick 回调不触发**：QMT 在 9:15-9:25 不推送 tick 回调，必须**自写定时器（每 1-3 秒）轮询 `get_full_tick`** 取竞价高开幅度/竞价量能/虚拟封单。能否拿到竞价撮合价需在 VPS 上做一次性实测；实测前竞价择时按「假设参数」运行，并保留**一键降级为只看 `last_price` + 开盘后确认**的路径。
- **报告无实时/竞价数据**：现有报告全部基于 T 日盘后快照，竞价/盘中字段 `bid_*`/`open_times`/`open_time`/`real_time_pct_chg` 真实全为 `None`。竞价高开、是否秒板/炸板、是否破开盘价这类信息**物理上要等 T+1 盘中（9:25 定盘后）才存在**——这正是把「卖不卖决策」放到 QMT 实时侧的根因。
- **现有实时表承载力有限**：`realtime_quote_snapshot` 当前仅 `last_price` 一列，承载不了盘口/竞价/封单；卖出决策所需信号须由 QMT `get_full_tick` 直接采集。

### 3. 已有可复用基础设施（无需新建）

| 基础设施 | 用途 |
|---|---|
| `a_stock_basic`（ts_code/name/market/list_status/list_date/delist_date/exchange） | 退市/暂停状态、板块二次校验、上市天数计算 |
| `a_trade_calendar`（cal_date/is_open） | T→T+1 交易日映射、新股上市交易日计数（禁止按自然日 +1） |
| `tencent_unadjusted_daily_quote`（adjust_type='NONE'） | 回测**不复权**行情、一字/封板判定 |
| `stock_identity_resolver` | ts_code 身份解析、信号表与实时表对齐 |
| 现有多轮 LLM pipeline（limit-up-multi-stage-v3 / deepseek-v4-pro） | 已含 selected_first_board/chain/high_board_stocks 与 FOCUS 阶段判断，本方案在其上增量结构化，不重建 |

### 4. 需新建的最小增量

- `a_stock_st` 表（落 Tushare `stock_st` 接口，按 trade_date 存每日 ST 名单；历史自 20160101 起，每日 09:20 更新，单次≤1000 行需分页）。
- `universe_filter` 纯函数（白名单常量 + ST 判定 + 新股判定），信号侧/回测/QMT 三处复用。
- `limit_up_selected_stock`（一股一行，trade_date=T、target_trade_date=T+1）落表及其结构化扩展字段。
- QMT 侧执行引擎模块组（见第九章）。

> 样本起点提示：`a_stock_st` 历史自 20160101 起，回测区间早于 2016 年时 ST 过滤会不完整，须在结果标注样本起点。

## 三、策略链路总览

```
[选股 = 信号侧 / Linux / 现有 stock-ah-premium-ai]
  Tushare EOD 取数（基于 T 日涨停）
    → 多轮 LLM pipeline（limit-up-multi-stage-v3 / deepseek-v4-pro，约 08:31）
    → 【新增】确定性增强层：
         universe 硬过滤（去 ST、只主板+创业板）
         + 龙头强度打分（按 limit_type 分形态标准化）
         + 角色 / 战法 / 可成交性标注
         + 空仓闸门（情绪差 → market_state=空仓、候选全标放弃）
         + FOCUS 连板/溢价先验结构化（continuation_prob 等）
         + trade_cal 把 T 映射 T+1（写入时固化 target_trade_date）
    → 【新增】落表 limit_up_selected_stock（一股一行）+ 结构化 JSON
    → PushPlus 推送 HTML 报告（人读，不变）

         │  契约（唯一通信边界）：选股表 + 只读 JSON 接口
         ▼

[交易 = 执行侧 / Windows / QMT 独立模块]
  盘前加载当日 watchlist + market_state（空仓则只守仓不开新仓）
    → 9:15-9:25 自写定时器轮询 get_full_tick（竞价能力先实测）
    → 9:25 定盘/9:30 开盘：按战法标签 entry_router 建仓（一字/秒封买不进按现实处理）
    → 下单（T+1 买入，当日不可卖）
    → 持仓管理 / exit（守 T+1，T+2 起按先验 + 实时盘口做卖出决策）
  仓位与满仓由 QMT 账户内自决，风险自担

[回测 = 离线，只读选股表]
  把 watchlist 当「信号」评估：universe 过滤在信号构建阶段、join T+1 不复权行情
    → 信号收益分布 / 相对全涨停池增量 / 分组区分度（主板、创业板分桶）
    → 可插拔 entry-exit 规则分别回测
    → 一字/秒封硬剔除、空仓日记 0 留痕
```

关键不变量：T = 涨停数据日，**买入在 T+1（当日不可卖），最早 T+2 卖**，必须用 `a_trade_calendar` 映射；短线用**不复权**价；一字/秒封买不进剔除；竞价数据当前全为 None（已知缺口，竞价段由 QMT 实测补齐）。

## 四、股票池硬过滤

> 所有策略避开 ST/*ST/退市整理，只参与**主板 + 创业板**，排除科创板、北交所、新三板、B 股、CDR。该过滤同时作用于三处：**信号侧落表、回测样本构建、QMT 下单**，三处共用同一份 `universe_filter`、同一白名单常量、同一 ST 判定，禁止各自重写。

### 1. 单一实现，三处复用

无副作用纯函数 `universe_filter(ts_code, name, *, as_of_date, board_level=None) -> FilterResult`（`passed/reason/board/source`）。

| 调用点 | 触发时机 | 过滤层 |
|---|---|---|
| 信号侧落表 | 展开 selected_* → 写 `limit_up_selected_stock` 之前 | 前缀白名单 + ST/退市（T 日状态）+ 新股 |
| 回测样本构建 | 从全样本池构造可参与信号集时 | 同上三层（数据加载阶段仍保留 ST/退市全样本，过滤发生在信号构建阶段，防幸存者偏差） |
| QMT 下单前 | 盘前加载、生成可交易名单时 | 同上三层 + 实盘「账户权限可交易性」层 |

`as_of_date` 信号侧/回测传**信号日 T**，QMT 盘前传 T 日（ST 名单 09:20 已更新）。这是「按信号日 T 状态判定」的硬约束载体，**禁止用最新或 T+1 状态**。返回 `reason`/`source` 全程留痕便于审计。

### 2. 第一层 — 代码前缀白名单（保留集逻辑，非黑名单；对未来新号段天然安全）

```
PREFIX_WHITELIST = {
  "600","601","603","605",   # 沪市主板
  "000","001","002","003",   # 深市主板（含原中小板 002/003，2021-04 已并入主板）
  "300","301",               # 创业板
}
```

取 `ts_code` 去后缀的前 3 位；落在白名单内才保留，其余一律排除，明确覆盖：科创板 **688 + CDR 689**（含 689009 等存托凭证）、北交所旧段 **8xx（83/87/88/82）+ 新段 920**（2024-04-22 启用、2025-10-09 存量切换，白名单逻辑下跨切换不漏排）、新三板 400/430/830、B 股 200/900、基金/可转债等。二次校验用 `a_stock_basic.market`（以前缀为主、market 为辅）。

> 注：004 段官方列为深市主板号段但目前无普通流通标的；白名单仅保留 000/001/002/003，004 一旦出现需复核后再决定纳入，不要默认其被排除而埋漏纳风险。

### 3. 第二层 — ST/*ST/退市整理（按信号日 T 状态）

- **主源**：`a_stock_st`（Tushare `stock_st`，按 trade_date 取历史每日 ST 名单）。`as_of_date` 当日命中 → 排除，`source=stock_st`。这是**唯一能还原「信号日 T 当天 ST 状态」**的字段，避免用最新 name 造成已摘帽误排、已退市丢失的幸存者偏差。
- **退市/暂停**：`a_stock_basic.list_status ∈ {D,P}` 或 `delist_date <= as_of_date` → 排除。
- **兜底名称匹配**（仅主源缺数据时启用）：信号日快照 name 含 `*ST/ST/退` → 排除，`source=name_fallback` 留痕。
- **判定时点硬约束**：用信号日 T 名单，不用 T+1 或最新。
- **不可用的反推**：`limit_list_d` 不提供 ST 统计，禁止反推；`namechange` 接口 date 参数有已知缺陷（映射到 ann_date 且静默丢 NULL 早期记录），禁止依赖其 date 过滤，仅作本地全量拉取后的辅助校验。

### 4. 第三层 — 新股上市初期（前 5 交易日无涨跌幅）排除

全面注册制后全板块新股上市**前 5 个交易日不设涨跌幅**，第 6 日起恢复（主板 ±10%、创业板 ±20%）。前 5 日无明确「涨停/封板价」会污染封单/封板时间特征与「是否一字」判定。口径：排除「上市 < 6 个交易日」的次新，用 `list_date` + `a_trade_calendar` 按**交易日**计数（含首日为第 1 日），基准日 `as_of_date`，`reason=EXCLUDE_NEW_LISTING`。

### 5. 涨跌停按板取两档（去 ST、去科创后）

- 主板（600/601/603/605/000/001/002/003）：**±10%**。
- 创业板（300/301）：**±20%（20cm）**。

涨停价 = `round(前收 × (1+幅度), 2)`，幅度按 `board` 选 0.10/0.20。回测/实盘判定「是否涨停/一字/封单」**必须按 board 选幅度，禁止全市场统一 ±10%**。不再有 ST ±5% 与科创/北交分支，幅度收敛为两档。

### 6. 三处统一生效的关键纪律

- **信号侧/回测**：过滤发生在**信号构建**阶段而非数据加载；原始全样本（含历史 ST、已退市）完整入库，用于防幸存者偏差对照。
- **QMT 实盘**：在信号侧已过滤基础上**再过一遍**作下单前兜底；契约里名称含 ST/*ST/退 → 直接拒单告警。
- **实盘可交易性层（仅 QMT）**：回测必须含创业板（防偏差）；实盘创业板（300/301）需账户已开通创业板权限（满 2 年、前 20 交易日日均资产 ≥ 10 万、风险等级 C3+），未开通则下单前排除 300/301，依账户实际权限**动态决定**，不硬编码。

### 7. 过滤后样本影响

去掉 ST/科创/北交/新三板/次新后，打板池主要剩**主板 + 创业板强势股**，题材连板/接力主战场基本完整保留；被剔除的多是规则特殊/不可参与的尾部，样本更干净、可成交性与分桶口径更一致。创业板（20cm）样本占比不低（301 扩容），故分板分桶尤为必要。

## 五、信号侧空仓闸门

> 市场整体情绪不好时，LLM 输出侧直接执行「空仓策略」——当日不产出可参与候选，QMT 据此**不开新仓**。本闸门把现状写在 prompt 里让 LLM 自由文本判定的「退潮/冰点只观察」，变成**代码里的确定性字段**，可回测、可复现、不被 LLM 散文改写。

### 1. 它是什么 / 不是什么

- **通俗讲**：每天 08:31 先量「今天打板情绪好不好」。好 → 正常出候选；一般 → 出候选但整体降级、只挑最硬的；差 → **当天一只都不让买**（候选全标「放弃观察」）。QMT 读到「情绪差/无可参与候选」当天不开新仓，但**已持仓照常按卖出逻辑管理**。
- **确定性闸门，非 LLM 建议**：判定全部由 Python 用 `emotion_cycle` 的**数值**跑死规则得出（同输入同输出），叠加在 LLM pipeline **之后**做后处理；LLM 散文阶段只负责「写报告给人看」，无权推翻闸门枚举。
- **职责硬边界**：信号侧绝不产仓位/买卖指令；闸门只关「开不开新仓这一道门」，不给「该买多少仓位」。产物是**情绪枚举 + 路由动作**，不是仓位数字。

### 2. 判定输入（全部已存在，不依赖任何 None 字段）

只用 `market_emotion` 及内嵌 `emotion_cycle` 的盘后数值：涨停家数 `limit_up_count`、炸板率 `broken_board_rate_pct`、跌停家数 `limit_down_count`、最高板 `highest_chain` 及其变化 `highest_chain_change`、昨涨停今溢价 `prev_limit_up_premium.avg_pct_chg`/高开率、各级晋级率 `advancement[...]`。全部是 T 日盘后快照，与回测铁律一致。

### 3. 数值 → 情绪六档（确定性分类器）

新增分类器把数值映射到现有契约六档 `sentiment_cycle ∈ {冰点, 启动, 发酵, 高潮, 分歧, 退潮}`（注意是六档情绪周期，不是三档；三档是另一字段 `action`）。两步判定：先看硬否决，再看综合健康度。所有阈值按 board 分桶、标 TBD、须回测校准并以 `threshold_version` 固化。

**硬否决（命中任一 → 判退潮/冰点，无可参与候选）**（主板先验，创业板单列；均 TBD）：

| 否决项 | 主板先验 | 依据 |
|---|---|---|
| 涨停家数过少 | `limit_up_count < 20` | 赚钱效应枯竭 |
| 炸板率过高 | `broken_board_rate_pct > 50%` | 封不住板、追高必埋 |
| 跌停放大 + 大票下杀 | `limit_down_count > 30` 且大票大跌 | 系统性恐慌 |
| 最高板掉头 | `highest_chain_change` 较昨下降 ≥ N 级且昨日 `highest_chain ≥ 5`（N 由本地数据标定，避免正常梯队切换误判） | 高度坍塌 |
| 昨涨停今溢价「必亏档」 | `avg_pct_chg <= -5%`（用「必亏档」而非「转负」，避免把仅「分歧」误判退潮） | 接力必亏 |

### 4. 六档 → 「可参与/不可参与」映射（单调一致 + 一致性断言守护）

| 情绪六档 | 当日候选产物 | QMT 行为 |
|---|---|---|
| **冰点 / 退潮** | 候选全部 `action=放弃观察` | 不开新仓 |
| **分歧** | 候选保留但整体降级（多 `谨慎观察`）；方向**待 T+1 盘中证实**，作可降级先验默认 | 收紧、少开、只开最硬的 |
| **启动 / 发酵** | 正常出候选 | 正常按候选下单 |
| **高潮** | 出候选但**叠加约束**：追高风险最高处，倾向只允许断板反包/低吸、对追涨停板候选降级 | 谨慎放行、警惕过热 |

三点采纳自核验：① 高潮不等于无差别参与（总龙头筑顶、追高必埋）；② 分歧方向只在盘中产生，EOD 给「谨慎」作先验、标注待 T+1 证实；③ 溢价用「必亏档」而非「转负」作硬否决。

### 5. 空仓日产物形态（留痕 + 不开新仓 + 仍卖出）

1. **仍跑、仍落表留痕**：T 日盘后仍生成候选并写表，`sentiment_cycle=退潮/冰点`、每行 `action=放弃观察`。**不是空表**——回测/复盘可回答「空仓日如果买了会怎样」，验证闸门是否误杀。（工程取舍：纯省 token 时可在入口算情绪、命中空仓即短路 LLM 阶段、只写日级留痕一行；「是否逐股落表」作为由回测需求驱动的可配置项。）
2. **QMT 侧**：盘前读到无可参与候选 → 当日不提交任何买入单，进「守仓模式」。
3. **持仓管理照旧**：空仓闸门只关开新仓，不关卖出。**空仓 = 不开新仓 ≠ 清仓**；减/清仓由个股卖出信号或独立系统性风险开关决定。
4. **幂等**：同输入同输出（threshold_version 不变即结论不变）。

`sentiment_cycle` 同时作打分的周期系数（高潮放大高度权重、退潮/分歧压制高位股得分）与战法 action 的条件变量。它与空仓闸门从同一组数值、同一套阈值派生，靠单调一致映射 + 一致性断言保证不矛盾。

## 六、龙头战法增强选股层

### 定位与边界

叠加在现有 pipeline 之上的**确定性「可计算特征 + 龙头强度打分 + 可成交性标注」层**：复用 LLM 已选出的 `selected_*_stocks`（不重新选股、不做买卖/仓位决策），输出每只 LLM 选中股一行的结构化记录，落表供回测与 QMT 消费。

**相对 LLM 既有产物的增量**：现有 pipeline 已对每股产出 `score_detail`（强/中/弱定性）+ priority + selection_reason。本层增量不是「再打一次分」，而是：(1) 把定性转为**可复算数值分**，跨股横向可比、可回测校准；(2) 消除 LLM 主观漂移；(3) 补 LLM 散文里有但未结构化的维度（梯队位置/位置风险/可成交性）。

### 硬约束

- `open_times / bid_* / real_time_pct_chg / open_time` 真实**全为 None** → 依赖盘中炸板/竞价强度的维度必须做缺失处理。
- `seal_ratio_pct` 在本项目实为**封流比（封单/流通市值）而非封成比**，落表须在列注释固化公式；它对一字/缩量板天然偏高，**严禁跨 limit_type 直接比大小**，必须分形态标准化。
- `board_level` 与续板概率**非单调**，必须分桶。
- 权重/阈值全为经验先验，须样本外回测校准，不当固定参数。

### 1. 龙头强度打分卡（维度 + 方向 + 标准化口径，权重 TBD）

**不钉死小数权重**。实际权重由样本外 IC/IC_IR 或带正则拟合产生；更稳妥的是**分桶/分域或树模型让交互项自己长出来**——这些维度本质是异质、条件化、强缺失因子，固定线性权重几乎必然失真。

| 维度 | 关键字段 | 方向 | 标准化/分桶口径 |
|---|---|---|---|
| 封板质量 | seal_ratio（封流比）、封单回落=1−limit_order/max_limit_order、first_limit_time、last−first | 正/负混合 | **同 limit_type 桶内分位归一**；一字/T字/换手分桶 |
| 题材梯队 | 同题材涨停家数、连板天梯完整度、题材内相对高度排名 | 正 | 容量分 + 梯队完整度（断层=压分）+ 龙地位 |
| 高度 | board_level | 非单调 | **分桶非线性**；高位优势靠情绪周期系数条件化，不写死拐点 |
| 资金 | top_list 净买额/circ_mv、席位类型 | 不对称 | **弱权重，以净卖出做负向过滤为主** |
| 位置筹码（负） | return_5d/10d、对 ma5/10/20 乖离、winner_rate、上方压力%、集中度 | 负（风险扣分） | 高位+winner 饱和+大乖离=高风险 |
| 辨识度 | 题材内是否唯一最高板 | 正 | 唯一最高=强；并列=弱 |

**资金维度口径**：净买入仅在极短持有期、且限于特定强势游资席位时才有正向信号，持有期一拉长即被反转吃掉且有幸存者偏差。落地为：龙虎榜拆两特征——「净卖出/机构净卖出」作硬性**负向过滤**，「净买入额/circ_mv + 是否知名游资席位」仅作**弱正向辅助分**且与其他因子交互，禁止单独线性加权。

**综合分结构**：`封板质量正分 + 题材梯队正分 + 辨识度正分 − 位置筹码风险分`，再乘**情绪周期系数**，最后用**可成交性硬标记**过滤。

### 2. 角色标签（多标签 + 置信）

`board_level > first_limit_time > limit_order` 三级排序**只用于题材内选龙头**，不能区分中军/卡位/助攻/杂毛，不可当 6 分类器；角色可重叠、动态变化，应**多标签 + 每角色置信度**：

| 角色 | 可判性 | 判定口径 |
|---|---|---|
| 总龙头/板块龙头 | 规则可判 | 题材内 board_level 最高 → first_limit_time 最早 → limit_order 最大 |
| 杂毛（跟风） | 规则可判 | 题材内低板 + 晚封 + 小封单 + 不在梯队 + 封板质量分低 |
| 中军 | 规则不足，标待定 | 需基本面/机构主导，pipeline 无这些字段，纯 circ_mv 常误标 |
| 卡位/助攻 | 规则不可靠，保留 LLM 辅助 | 盘中时序/博弈行为，盘后快照无法重建 |

中军/卡位/助攻保留 LLM `selection` 文本辅助或标「待定」。

### 3. 战法适配标签（分两层，供 QMT 路由；本层不做买卖决策）

主流短线只有「打板/半路/低吸」三大流派；「打板跟买」与「竞价高开追」是同一打板家族不同时点子模式。两层：

- `strategy_family ∈ {打板, 半路, 低吸}`；可加 `entry_timing ∈ {竞价, 盘中, 收盘}`、`setup ∈ {首板, 接力, 龙回头, 弱转强}`。
- `action ∈ {重点, 谨慎观察, 放弃}`——直接复用项目已有三档。

「竞价高开追」依赖 9:15-9:25 竞价数据（当前全为 None），标注为**待数据补齐后启用**，不硬编进路由。

### 三、写入时机 / 幂等

- **切入点**：`LimitUpPushService.ensure_analysis_for_trade_date` 的 `status=READY` 收口段（设置 content_html/context_json/status=READY/generated_at 之后、commit 之前）调用新增 `_persist_selected_stocks(analysis, context)`。
- **原子性**：该区间无中间 commit，选股行可与 READY 翻转**原子提交**（准确表述是「与 READY 状态翻转原子提交」，非「整条 pipeline 单事务」）。
- **幂等**：唯一键 upsert + 整组 delete-then-insert；force 重生成走同一收口；同交易日命中已有 READY 早退时不经收口段，需补「该 source_analysis_id 已有行则跳过」守卫。
- **写入失败降级**：选股写入异常 try/except 降级 + 留痕，不阻断报告 READY。
- **advice 降级日**：watchlist 源自 advice 之前的 `pipeline.selected_*`，advice 失败仅置 `advice_degraded=1` 留痕，选股仍完整可回测。

### 四、衔接行情（回测 join 次日）

落库时 `target_trade_date` 由 `a_trade_calendar` 计算（`cal_date > trade_date AND is_open=1` 的最小值）写入时固化。

```sql
SELECT w.trade_date, w.ts_code, w.tier, w.leader_strength_score,
       q.open AS buy_open, q.close AS t1_close, q.pct_chg AS t1_pct
FROM limit_up_selected_stock w
JOIN tencent_unadjusted_daily_quote q
  ON q.ts_code = w.ts_code AND q.trade_date = w.target_trade_date AND q.adjust_type = 'NONE'
WHERE w.prompt_version = 'limit-up-multi-stage-v3';
```

口径：优先 `tencent_unadjusted_daily_quote(adjust_type='NONE')`，缺则回落 `a_daily_quote`（差异须标注）。**除权日边界**：T+1 逢除权除息日时收益在 T+1 当日内算（开盘→收盘），不跨除权日做差。

## 八、连板可能性与次日卖出决策

> 直接回答：选中股票买入后「连板可能性」「第二天是否卖出」放哪、是否要在 LLM 侧新开模块。

### 一句话结论

**拆成两段，分属两侧，不新开独立 LLM pipeline：**

- **连板/次日溢价「先验」** → 放 LLM 信号侧 EOD，复用现有 FOCUS 阶段，把已在产出的 HTML 散文判断**结构化成机读字段**落表。
- **「当天到底卖不卖」的最终决策** → 放 QMT 执行侧的实时模块（新建），纯规则、不含 LLM、盘中实时跑。

两段职责物理隔离，只通过 `limit_up_selected_stock` 契约通信。

### 一、为什么不在 LLM 侧新开模块

代码已证实：`_chain_focus_prompt`、`_high_board_focus_prompt` 这两个 FOCUS 阶段**已经在让 LLM 判断**「晋级三板/四板可能性、次日溢价、触发条件、失败条件、筹码压力、次日竞价观察清单」，只是结果以 `*_focus_html` 散文塞进 `context_json.pipeline`，机器无法消费。三点论证：判断逻辑已存在（新建是重复算力）、输入完全相同（无新数据要拉）、新建会制造口径冲突。

**实现说明**：这两个 FOCUS 阶段当前走 `_run_text_stage`（json_mode=False）产 HTML。机读化**不是「同一次调用额外吐一段独立 JSON」**——DeepSeek 的 `response_format=json_object` 要求整条消息为单一 JSON。正确做法是把 FOCUS 改造为 **JSON-mode 输出**，在同一 JSON 对象内同时承载结构化先验字段和一个 `html_fragment` 供报告渲染；并把 `_run_text_stage` 现有的「FOCUS 失败可降级、报告仍 READY」容错等价迁移到 JSON 阶段（保留同等兜底，避免 READY 率回退）。

### 二、信号侧落哪些先验字段（结构化、机读、可比对）

| 字段 | 类型 | 含义 |
|---|---|---|
| `continuation_prob` | **枚举档位**（高/中/低/极低） | 次日续板可能性。**用档位不用裸概率**——续板是低基率事件（外部统计：进二板约 15.7%、二进三约 34.65%，五板以上才过 50%），LLM 在低基率事件上过度自信、校准差，吐 0-1 裸概率是伪精度；须用 `limit_step`/`limit_list_d` 按板块 × 情绪周期分桶自校准 |
| `next_day_premium_prob` | 枚举档位 | 次日溢价可能性，与续板**分开**（很多票不再封板但仍能冲高止盈） |
| `boost_conditions` | 结构化列表 | 加分/触发条件，如 `{signal:"竞价高开", op:">=", value:5, unit:"%"}`、`{signal:"秒板封住"}` |
| `fail_conditions` | 结构化列表 | 失败/放弃条件，如 `{signal:"竞价高开", op:"<", value:2}`、`{signal:"破开盘价"}`、`{signal:"炸板"}` |
| `suggested_hold_thesis` | 短文本 + 标签 | 持有逻辑（"题材总龙头赌主升续板" / "孤位高位板只吃次日高开"），供 QMT 选状态机分支 |

口径：① 续板与溢价两档独立、按 board_level 分桶给阈；② 这些是 **T 日盘后 EOD 先验**，须显式标注，不能误用为盘中决策；③ 守契约硬边界——是先验与条件，**不是买卖指令、不含价位/仓位**。

**持仓股不需专门维护先验表**：报告每天基于上一交易日涨停池全量刷新——某持仓股若 T+1 又涨停 → T+1 盘后作为新涨停股重进 T+2 报告、自动拿刷新先验；若 T+1 没涨停 → 已断板、不在新报告中，**转为纯 QMT 技术退出**（破位止损/均线离场）。这是「先验每日刷新、断板即交回技术面」的自洽闭环。

### 三、为什么「卖不卖」必须放 QMT 侧

1. **决定性信息只在盘中产生**：能否兑现取决于 9:25 定盘的竞价高开/量能、开盘是否秒板、分时强弱、是否破开盘价、盘中是否炸板、封单是否被砸开——这些在 T 日盘后快照里全是 None，08:31 批处理时**物理上还不存在**。
2. **时序对不上**：报告开盘前推送，而「卖不卖」要在早盘竞价 5 分钟窗口实时决断。
3. **职责分离更稳**：LLM 给「先验 + 触发/失败条件」（规则，可回测），QMT 拿实时行情**校验条件**并真正下卖单（执行，可实时）。

执行侧定位为**确定性、低延迟、可回测规则/状态机，盘中默认不依赖外部 LLM 在线推理**（不进下单关键路径）。

### 四、完整时间线（澄清「第二天」是哪天）

```
T   (涨停日)         盘后跑出涨停数据
T盘后/T+1早08:31     报告基于 T 数据跑批 → T+1 开盘前推送（含 FOCUS 连板先验，target_trade_date=T+1）
T+1 (买入日)         QMT 打板买入  ←★ A股 T+1：买入当日【不可卖】
T+1盘后/T+2早        新报告基于 T+1 数据再跑批 → 刷新先验
                       · 若 T+1 又涨停 → 重进报告拿刷新先验
                       · 若 T+1 没涨停 → 不在新报告，转纯 QMT 技术退出
T+2 (最早可卖日)     ←★ 这才是用户口中的「第二天卖不卖」
                       QMT 状态机用【T+1盘后刷新先验 + T+2 实时竞价/分时】做卖出决策
```

**澄清**：对买入头寸而言，「第二天卖/隔日卖」= 持有后的下一交易日 = 买入日（T+1）的次日 = 相对最初涨停日 T 的 **T+2**。绝不是 T+1（T+1 是买入日，当天卖不了）。跨周末/节假日用 `a_trade_calendar` 映射，**不能按自然日 +1**。

> 时序补充：同一份 EOD 先验里有两条时序链——FOCUS 对一只票预测的「次日溢价」，其「次日」指**涨停日 T 的次日（即 T+1，正是该票被打板买入当天）**；而「为我们 T+1 买入的票在 T+2 决策卖出」是另一条链。工程上高频错误是把代表**买入日**的 `target_trade_date=T+1` 误读成卖出日；代码/字段须区分「买入日 = T+1」与「最早可卖日 = T+2」。

### 五、持仓-决策数据流（大白话）

每天收盘后，AI 把「哪些涨停股明天可能继续涨停、要满足什么条件才看好、出现什么信号就要跑」从只能给人看的报告文字，变成机器能直接读的一行行字段存进表。第二天开盘，QMT 先把表读进来，再盯真实开盘价、有没有秒板、分时强不强、有没有炸板——拿「AI 昨晚定的条件」去对「今天盘面真实发生的事」，自动决定这只票继续拿着赌下一个板、还是冲高赶紧卖、还是开盘就走、还是跌破止损。**AI 负责「事前怎么看」，QMT 负责「当场怎么办」。**

### 六、客观能力限制

- 连板续板是低胜率/低基率事件，`continuation_prob` 须配套校准与回测标定，不能把 LLM 散文直接当概率。
- 卖出触发所需的竞价量能/封单动态/炸板，现有 `realtime_quote_snapshot`（仅 last_price）承载不了，须由 QMT `get_full_tick` 轮询补齐。
- 民间经验阈值（进二板约 15%-20% 等）只作量级参考，须用自有数据自校准。

## 九、QMT 侧执行引擎

> 面向不懂 QMT 的读者，用「如果…就…」讲清每天怎么跑。QMT = `xtdata`（行情）+ `xttrader`（下单），Windows-only，跑在独立 VPS；只消费 watchlist + 自采实时行情，自己决定择时、价位、仓位，承担全部交易风险。

### 1. 位置与边界

信号侧只产信号（绝不产仓位/买卖价/买卖量/止盈止损）；执行侧自己决定择时/价位/仓位/止盈止损。**契约绝不包含仓位/资金分配、买卖指令、止盈止损价位、实时/竞价数据。** QMT 永远不从契约读「该买多少/什么价/止损价」。

### 2. 股票池硬过滤（QMT 是最后一道闸门）

下单前再跑一次与信号侧**同一个** `universe_filter`：前缀不在白名单 → 拒单；名称含 ST/*ST/退 → 拒单告警；账户未开通创业板权限 → 过滤 300/301。

### 3. 怎么收到空仓信号

- 盘前拉到的 watchlist 无可参与候选（全标放弃、`market_state=空仓`）→ 当日不开任何新仓，进「守仓模式」（继续监控既有持仓、执行卖出、盘前盘后对账）。
- 本来就空仓（既无持仓又无候选）→ 整天只做心跳健康检查与对账。
- 空仓 = **不开新仓 ≠ 清仓**；已有持仓照常按卖出逻辑管理。

### 4. 一天的完整时间线状态机

| 状态 | 时段 | 通俗逻辑 |
|---|---|---|
| **PREMARKET** 盘前 | ~8:45-9:14 | 取 watchlist + 情绪枚举；对每只候选用 `close` 按 `board` 预算今日涨停价（主板±10%/创业板±20%，**禁止全市场套 10%**）、合理高开区间缓存；`tradable_flag=false`（一字/封死）进观察名单不进可交易名单；无可参与候选则置「禁开新仓」；健康检查（xttrader/账户/xtdata/MySQL/交易日历），任一不通按「安全默认」处理 |
| **AUCTION** 集合竞价 | 9:15-9:25 | 自写定时器（每 1-3 秒）轮询 `get_full_tick`，不依赖回调；监控竞价高开/竞价量能（与首板爆量比例对照）/分时重心/虚拟封单；9:15-9:20 可撤、9:20-9:25 不可撤、9:25:00 定盘；竞价数据实测不可得则降级为只看 last_price + 开盘后确认 |
| **OPEN_DECISION** 定盘+开盘 | 9:25 / 9:30 | 拿开盘价后按 `strategy_family + action` 调 entry_router 选规则；9:30 后用真实分时确认触发条件，满足则建仓、不满足则放弃/转观察；仓位由 QMT 自定 |
| **INTRADAY** 盘中 | 9:30-14:57 | 持仓监控（分时强弱/是否炸板/封单是否被砸开/是否破开盘价或前日低点）；对**可卖持仓**（T+2 起）执行卖出；当日新建仓位守 T+1 不参与卖出判断；触发风控则进 risk 限制态 |
| **PRECLOSE** 尾盘 | 14:57-15:00 | 完成「今日必须了结」的卖出；取消所有未成交挂单（避免隔夜悬挂）；核对预期与实际持仓 |
| **POSTMARKET** 盘后 | 15:00 后 | 查当日成交/持仓/可用资金，与本地台账逐笔对账，不一致告警；算已实现盈亏与隔夜持仓清单（供次日判断哪些 T+2 可卖）；生成运行报告 |

### 5. 建仓 entry 规则（按 `strategy_family + setup + action` 路由）

`action=放弃观察` 一律不开仓。价位/数量全部 QMT 自定。

| 战法分支 | 如果…就买 | 如果…就弃 |
|---|---|---|
| **打板跟买/竞价强开追**（打板+重点） | 竞价强（高开合理区间 + 竞价量能达首板爆量足够比例 + 分时重心平稳）→ 以接近涨停价限价排队冲板跟买 | 竞价弱/量价背离/开盘走弱 → 放弃 |
| **均线低吸**（低吸+中军+谨慎） | 开盘回踩 ma5/ma10 缩量企稳、不破关键均线 → 限价低吸 | 破位（穿 ma10/前日低点）或放量下杀 → 不接 |
| **高开追/半路**（半路+卡位） | 竞价确认承接、开盘量价配合突破 → 跟入 | 缩量低开或冲高即回落 → 放弃 |
| **龙回头低吸**（低吸+龙回头） | 回踩 10 日线后放量突破 → 买 | 放天量疑出货、突破失败回落 → 放弃 |

「竞价高开追」强依赖竞价数据，实测可得前不上线，退化为「开盘后确认」版本。以上是**初始策略空间而非穷尽枚举**，阈值须回测标定。

### 5.1 一字/秒封「买不进」的现实处理（重点做实）

- 开盘即一字板（涨停价无卖盘）→ 判「买不进」，按「挂涨停价排队 + 设最长等待时限」，超时仍无成交则撤单。
- 秒封（开盘瞬间封死）→ 同样排队；排队靠后、估算成交概率低 → 撤单转同梯队次优标的或当日放弃。
- **绝不**用「以为买到了」计入持仓——**只有 xttrader 实际成交回报才认定建仓成功**（与回测口径一致）。

### 6. 卖出 exit 规则（守 T+1，次日竞价/分时定夺）

**铁律 T+1**：当日买入当日绝不可卖；exit 只对「昨日及更早建仓」的持仓生效（对买入头寸是最早 T+2 卖）。「第二天卖」= 买入日的次日，绝不误算成买入当天。止盈/止损价位由 QMT 自定。

- **次日竞价定夺（9:15-9:25 轮询 → 9:25 定盘）**：高开强且量价匹配 → 续持等分时；高开但量价背离明显 → 倾向竞价或开盘即减/清。
- **次日分时定夺（9:30 后）**：秒板续持（开盘秒板/快速封死且封单扎实 → 继续持）；冲高止盈（达目标但封板乏力/上方压力大 → 止盈）；走弱破位止损（有效跌破开盘价/前日低点或分时持续转弱 → 严格止损）；炸板出（封后被砸开 → 强离场）；烂板出（反复开合、封单薄、封流比掉得快 → 倾向出）；冲高不及预期出（全天达不到溢价目标 → 尾盘前了结）。

> 续板是低胜率事件（进二板约 15%-20%、二进三约 35%、五板以上才升到 50%+），当日能否兑现取决于盘中实时信息——这正是「卖不卖」必须盘中实时定的根因。

### 7. 风控与工程

- **账户隔离**：QMT 跑独立 VPS 与独立资金账户，与信号侧物理隔离；QMT 绝不持有信号侧写权限（只读契约）。
- **下单幂等/防重**：每笔下单意图生成唯一业务单号（日期+ts_code+战法+序号），下单前查本地台账，已存在则拒绝重复下单；进程重启先从 xttrader 拉真实委托/成交 + 本地台账对账重建状态；**只认 xttrader 成交回报才算建仓/平仓**。
- **撤单**：每个挂单带最长存活时限，超时未成交撤单；尾盘统一撤未成交单。
- **网络/行情中断兜底**：xtdata 中断 → 暂停依赖实时价的判断进「冻结态」，恢复后重对齐；xttrader 链路中断 → 停发新单告警；取契约失败 → 当日按「无 watchlist = 禁开新仓」只守仓。一切兜底遵循「安全默认」：不确定时宁可不交易。

### 8. 模块划分（`qmt_strategy/`）

| 模块 | 职责 |
|---|---|
| `watchlist_loader.py` | 盘前取整批候选；白名单 + ST 二次校验；按 tradable_flag 拆可交易/观察；用 close 预算涨停价；判断空仓闸门 |
| `auction_poller.py` | 9:15-9:25 轮询 get_full_tick，产竞价高开/量能/重心/虚拟封单；竞价不可用降级 |
| `entry_router.py` | 按 strategy_family+setup+action 路由 entry 规则，只出决策（开/弃/转标的），不直接下单 |
| `order_executor.py` | xttrader 下单封装；唯一业务单号、幂等防重、限价/排队/撤单、成交回报确认、本地台账、账户隔离 |
| `position_manager.py` | 持仓状态机；守 T+1 标记每笔「最早可卖日」；对可卖持仓执行 exit；尾盘了结/撤单；隔夜持仓清单 |
| `risk.py` | 账户级/单票级风控阈值、行情/下单中断冻结态、空仓闸门联动、安全默认裁决 |
| `logger.py` | 结构化运行日志、下单/成交/撤单/异常留痕、当日对账与运行报告 |

**QMT 怎么用先验**：`position_manager` 拿契约的 `continuation_prob`/`next_day_premium_prob` 与 `boost_conditions`/`fail_conditions` 作「今天重点盯哪些、按什么条件卖」的先验输入；真正的卖/留由 `auction_poller` + 盘中实时行情按第 6 节触发条件裁决。**先验决定「倾向」，实时决定「扣扳机」。**

## 十、回测重构与口径补充

### 1. 回测目标（核心命题）

**评估 watchlist 作为「信号」的 alpha 质量，而非某个仓位方案的收益。** 核心命题：LLM 高分组相对同日全市场涨停池（`limit_list_d`）对照组，是否有显著、稳定、可执行的增量；按「层 × 龙头分桶 × 角色 × 情绪周期」分组后是否有区分度（高分组显著优于低分组）。**区分度才是 alpha 证据，绝对收益不是。** 非目标：不回测仓位大小、不给仓位建议、不做全仓/分散对比。

### 2. 评估指标

- **信号收益分布**（T+1开/T+1收/T+2/T+3/T+5/T+10 各一套）：均值/中位数/标准差、P10/P25/P75/P90 分位、命中率、盈亏比、期望、最大单笔回撤、左尾 5% 平均亏损、一字/不可成交占比。强调「分布」而非「均值」。
- **相对对照组增量**：以 `limit_list_d` 为基准算超额/命中率增量/盈亏比增量；**IC/RankIC**（leader_strength_score 与收益的截面相关）及 IR、t 值；分层多空 spread 及显著性。
- **分组区分度（核心）**：按层 × 龙头分桶 × 角色 × 情绪周期及交叉，检验各组收益/命中率/样本量、单调性、显著性（t 检验，样本不足/非正态用 Mann-Whitney，报 p 值 + 效应量）；情绪周期条件化（同分桶高潮 vs 退潮收益差）。
- **可执行性/口径敏感性**：`v_exec.0/1/2 × v_cost.0/1/2` 各组合下的收益对照、不可成交率、平均滑点、冲击成本占比。

### 3. 可插拔 entry-exit

`SignalSource（选股落表）+ EntryRule/ExitRule 插件 + MatchingEngine（内建约束）+ Evaluator`。信号固定、规则可换，便于 QMT 侧独立迭代。规则只读信号 + 当日及之前行情，不读未来数据。日线先行（竞价/分时类规则用开盘价/最高最低价近似并标低保真），分钟权限到位后升级触发精度。

### 4. 真实约束（内建撮合引擎）

| 约束 | 口径 |
|---|---|
| T→T+1 映射 | trade_date=T，买入在 T+1，`a_trade_calendar` 映射；**禁止用 T 日价当买入价（头号陷阱）**，禁按自然日 +1 |
| 不复权 | 取原始价不用复权因子；T+1 逢除权则收益在 T+1 当日内算 |
| T+1 卖出 | T+1 当日不可卖，多日持有从 T+2 起算卖出腿 |
| 无量跌停顺延 | 卖出腿遇跌停封死无量 → 顺延，记录实际卖出日 |
| 一字买不进 | T+1 一字/封死无量 → 不可成交（见下） |
| 涨跌停制度 | 主板±10%、创业板±20%（无 ST、无科创/北交） |

### 5. 回测口径补充（硬增量）

**(a) 秒封/一字「买不进」必须从收益剔除（`v_exec` 头号硬规则，做实）**。判定 T+1 买不进命中任一即剔除（**不计任何收益，既不算盈也不算亏，视为未成交**）：① `limit_type` 一字；② T+1 `high = low = 涨停价`（全天封死无成交区间）；③ 开盘即涨停且全天未打开（T+1 `open = high = low = 涨停价`）。分级：`v_exec.0` 全部按 T+1 开盘无条件成交（仅作收益上界）；`v_exec.1` 命中即剔除；`v_exec.2` 对非一字强势板按 limit_type + 实际换手/盘中打开分级成交概率/部分成交（不靠封流比单阈值二元开关）。每次回测输出「被一字/秒封剔除占比」，量化「买不进」虚高贡献。日线无法区分盘中秒封时刻，判定 ②③ 是保守近似，分钟权限到位用 `stk_mins` 升级；**日线口径下宁可保守剔除，不可放进收益**。

**(b) 空仓日如实体现（记 0 留痕，不偷偷跳过）**。空仓闸门生效时（退潮/冰点、无可参与候选），空仓日在时间轴**保留为一个交易日条目**，当日买入腿收益 = 0、记空仓标记、**计入年化/回撤/胜率的分母**。**严禁从样本静默删除**（会人为抬高平均收益与胜率，幸存式择时偏差）。输出三项：空仓天数/占比/**空仓日若强行参与的反事实收益**（反事实为负则闸门有效）。空仓日只禁新开仓，对昨日及更早持仓的 exit 仍照常执行并计入收益。

**(c) 涨跌停按板取值**：所有涉及涨跌停的判定（一字判定阈值、卖出腿跌停顺延、打板排板价）按 board 选幅度，**禁止统一 ±10%**。

**(d) 主板 vs 创业板分桶**：创业板 ±20% 单日空间翻倍、题材持续性/连板节奏/炸板率/次日溢价分布与主板不同。`board_level` 分桶、龙头分位分桶、各类阈值**主板与创业板分开统计**；分组区分度检验在两板内各做一遍再跨板对比。**顺序硬约束：先做 (a) 可成交性硬过滤，再分桶**，否则创业板一字板封流比天然偏高会污染分位分桶。

**(e) 同口径 + 不复权 + 全样本**：实盘「只认 xttrader 成交回报」与回测「一字/秒封不计收益」是同一套可成交性判定；不复权用 `tencent_unadjusted_daily_quote`；含退市/ST 全样本防幸存者偏差，过滤在信号构建阶段调用 `universe_filter`。

### 6. 验收要点

- 买入腿全部取 trade_cal 映射后 T+1 行情，无一例用 T 日价（边界样本含节假日/停牌验证）。
- 一字/秒封：`v_exec.1+` 三条判定全生效并输出被剔除占比；含一字板样本验证确无收益泄漏。
- 空仓日：时间轴留存、收益 = 0、计入分母；删除任一空仓日会改变年化/胜率（反向验证未被静默跳过）。
- 涨跌停：抽样校验主板 ±10%、创业板 ±20%，无 ST/科创分支命中。
- 分桶：主板桶与创业板桶各自独立产出区分度结论，且分桶在可成交性过滤之后。
- 能产出 watchlist vs `limit_list_d` 的超额/IC/RankIC 及显著性、分组区分度（单调性 + p 值 + 效应量）；结论在可执行口径（`v_exec.2+v_cost.2`）下复核，明确区分「乐观上界 alpha」与「可执行 alpha」。
- 现状提示：项目当前无龙头/打板短线回测引擎（仅有分红再投长持回测），本章是需内建的设计，不是已生效实现。

## 十一、量化复盘看板（交易复盘与信号归因）

> **权威详档另存**：本节是概要 + 结论。完整设计（含全部 DDL、API 端点、收益率公式、归因 SQL）已落在实际代码项目 `stock-ah-premium-ai/resources/doc/` 的三篇文档，以那三篇为准、本节只做索引与要点：
> - `qmt-trade-review-design.md`（QMT 数据回流 + 复盘总体）
> - `qmt-trade-review-board-design.md`（服务侧复盘看板）
> - `qmt-trade-review-closed-loop-attribution-design.md`（信号-执行-结果闭环归因）

### 1. 目标与边界

在信号侧 Linux 现有项目（`stock-ah-premium-ai`，FastAPI + React + MySQL）新增**量化复盘看板**，回看执行侧 QMT 实际交易了什么、收益率如何，对**当日**与**历史**交易做统计、复盘与信号归因。沿用解耦架构：**信号侧只读展示**（读 `qmt_*` 表与只读视图，不直连 QMT、不下单）；**执行侧 Windows QMT 只写**（回调落明细 + 收盘落快照）。

### 2. QMT 能拿到什么 / 拿不到什么（决定哪些要自己算）

**直接能拿（仅当日）**：`query_stock_asset`（资产：cash/frozen_cash/market_value/total_asset，仅 6 字段）、`query_stock_positions`（持仓：volume/can_use_volume/open_price/market_value）、`query_stock_orders`（当日委托）、`query_stock_trades`（当日成交：traded_id/order_id/traded_price/traded_volume/traded_amount/traded_time）；以及 `on_stock_trade`/`on_stock_order`/`on_order_error` 等回调（**废单/拒单是回调独有，query 无等价物**）。

**拿不到、必须自落快照 / 自算**（这是看板架构的根因）：
- **历史跨日成交/委托、历史每日净值曲线**——`query_*` 只返当日、隔日清空、无历史区间接口 → **执行侧每个交易日盘后主动落快照**才能还原历史。
- **已实现盈亏 / 当日盈亏 / 账户收益率**——原生 API 不给，须用成交流水撮合 + 资产快照差分自算。
- **浮动盈亏 / 持仓现价**——原生 `XtPosition` 不含（社区文章里的 `float_profit`/`last_price` 是 KhQuant 等第三方封装补充字段，**不得假定原生存在**；新版若有则优先取、并保留"现价×量"自算兜底），用 `realtime_quote_snapshot.last_price × volume − 成本` 自算。
- **出入金流水**——QMT 交易 API 完全不提供 → 必须自建出入金台账（人工/对账单导入），否则净值/收益率被资金进出污染。

### 3. 数据回流与新增表（DDL 详见权威详档）

执行侧把数据写回 Linux MySQL，**优先经信号侧带鉴权的内网写接口（FastAPI /ingest）**，仅在可信内网 + DB 仅内网监听 + 白名单时才用"Windows 直连 MySQL"备选。新增四表（均 upsert 幂等）：`qmt_trade`（成交明细）、`qmt_order`（委托，含废单/拒单）、`qmt_position_snapshot`（按日持仓快照，CLOSE 为历史唯一来源）、`qmt_account_daily`（账户资产日快照 + 出入金台账）。
- **唯一键纳入 `trade_date`**（柜台 `order_id`/`traded_id` 不保证跨日唯一，否则次日复用 ID 会静默覆盖）。
- **与 `limit_up_selected_stock` 关联用不可变的 `signal_trade_date=T`**（下单时 `order_remark` 携带"信号日+短id"确定性写入），**不用 `target_trade_date=T+1`**（可变/派生，实际成交可能落 T+2，INNER JOIN 会丢行）；计划侧用 LEFT JOIN 保住"买不进"的票。

### 4. 当日复盘

成交清单（回挂信号：战法/题材/情绪周期/角色/连板先验）；**盈亏三列分开**——当日已实现（T+1 当日买入不可卖，当天不可能产生已实现）、当日浮动（收盘盯市）、当日总盈亏（`今total_asset − 昨total_asset − 当日净入金`）；当日收益率（单日 Modified Dietz，**剔出入金**）；下单成功率 / 买不进只数；成交漏斗（计划→挂单→成交）。

### 5. 历史复盘（收益率口径是重点）

数据只认 `qmt_account_daily(CLOSE)` / `qmt_position_snapshot(CLOSE)`。
- **账户级收益率主口径用 TWR**（按出入金切子区间）；工程近似用日频 Modified Dietz，大额出入金当日做真实估值切段。**严禁用 `(今日资产−昨日资产)/昨日资产`**——会被出入金污染。`total_asset` 含浮动盈亏，不可直接当净值。
- 指标：累计/年化、最大回撤（+回撤时长/恢复时间）、夏普/索提诺/卡玛、日胜率、单票胜率盈亏比、持有天数分布、滑点分布。
- **单票已实现盈亏成本法须与券商对账单一致**——A 股对账单/主流软件通常是**移动加权平均（摊薄成本）**，优先用它而非 FIFO；浮动 vs 已实现严格分两列。

### 6. 信号-执行-结果闭环归因（看板最有价值处）

把 `信号(limit_up_selected_stock)` × `实际成交(qmt_trade)` × `次日行情(tencent_unadjusted_daily_quote)` 三方 join，回答"信号执行得怎么样、信号本身准不准"——**这是第三章那个回测 go/no-go 闸门的实盘版延伸**：闸门用回测决定是否上实盘，这里用真钱回看闸门是否成立。
- **计划 vs 实际漏斗**：N 重点候选 → M 下单 → K 成交，买不进按 未下单/一字/秒封/排队未成 细分。
- **买不进逆向选择量化**：买进组 vs 买不进组的龙头强度分位差、错失最强单——把"最强的票没买到"翻译成钱。
- **先验 vs 实际校准**：`continuation_prob`/`next_day_premium_prob` 先验档位 → 实际续板率/溢价为正率（reliability diagram + ECE）；区分"信号准不准（全候选）"与"我们买的票准不准（买进组）"两条曲线。
- **分组对照**：按 龙头分位/角色/战法/情绪周期 对照回测预期算 `实盘 − 回测 gap`。
- **空仓闸门反事实**：空仓日当时买了会怎样，验证闸门累计避损。**反事实/买不进组全程标注"假设未成交"、物理区隔，禁并入真实盈亏汇总。**

### 7. 后端 API / 前端 / 时间口径

后端 `/api/review/*`（daily/trades/orders/positions/history/funnel/attribution，挂 `CurrentUser` 按账户隔离）；只读视图走 `sql_guard_service` 白名单，链式/有状态计算（净值/夏普/撮合/Modified Dietz）放服务层 Python；前端新增 `QmtReviewPage.tsx`（当日复盘 / 历史复盘 / 闭环归因 三屏）。时间口径遵循项目 AGENTS.md：QMT `traded_time` 是 Unix 时间戳，按 `Asia/Shanghai` 解释后转 UTC naive 入库 + 留 `*_east8` 原值对账，前端 `formatEast8DateTime`，**严禁手工 ±8h**。

## 十二、顾问建议与推进次序

### 最可能产生 alpha 的点

1. **结构化 + 分组区分度本身是最扎实的价值，且独立于「打板能不能赚钱」**——让你第一次能用自己的数据回答「LLM 选股到底有没有增量、哪一类更稳」，从「信仰」到「可证伪」。
2. **真正最可能有 alpha 的是「分组条件化」而非「绝对收益」**——龙头带动效应只在 特定层(龙头) × 极短窗口(T+1) × 特定情绪(发酵/高潮) 下显现。把情绪周期、角色、梯队完整度做成条件变量，比追求全天候打分卡更可能找到真东西。
3. **「次日动量 + 数日反转」「涨停延迟价格发现」有学术基础，支持极短持有吃次日高开**——但只在 T+1 这天，持有期拉长就被反转吃掉。回测里 T+1开/T+1收 窗口最值得信，T+5/T+10 大概率证伪。

### 最可能证伪/失效的点

1. **「买不进」单向、系统性高估收益，是头号杀手**——一字板最强也最买不进，封流比对它们天然爆表。这条防线比任何打分卡权重都重要。
2. **龙头战法绝大多数是民间经验 + 幸存者偏差**——情绪周期阈值、龙头判定、角色分层都没有显著性检验，当**待验证特征假设**，不要当已证规则。
3. **因子漂移**——注册制 + 20cm 已压薄打板溢价，旧制度样本拟合新制度会系统性偏差，权重/阈值须滚动重估、按制度分段。
4. **角色标签里中军/卡位/助攻规则判不准**——该保留 LLM 文本辅助或标「待定」，强行用规则判会污染分组。

### 龙头战法量化化：哪些靠谱、哪些是叙事

- **相对靠谱**：龙头三级排序（连板数 > 封板时间 > 封单，仅用于选龙头）；封流比/封单回落作封板强度（必须分形态）；位置筹码作负向风险；龙虎榜净卖出作负向过滤（比净买入干净）；梯队完整度作板块级因子。
- **更偏叙事**：抗分歧性/快速回封/弱转强（需盘中数据，当前全缺，别硬造）；情绪周期精确阈值（经验区间非统计阈值）；净买入跟风游资（预测力短暂且有偏）；「发酵/高潮期最赚钱」（方向可信但要用自己数据验）。

### 推进次序（按依赖，不含工期；含一个 go/no-go 闸门）

1. **落表先行**：把 `selected_*` 展开为 `limit_up_selected_stock` 一股一行，做对 universe 过滤、trade_cal 映射、幂等（整组 delete-then-insert）、命名避让。先用占位/确定性规则填标签，不追求权重精确。
2. **回测引擎 + 可成交性硬过滤**：项目现在没有短线回测引擎，这是真前置缺口。先搭撮合引擎 + 口径版本化（`v_exec`/`v_cost`）+ 空仓日记 0 留痕，产出「乐观上界 vs 可执行口径」对照。
3. **闸门：先验证信号有没有增量**：用最简口径（等权 + 三组分桶）跑 watchlist vs `limit_list_d` 的 IC/RankIC 和分组区分度。**如果证明相对涨停池没有增量，后面所有精细打分卡都不用做了。** 这是 go/no-go。
4. **打分卡校准 + 情绪周期条件化 + FOCUS 先验结构化**：过了闸门再做样本外权重标定、树模型、情绪周期分桶；把 FOCUS 改 JSON-mode 落连板先验字段。
5. **QMT 解耦与竞价实测**（独立于前四步）：QMT 账户权限到位后先把竞价段 `get_full_tick` 能不能用实测清楚——实测前所有竞价择时都是假设；再按第九章状态机接 entry/exit 与卖出决策。

**一句话**：这套方案工程上可行、解耦边界清晰、结构化落表是确定能拿到的价值；但「打板信号有 alpha」是待证伪的假设，**第 3 步是整个项目的成败闸门**——先证明增量存在，再谈精细化。
