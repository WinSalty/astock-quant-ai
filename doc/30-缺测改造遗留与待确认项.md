## 项 1 · 导出坏行被 skip 会绕过「缺测持仓强卖」（待负责人决策口径）

- **现状**：信号侧 watchlist 导出 `export_watchlist`（`backend/app/api/routes_watchlist_export.py`，约 135–160 行）对每行 `LimitUpWatchlistItem.model_validate(row)` 失败时**记日志跳过该行**（`skipped_count += 1`，`skipped_count>0` 升 `error` 日志），不让整请求 500。
- **问题**：若某 `tradable_flag=DATA_MISSING` 的行因 JSON 列（如 `strength_dim_json`/`boost_conditions`）损坏在导出时校验失败被跳过，则该票**从 watchlist 消失**。执行侧 `prior_provider`（`qmt_strategy/app/run.py` `_build_prior_provider`）对「不在今日名单」的票返回 `prior=None` → B3 缺测强卖**仅在 `prior.data_missing=True` 时触发** → 该持仓被当作「隔夜不在名单」按口径③**不强卖**，走普通技术退出。即：**应强卖的缺测持仓因导出坏行被静默漏掉而未强卖**。
- **影响**：窄边界（需该缺测行恰好 JSON 列损坏导致 Pydantic 校验失败），且导出侧已对 `skipped_count>0` 打 `error` 留痕。但方向上是「该清的缺测仓没清」，属真金风控缺口。
- **口径取舍（交负责人决策，本期未擅自改）**：
  - 选项 A（执行侧 fail-closed）：执行侧消费 watchlist 时若响应 `skipped_count>0`，对**已持仓**标的做缺测兜底/不信任「不在名单=安全」——例如对当日不在名单但仍持仓的票按保守口径核查（与隔夜不在名单的「不强卖」口径冲突，需明确边界）。
  - 选项 B（导出契约携身份）：导出响应在 `skipped_count` 之外**带上被跳过行的 `ts_code` 列表**，执行侧据此对「被跳过且在持仓」的票做缺测兜底/告警，避免「身份都丢失」。
  - 选项 C（接受现状）：认定该边界概率极低 + 已有 `error` 留痕 + 运维告警闭环即可，不改码。
- **验收方向**：构造一行 DATA_MISSING + 坏 JSON 列 → 导出 200 且 `skipped_count=1`；按所选选项验证执行侧对该持仓的处置（兜底/告警/不动）符合口径。

## 项 2 · `seal_ratio_pct` 作为缺测触发项的实际缺测率（待生产数据核实）

- **现状**：A7 落表缺测裁决把 §2「约定核心交易指标」任一为 None 即判 `DATA_MISSING`；其中 `seal_ratio_pct`（封流比）与其同分母衍生项 `float_mktcap` 均在核心集（`backend/app/services/limit_up_push_service.py` `_do_persist_selected_stocks` 的 `core_indicators`，约 791/813 行）。
- **问题**：`seal_ratio_pct` 在 `_compact_stock_row`（约 2576–2594 行）有三类置缺成因——①分子缺（无实时 `limit_order` 且 `fd_amount` 回退当前为死代码，见项 3）；②分母缺（无 `free_float`/`float_mv` 且无 `circ_mv`）；③`>100%` 视为单位错配/脏数据置缺并告警。这些成因叠加，可能使「真实弱封/分母偶发缺」的票较常 `seal_ratio_pct=None` → 触发 `DATA_MISSING` → 执行侧对其已过 T+1 持仓 B3 强卖。
- **影响**：若实际缺测率偏高，会**过频触发持仓强卖**（把本可继续持有的弱封票清掉），偏离「宁可错过不买盲」的本意（缺测口径本为保守，但强卖频度需可控）。属取数口径核实项，非代码缺陷——代码按既定保守口径自洽。
- **核实方法（不改码，先量化）**：用生产/回放数据统计入选票 `seal_ratio_pct` 与 `float_mktcap` 的实际缺测率，**区分分子缺/分母缺/`>100%`置缺三类成因占比**，评估其触发 B3 强卖的频度是否可接受。
- **可选处置（核实后再与负责人定）**：若偏高——对「同分母衍生项」`float_mktcap` 与 `seal_ratio_pct` 做**去重/降权**（避免一个分母缺同时点亮两项核心指标）、或对封流比类做**接口级覆盖率护栏**（参照已实现的 `limit_list_d_ok` 接口级护栏模式：某类数据接口整体不可用时不级联全表缺测）。
- **验收方向**：先出缺测率统计报告；若改码，补「分母缺/分子缺/>100%」三类成因对缺测裁决影响的对照单测，并验证强卖频度下降到约定阈值内。

> 已同步用一次性任务（spawn_task）登记；本条为台账记录，便于与项 2 一并评估（二者都关联封流比/封单额取数）。

- **问题**：`fd_amount` 字段只在 `limit_list_d`（`OPTIONAL_APIS`，约 170 行），**不在 `KPL_FIELDS`**；而 `row` 是 kpl 涨停行，顶层从不带 `fd_amount`。缺测改造 A5/A6 把 `limit_list_d` 合并进 kpl 行的**嵌套键** `row["_limit_list_d"]`（非平铺 `row["fd_amount"]`），故 `row.get("fd_amount")` 仍取不到。commit `6dd670a`（打板因子 P1-2「封单额口径统一：limit_order 缺失回退 limit_list_d.fd_amount」）的意图从未在生产生效；注释（约 2614–2616 行）描述的「已统一」状态实际未达成。
- **影响**：有 `fd_amount` 无实时 `limit_order` 的票 → 落表 `limit_order` 列 NULL + 封流比分子 None → `seal_ratio_pct` 缺（与项 2 的缺测率直接相关）。数据完整性缺口，非崩溃。
- **建议方案**：把 2576/2617 的 `row.get("fd_amount")` 改读已合并的嵌套键 `(row.get("_limit_list_d") or {}).get("fd_amount")`（A5/A6 已把 `limit_list_d` 合并到此）。
- **注意（行为变更，须独立评审）**：激活后 fd_amount-only 票的 `seal_ratio_pct` 由 None 变有值 → 影响 `leader_strength_score` → tier/选股 → 执行侧实际买入。须补单测（有 fd_amount 无 limit_order → `limit_order` 落 fd_amount 值、`seal_ratio_pct` 算得出）+ 全量回归（信号侧基线 533 passed/1 skipped）+ 因触及龙头打分→选股，建议走一轮评审再合。本次缺测改造为避免 scope creep 未动此处。
- **依赖**：与项 2 一并评估更优（都关联封流比/封单额取数；项 3 修复会降低项 2 中「分子缺」一类的缺测率）。

### 项 1 处置 · 下线「缺测持仓强卖」(B3)，项 1 漏洞随之消解

- **决策**：不再保留「缺测 → 持仓强制清仓」逻辑；**卖出决策完全交由执行侧 xtdata 实时盘口扳机**（封板续持 / 量价背离 / 弱开 / 破位 / 炸板 / 烂板 / 单票止损）裁决，不再因信号侧缺测标记强清持仓。缺测语义**仅保留买入侧**：信号侧仍产出 `tradable_flag=DATA_MISSING`，执行侧 `buy_prefilter`/`entry_router`/`prior_gate_reason` 仍据此**放弃买入**(B2 不变)。
- **依据**：第一层盘口驱动卖出本不依赖 watchlist、对所有持仓照常生效；B3 是唯一依赖 watchlist 传递缺测标记的卖出路径，下线后「导出坏行被 skip 绕过缺测强卖」(项 1) 不复存在，**无需改导出契约**。
- **执行侧改动（`astock-quant-ai/qmt_strategy`，main）**：
  - `contracts/models.py`：移除 `SignalPrior.data_missing` / `data_missing_reason`（卖出先验不再携带缺测字段；买入侧 `SelectedStockRow`/`PlanRow`/`EntryDecision`/`CandidateView` 的缺测字段**保留**）。
  - `app/run.py`：`prior_provider` 不再把缺测标记搬进 `SignalPrior`。
  - `position/sell_decider.py`：移除 `decide_auction`/`decide_intraday` 两处「缺测 → CLEAR」分支。
  - `app/main.py`：`_evaluate_and_sell_unit` 的 `book is None` 分支简化为「无盘口 → 安全默认不卖」（移除缺测无盘口强清）。
  - `watchlist/remote_watchlist.py`/`storage/mappers.py`/`storage/schema.py`：缺测标记注释口径改为「仅服务买入侧拦截」（落表 round-trip 仍保留，买入侧需要）。
  - 单测：移除 B3 卖出相关用例（`test_sell_decider.py` 三例、`test_app_engine.py` 两例、`test_round7_sell_chain.py` 透传用例），买入拦截用例全部保留。
- **验收结果**：执行侧全量 `pytest` **753 passed**（基线 759 − 移除 6 个 B3 卖出用例 = 753，一致）。

### 项 3 处置 · `fd_amount` 回退死代码修复并激活

### 项 2 处置 · 生产历史数据缺测率核实（只读）

> 数据源：生产 MySQL `stock_ah_ai.limit_up_selected_stock`（只读 SELECT，未做任何写入/变更）。样本：139 行，`trade_date` ∈ [2026-06-04, 2026-06-18]，11 个交易日。

- **关键发现 1 · 数据陈旧，无法直接测 1.3.0 真实缺测率**：全表 `schema_version` 仅 1.0.0（96 行）/1.1.0（43 行），缺测改造契约 1.3.0 **尚未部署**；`tradable_flag` 仅 TRADABLE/BLOCKED/WATCH，无 `DATA_MISSING`；`data_missing_reason` 列**尚未迁移**。1.2.0 起新增指标（`volume_ratio`/`return_5d/10d_pct`/封板时序 `first/last_limit_time`/`open_times`）在本数据 **100% NULL——纯因 ≤1.1.0 代码从不写这些列，非取数失败**。
- **关键发现 2 · 高 `float_mktcap` 缺测是迁移伪缺测（已坐实）**：全表 `float_mktcap` NULL 53.96%，但**41 行有 `seal_ratio_pct` 值却 `float_mktcap` 为 NULL**——现行代码二者同分母不可能出现，证明 `float_mktcap` 系后加列、旧行未回填；且全部 75 个 NULL 行 `created_at` 同属一个 2026-06-13 22:22 批次、非 NULL 行均 2026-06-16 之后创建。**按生成各行的代码口径看，最近数据（2026-06-16~19）`float_mktcap`/`seal_ratio_pct` 缺测率为 0%。**
- **关键发现 3 · `seal_ratio_pct` 缺测从不由分子驱动**：`limit_order`（封单额）**全表 0% NULL**；真·`seal_ratio_pct` 缺测（34 行，2026-06-04~09 最早期数据）100% 为分母侧成因，**0 行分子缺、0 行 >100% 置缺**。
- **对项 3 的回测含义**：项 3（`fd_amount` 分子回退）在本历史数据上**可改善上界 = 0 行**——无任何一行因分子缺导致 `seal_ratio_pct` 缺测。项 3 仍是**正确性修复**（消除真实死代码、堵未来「实时 `limit_order` 缺但 `limit_list_d.fd_amount` 有」的静默缺口），但**不改变任何已观测历史缺测率**。
- **结论与建议**：
  - 因缺测强卖(B3) 已下线，缺测率现**仅影响买入侧拦截**（缺测率高 = 过频「放弃买入」，不再有「过频强卖持仓」风险）。
  - 现有生产数据**无法测 1.3.0 真实缺测率**（版本太旧）。正确量化路径：**部署 1.3.0 → 实跑若干交易日 → 复测**；届时 `data_missing_reason` 列已就位，可直接给出「分子缺/分母缺/>100%」精确三类拆解。
  - 复测前**不对封流比类做去重/降权/接口级护栏**——无证据表明现行代码存在高真实缺测率，避免无依据改码。

> **本文三项至此处理完成**；执行侧 B3 下线、信号侧 fd_amount 修复均带单测、两侧全量回归绿。提交见各侧分支。
