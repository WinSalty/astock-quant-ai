# 执行侧本地化改造 · 方案与改动说明（含信号侧配合项）

> 文档日期：2026-06-13。本文是 `03-QMT侧执行引擎-开发设计.md` / `04-…开发计划.md` 之后的一次**架构变更**说明，
> 因涉及**信号侧需配合改造**，单独成文。变更范围：执行侧由「盘中直连远端 MySQL 读写」改为
> 「**单进程 + 本机 SQLite，异步持久化，盘前同步名单入库、盘后同步回流回远端**」。
> 按项目约束：不含工期 / 工时预估，只按阶段与任务划分，给出验收标准与依赖顺序。

## 一、背景与目标

- 信号侧（Linux）与执行侧（Windows VPS）在**两台不同服务器**上。原设计中执行侧盘中直连远端 MySQL（读 watchlist、写 `qmt_*` 回流），一旦两机之间网络抖动 / 中断，会直接波及**盘中下单与回报落库**。
- **目标**：盘中交易**完全不依赖跨服务器网络**——下单只靠本机 QMT + 本机库；网络问题只影响「事后同步」，而同步是可重试、幂等的。

## 二、方案总览

执行侧采用 **单进程 + 本机 SQLite** 的本地化数据栈，数据流分三段：

1. **盘前（同步入）**：信号侧产出当日 watchlist 后，执行侧盘前一次性把 watchlist 拉取并写入**本机 SQLite 的 `watchlist` 表**；`watchlist_loader` 改为从本机 SQLite 读（不再盘中跨网络读远端）。同步失败 → 执行侧无当日名单 → 触发既有降级「只守仓、不开新仓」+ 告警。
2. **盘中（异步落盘）**：交易热路径（下单决策、`xttrader` 回报回调）**只读写内存结构**；持久化（`qmt_*` 回流明细 / 快照、下单台账）经**单一后台写线程 + 队列**异步写入本机 SQLite。**生产者只入队即返回，绝不在热路径做磁盘 I/O**。
3. **盘后（同步回）**：独立的盘后任务把本机 SQLite 的 `qmt_*` 当日数据**幂等同步回远端 MySQL**，供信号侧复盘 / 归因。须当日完成、失败重试、完成对账（本地条数 vs 已同步条数）。

### 关键不变量（本次变更的验收灵魂）

| 不变量 | 落地手段 |
| --- | --- |
| **不阻塞交易** | 持久化一律 write-behind：热路径 `put_nowait` 入队即返回；写线程异常**只吞不传播**到交易线程 |
| **线程安全** | 单进程多线程（QMT 回调线程 + 决策线程 + 调度线程 + 写线程）；内存台账 / 共享状态加锁；SQLite 仅由**单一写线程**写（WAL，1 写 N 读） |
| **重启幂等** | 进程启动时从 SQLite 重建内存下单台账 → 重启后 `has_active` 仍有效，不重复下单 |
| **同步幂等** | 远端 `qmt_*` 唯一键 `upsert`，同步可重跑、重传不产生重复行；本地行带 `synced` 标志，已同步不重推 |
| **当日完成** | QMT `query_*` 隔日清空、远端复盘隔日读；同步必须当日完成，失败回调（废单/撤单失败）在同步前不得丢（SQLite 持久化保证） |

## 三、执行侧改动点

新增本地存储子包 `qmt_strategy/storage/`，对**业务模块零侵入**（仍只依赖既有 `QmtRepository` / `LocalLedger` / `SelectedStockSource` 协议，本次只是**换实现**）：

| 模块 | 职责 |
| --- | --- |
| `storage/schema.py` | 本机 SQLite 建表（`qmt_*` 四表 + `local_order_ledger` + `watchlist` + 同步标志）、WAL 等 pragma |
| `storage/mappers.py` | 记录 ↔ SQLite 行的序列化（Decimal 存 TEXT 保精度、date/datetime 存 ISO、枚举存值），单一来源 |
| `storage/write_queue.py` | `AsyncWriteQueue`：单后台写线程 + 队列；`submit` 非阻塞入队；写线程异常只吞不传播；`flush`（盘后用，阻塞至清空）/`stop` |
| `storage/sqlite_repository.py` | `SqliteQmtRepository`（实现 `QmtRepository`）：写经队列异步落盘；读（对账盘后用）直连本地 SQLite |
| `storage/sqlite_ledger.py` | `PersistentLocalLedger`（实现 `LocalLedger`）：内存为权威（加锁，热路径只读内存）+ 异步镜像落 SQLite + 启动重建 |
| `storage/watchlist_source.py` | `SqliteSelectedStockSource`（实现 `SelectedStockSource`）：从本机 `watchlist` 表读当日名单；`save_watchlist` 供盘前同步入写库 |
| `storage/sync_job.py` | `RemoteSyncJob`：本地 SQLite `qmt_*` → 远端 MySQL 幂等同步 + 标记 `synced` + 条数对账 + 重试 + 告警 |
| `storage/local_stack.py` | 装配器：建表 + 起写队列 + 组装上述实现，暴露 `start/flush/stop`；`Engine` 注入其产出的 repository/ledger/source |

`Engine`（编排层）改动：台账改为**可注入**（默认仍 `InMemoryLocalLedger`，向后兼容）；`close_batch` 在对账前先 `flush` 持久化队列（保证对账读到一致数据）；新增启动重建与停机 drain 钩子。**各业务模块（watchlist/auction/entry/order/position/risk/data_writer/callbacks/reconcile）代码不变。**

## 四、信号侧需要配合的改造（重点）

本次变更的跨侧协调项，信号侧须配合以下两点：

### 4.1 盘前向执行侧交付当日 watchlist（核心）

执行侧盘前不再直连信号侧 MySQL 读 `limit_up_selected_stock`，改为**拉取当日 watchlist 并落本机 SQLite**。需要信号侧提供一个**稳定的当日 watchlist 交付方式**，二选一（落地择定）：

- **方式 A（推荐）：内网只读接口** `GET /api/internal/watchlist?trade_date=YYYY-MM-DD`，返回当日 watchlist 全集（JSON，字段见下）。执行侧盘前（约 9:00 前）pull 一次写入本机 SQLite。鉴权用固定 token、来源 IP 白名单。
- **方式 B：导出文件**：信号侧把当日 watchlist 导出为 JSON / SQL 放到执行侧可取位置（共享目录 / 对象存储），执行侧盘前导入。

**字段契约（与既有 `limit_up_selected_stock` 一致，不新增语义）**：`ts_code`、`trade_date(T)`、`target_trade_date(T+1)`、`leader_strength_score`、`role`、`strategy`、`market_state`、`tradable_flag`、`continuation_prob`、`next_day_premium_prob`、`boost`、`fail_conditions`、`signal_close`、`limit_up_price`、`reasonable_open_high_low/high`、（可选 `first_board_vol`、`float_mktcap`、`strategy_family`、`setup`）。

> 注：`limit_up_selected_stock` 表当前**尚无 DDL/迁移/ORM**（设计文档自注 grep 零命中），其建表仍由信号侧 watchlist 契约产出设计负责。本变更**不改字段语义**，只新增「盘前把它交付给执行侧」这一动作。

**失败口径**：盘前同步失败（接口超时 / 文件缺失 / 解析失败）→ 执行侧当日无名单 → `WatchlistContext.degraded=True`、`open_new_position_allowed=False`（只守仓不开新仓）+ 告警。**不阻塞、不崩溃**。

### 4.2 回流读取时点后移

执行侧回流由「盘中实时直写远端」改为「**盘后批量同步**」。因此：

- 远端 `qmt_*` 四表的**当日数据，在执行侧盘后同步完成后才齐**。信号侧复盘看板 / 闭环归因若按「当日」消费，触发时点须**后移到同步完成之后**（建议执行侧同步完成后写一个「当日同步完成」标志 / 通知，信号侧据此再跑当日复盘）。
- 远端 `qmt_*` 四表 **DDL 不变**（仍由信号侧 Alembic 落地，唯一键 / 时间口径 / `signal_trade_date` 回填口径均不变）；变的只是**写入来源与时点**：从「执行侧盘中直连写」变为「执行侧盘后批量幂等同步写」。同步幂等保证不产生重复行。

### 4.3 安全面（顺带收益）

盘中执行侧**不再需要对 Windows 暴露远端 MySQL 写端口**（只在盘后同步任务里用 MySQL 写账号）。MySQL 端口暴露面从「盘中常驻」收敛为「盘后短时」，更安全。原设计「待确认项 1（A 直连 / B 接口）」在本方案下统一为：**watchlist 走只读接口/导出（盘前）+ 回流走盘后同步（写）**。

## 五、开发计划（阶段 · 任务 · 验收 · 依赖）

### 阶段 0 · 规划与改动说明（本文档）
- 产出本方案 + 信号侧配合项 + 关键不变量。**验收**：本文 + 信号侧据此可启动其改造。

### 阶段 1 · 本地存储核心（锁定）
- T1.1 `storage/schema.py`：本机 SQLite 七表建表 + WAL/`synchronous=NORMAL` 等 pragma。
- T1.2 `storage/mappers.py`：记录 ↔ 行序列化（Decimal→TEXT、date/datetime→ISO、枚举→值），round-trip 无损。
- T1.3 `storage/write_queue.py`：`AsyncWriteQueue`（非阻塞入队 + 单写线程 + 异常只吞不传播 + flush/stop）。
- **验收**：建表幂等；mappers round-trip 单测通过；**`submit` 在写线程被人为阻塞时仍立即返回**（证明不阻塞）；写线程抛错不影响生产者。

### 阶段 2 · 协议实现（依赖阶段 1，可并行）
- T2.1 `storage/sqlite_repository.py`（`QmtRepository`）：异步写 + 同步读 + `ON CONFLICT DO UPDATE`+COALESCE 幂等。
- T2.2 `storage/sqlite_ledger.py`（`LocalLedger`）：内存权威 + 加锁 + 异步镜像 + 启动重建。
- T2.3 `storage/watchlist_source.py`（`SelectedStockSource`）：读本机 watchlist + `save_watchlist` 写入。
- T2.4 `storage/sync_job.py`：本地 → 远端幂等同步 + `synced` 标志 + 条数对账 + 重试。
- **验收**：各自单测覆盖——幂等 upsert/COALESCE；台账重启重建后 `has_active` 生效；watchlist 读回一致；同步重跑不产生重复行、漏同步可补。

### 阶段 3 · 编排接线（依赖阶段 2）
- T3.1 `storage/local_stack.py` 装配器 + `Engine` 台账可注入 + `close_batch` 对账前 flush + 启动重建/停机 drain。
- **验收**：集成测试——引擎在 SQLite 上跑通；回调入队非阻塞；`close_batch` 先 flush 再对账读到一致数据；重启后台账重建、不重复下单。

### 阶段 4 · 测试 / review / 修改 / commit
- 全量 `pytest` 绿；对**关键不变量**（不阻塞 / 线程安全 / 重启幂等 / 同步幂等）做对抗式复核并修复；提交并推送。

### 依赖顺序
```
阶段0 文档 → 阶段1 存储核心(schema/mappers/write_queue) → 阶段2 协议实现(repo∥ledger∥watchlist∥sync)
  → 阶段3 编排接线 → 阶段4 测试/review/修改/commit
信号侧并行：按 §4.1 提供盘前 watchlist 交付（接口/导出）+ 按 §4.2 后移复盘读取时点
```

## 六、待确认项
1. **watchlist 交付方式**（§4.1 A 只读接口 / B 导出文件）二选一固化；token / IP 白名单口径。
2. **同步触发与时点**：盘后同步由 Windows 任务计划触发的具体时刻；「当日同步完成」如何通知信号侧（标志表 / 回调）。
3. **SQLite 落盘耐久档位**：`synchronous=NORMAL`(WAL) 与 `FULL` 的取舍（NORMAL 性能好、极端断电可能丢最后事务；本场景有 QMT `query_*` 当日兜底，倾向 NORMAL）。
4. **本地库保留与清理**：本机 SQLite 历史数据保留天数 / 归档策略（同步成功后是否定期清理本地旧数据）。
