## 2. 修复口径与原则

1. **采纳对抗核验员结论**：以核验后严重度与修复方向为准，不重新争论；多条 `partly` 的真实残余问题比初审更窄，按核验口径落地（见各条「要点」）。
2. **门槛依赖顺序**：A（资金安全 P0+P1）→ B（watchlist/报告正确）→ C（go/no-go）→ D（复盘数据与鉴权）→ E（韧性观测）→ F（其余 P2 健壮性/口径）。同根因合并修复，避免反复触碰同一文件。
3. **同文件合并**：按文件簇（设计阶段 14 簇）批量改，中枢文件（`app/main.py`、`position_manager.py`、`limit_up_push_service.py`）单簇内一次性顺序改。
4. **TDD/测试守护**：每条给测试落点；先补/改测试再改实现（能 TDD 则 TDD），每批跑该簇测试 + 收尾跑两侧全量。
5. **中文注释**：所有触碰处补/改中文注释（业务意图 / 边界 / 重跑与异常口径），遵循工作区 `AGENTS.md`。
6. **不含工时**：仅按阶段 + 验收 + 依赖顺序。
