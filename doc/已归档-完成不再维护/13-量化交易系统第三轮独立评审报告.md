## 0. 评审方法与口径

- **方法**：按 19 个子系统区块**并行独立审计**（执行侧 9 块 + 信号侧 9 块 + 跨侧契约 1 块），每条初审发现再交由**独立对抗式核验员重新读码复核**（confirmed 成立 / partial 成立但描述或严重度有偏 / refuted 不成立 / uncertain 静态不可判），**只保留未被驳回项**，并以**核验员复核后的严重度为准**（多处 P0/P1 经核验下调，正文标注）。报告作者另对全部 P0 与关键 P1 亲自读码二次确认。
- **范围**：执行侧下单/台账/幂等、建仓路由/资金分配、集合竞价、持仓/卖出、风控、连接/调度/时间/日历、本机 SQLite 持久化、回流/对账、watchlist 拉取/契约/板规则；信号侧打板报告多阶段 pipeline/收口落表、报告上下文与因子装配、提示词与建议、报告投递/分享、龙头打分、watchlist 产出、回测 go/no-go、qmt 回流与复盘归因、供数 feeds；以及两侧 HTTP 契约一致性。
- **严重度口径**：`P0` 致命（直接资金损失 / 重复下单 / 漏卖 / 数据损坏 / 产出错误名单）；`P1` 严重（特定条件下出错或风控失效）；`P2` 中等（边界 / 健壮性 / 口径瑕疵 / 局部风控或数据缺口）；`P3` 轻微或优化。
- **统计**：原始 153 条 → 对抗核验后保留 141 条、驳回 12 条。**本报告聚焦其中需处理的 P0/P1/P2 共 72 条（P0 × 3、P1 × 14、P2 × 55）；69 条 P3 轻微/优化项按要求已略去**（全量明细同步精简至 [`13-评审发现明细.json`](13-评审发现明细.json)）。
- **重要前提**：本系统仅缺 miniQMT 客户端/账号，尚未实盘。**纯粹因“缺 miniQMT/账号”才产生的问题不在本报告内**；但模板/契约里“一旦接上就会出错”的逻辑缺陷、错误默认值、被吞异常仍如实报告（多为静态可证的潜伏缺陷，需在实盘前修复）。下方“待目标机实测”的标注，仅指其**触发的运行时表现**需在目标机确认，**不代表缺陷本身存疑**。

## 2. P0 致命问题详述

### P0-1 · EXEC-sched-01 — 断线重连未接线，真实掉线后永不重连
- **位置**：`astock-quant-ai/qmt_strategy/qmt_strategy/app/main.py:159-167`、`app/run.py:226,268`、`adapters/xt_real.py:209-210`、`data_writer/callbacks.py:287-296`
- **问题**：`ConnectionGuard` 设计了完整的断线兜底链 `on_disconnected() → reconnect()`（换新 `session_id` 重走 connect/subscribe、置 `ready=False`、成功后触发补采）。但装配时把 `ExecCallback` 的断线钩子接成 `on_disconnected_hook=self.on_reconnect_backfill`（Engine 补采），券商断线回调链路为 `broker.on_disconnected → xt_real._RealCallback.on_disconnected → ExecCallback.on_disconnected → engine.on_reconnect_backfill`，**全程只补采、从不调用 `guard.on_disconnected/reconnect`**。后果：(a) 掉线后 `guard.reconnect()` 永不触发，不换新 session、`holder` 仍指向已断的旧 trader；(b) `guard._ready` 只在 `connect_and_subscribe` 失败和 `guard.on_disconnected` 里置 False，后者永不触发 → 掉线后 `guard.ready` 恒为 True。`run.py:268` 注释甚至写着“on_disconnected→guard.reconnect 换 trader”，但实际接线并未如此。
- **影响**：实盘盘中一旦 xttrader 掉线，连接守护永不重连、session 永不更新，进程停在断开状态直到人工重启；主循环（`run.py:270-276`）因 `guard.ready` 恒 True 对一个已死 trader 反复 `run_forever()` 空转。期间隔夜 T+1 持仓无法卖出（漏卖、被迫扛单），新成交回报全部丢失。叠加 P0-2，末态是**整日全冻结**。每日仅一次的 `PREWARM`（`scheduler.py:136-138` 调 `connect_and_subscribe`）只能次日恢复，对盘中掉线无效。
- **修复**：断线回调必须先经 guard 重连：把 `on_disconnected_hook` 改为触发 `guard.on_disconnected()`（→ `reconnect` 换新 session → 成功后再调 `engine.on_reconnect_backfill`）。Engine 在 guard 之前装配，需延迟绑定（guard 构造后回填 hook，或 `lambda: guard.on_disconnected()`）。主循环对 `guard.reconnect` 失败做退避重试 + 强告警，绝不对已断 trader 反复 `run_forever`。
- **说明**：缺陷为**静态可证**的接线错误；`run_forever` 的阻塞/返回语义需目标机实测，但不影响“断线后不重连”这一结论。

### P0-2 · EXEC-sched-02 — 用已断 trader 补采，`_trade_conn_ok` 永久 False 致全天 FREEZE
- **位置**：`app/main.py:923-941`、`app/main.py:329-354`
- **问题**：`on_reconnect_backfill`（被当作断线钩子）先置 `_trade_conn_ok=False`，再用 `deps.trader`（`TraderHolder`，因 P0-1 仍指向已断 trader）跑 `run_backfill()` 与 `_rebuild_positions_from_broker()` 的 `query_*`。这些查询必然抛错 → except 分支记日志后 `return`，**保持 `_trade_conn_ok=False`**。设计本意“补采失败即保持冻结”是对的，但因 P0-1 没有任何重连真正发生，该标志将**永久停在 False**。`risk.gate` 据此恒 FREEZE，开新仓与卖出（`run_sell_pass` 同样喂 `_trade_conn_ok`）**全部被冻结**。
- **影响**：真实掉线后即使物理链路恢复，进程也永远 FREEZE：不仅不开新仓，连必要的止损/破位卖出都被冻结（`_evaluate_and_sell_unit` 在 FREEZE 时 `return False`），等于整日裸奔扛单到收盘，直接资金损失。把“连接健康权威源”错误地绑定在一个永不更新的标志上。
- **修复**：与 P0-1 一并修：断线必须先经 guard 重连换新 trader，补采用重连后的新句柄；`_trade_conn_ok` 解冻应由“重连就绪 + 新句柄补采成功”事件驱动，建议直接把权威源绑定 `guard.ready`。

### P0-3 · EXEC-storage-01 — 发单前落盘失败只告警不拦单，崩溃后重复下单
- **位置**：`order/order_executor.py:97-102`、`order/order_executor.py:376-391`（卖出同口径 `512-533`）
- **问题**：`_sync_persist_before_order` 调 `ledger.flush_pending()`（内部 `write_queue.flush_confirm`）。`flush_confirm` 的契约是“返回 False（PLANNED 行 commit 失败 / 写线程已死 / 超时）调用方据此 fail-closed 拒发单”，但代码在返回 False 时**仅 `logger.warn` 然后继续往下**，在 `382` 行照常 `trader.order_stock` 发出真实委托。于是在“磁盘满 / 写线程异常死亡 / commit 失败”时，券商收到委托而本机台账没有该 PLANNED 行；一旦进程在成交回报落盘前崩溃重启，`load_from_db` 读不到此行 → `find_active` 返回 None → 同一计划被**再次下单**。这正是 P0-C3 机制声称要堵的窗口本身。
- **影响**：真实资金重复下单 —— 存储故障叠加崩溃重启时，同一标的被买入两次，超额占用资金、放大敞口。
- **修复**：`flush_pending` 返回 False 时必须 fail-closed —— `place()` 在 `order_stock` 之前 `return None` 放弃本次下单，并触发 `on_storage_failure`/停开新仓告警。可区分两类：写线程死亡/commit 失败（确定性故障）必须拒单；纯超时（队列拥堵但写线程存活）可保留“继续但强告警”，同时触发健康检查与人工介入。当前对所有 False 一律放行，等于在故障时关闭了唯一的重复下单防线。

## 3. P1 严重问题详述

### 执行侧

**P1 · EXEC-position-01 / EXEC-DW-04 — 持仓单元字段跨线程无锁读改（丢更新/脏读）**
- **位置**：`position/position_manager.py:332-371`（`apply_sell_fill`）、`155-181`（加仓合并）、`384-398`、`376-379`；`data_writer/callbacks.py:98-114,142-152`；`app/main.py:594-605`
- **问题**：`RLock` 只保护 `_units` 字典的结构性增删，**单元字段级**读-改-写未取锁（代码注释 60-64 自承认“细粒度数据竞争尚未收敛”）。QMT 成交回报在回调线程跑（`_apply_trade_to_position`），`DailyScheduler` 在独立 daemon 线程跑卖出巡检/快照/`apply_position_snapshot`。`applied = min(sell_vol, unit.volume); unit.volume -= applied; unit.can_use_volume = max(0, ...)` 与 `counted_trade_ids.add(traded_id)`、`unit.state` 复位均在锁外，两线程并发触达同一 unit 会发生丢失更新 / 脏读。
- **影响**：`volume/can_use_volume` 与券商真相偏离 → 高估可卖量重复挂卖（超卖/废单/对账阻断），或漏减误判仍持仓继续挂卖；加仓侧 `avg_cost` 被错算会污染单票止损线判定（误触/漏触强平）；`counted_trade_ids.add` 非原子会使同一 `traded_id` 双计或漏计。
- **修复**：把 `apply_sell_fill / mark_position_on_fill / revert_selling / apply_position_snapshot / counted_trade_ids 维护` 的字段级读改全部纳入 `self._lock`（RLock 可重入）；`get_unit` 对外只返回只读快照，写入改走 PositionManager 的原子方法（内部一体完成“去重+扣减+状态推进”）；把 `main.py` 的 SELL 去重判断与 `apply_sell_fill` 放进同一临界区。
- **核验**：初评 P0，核验下调 P1 —— “同票两笔 SELL 回调并发”这条具体路径依赖 QMT 回调线程模型（可能单派发线程串行化，需目标机实测）；但**调度线程 ↔ 回调线程**的跨线程字段竞态与运行时无关、确定成立，作为定级主依据。

**P1 · EXEC-position-03 — REDUCE 部成后在途量被二次下单（超卖/废单）**
- **位置**：`position/position_manager.py:289-306,332-371`；`app/main.py:783,862,868,909-910`
- **问题**：`mark_selling` 进入 SELLING 时**不扣减 `can_use_volume`**（注释 319 自述）。正常靠 `run_sell_pass` 跳过 SELLING 防重复。但 REDUCE 是部分卖出：委托部成后 `apply_sell_fill` 把单元推进为 PART_SOLD（只扣已成量），PART_SOLD **不被 `main.py:783` 的 SELLING 跳过**，而 `can_use_volume` 仍含“在途未成量”。下一轮巡检把这部分在途量当可用量再次下卖单 → 叠加卖单累计在途量 > 真实可卖量。
- **影响**：减仓委托部分成交后剩余在途量被二次下单，产生超过可用量的废单、触发对账偏差阻断次日开仓（是否真超卖底仓取决于券商在途冻结口径）。
- **修复**：`mark_selling` 时冻结在途委托量（或引入 `frozen/on_road` 量字段，可卖上限 = `can_use_volume - 在途未成`）；或对 PART_SOLD 单元在巡检中先校验台账是否仍有 active 卖单再决定是否再次下单。
- **核验**：确认为“相邻 tick 重复”（非同 tick），机制成立，P1。

**P1 · EXEC-storage-03 — 磁盘满静默丢数据 + `is_healthy` 误报健康**
- **位置**：`storage/write_queue.py:43,80-104,118-150`、`106-108`
- **问题**：磁盘满时写线程 `commit()` 持续抛 `OSError`，被 except 捕获 → rollback → `_batch_ok=False` → 记日志 → `task_done()`，**写线程不死、`_failed` 不 set、`is_alive` 仍 True**。于是 `submit` 的存活性兜底（91 行）永不触发、`on_failure` 不被调用，后续写继续 `put_nowait` 进队列被取出→commit 失败→丢弃，每条只剩一行 `async_write_failed` 日志；而 `is_healthy` 只看 `is_alive`（106-108）恒返 True。即磁盘满下系统既不 fail-closed 也不告警，持续静默丢数据（盘中成交回报、台账状态推进大量丢盘）。
- **影响**：重启后台账/回流四表大面积缺行，对账失真、可能重复下单或误判持仓。
- **修复**：(1) 写线程对“连续 commit 失败”计数超阈值即 `_failed.set()` + `on_failure`，区分瞬时锁等待与持续 I/O 故障；(2) `is_healthy` 纳入“最近写是否成功”而非只看线程存活；(3) 给队列设上限/积压字节阈值，超限触发 `on_failure`（避免写线程 fsync 永久阻塞时内存无限堆积）。
- **核验**：核验修正 —— “无界队列内存无限增长”仅属“写线程阻塞挂起”一类（与磁盘满抛错互斥），其余核心论断（静默丢数据 + `is_healthy` 误报 + 不 fail-closed）成立，维持 P1。

## 4. 专项：信号侧「打板报告产出 + watchlist 产出」链路（重点审阅）

按要求对该链路逐环梳理，下列问题**集中威胁“执行侧拿到的当日买入名单是否完整、口径是否正确”**，建议作为信号侧第一优先修复簇：

1. **整日空名单风险（P1 SIG-RP-01）**：定时主入口 READY 早退不补落选股 → 选股缺失日执行侧整日空 watchlist。这是该链路最高优先级缺陷。
2. **核心因子结构性缺失/失真**：
   - **P1 SIG-RC-01** `winner_rate` 落表恒 NULL（键名不匹配）；
   - **P1 SIG-LS-01** 位置筹码安全分恒中性 0.5（cyq 从不在被打分行上）→ 高位风险不扣分、直接污染资金权重；
   - **P1 SIG-LS-02** 题材按多题材串整串分桶 → 板块龙头滥发、题材梯队失真。
   - 三者叠加 → `leader_strength_score`（执行侧资金加权与开仓 top-N 名额的唯一依据）系统性偏差。
3. **LLM 失效时的“伪精选”落表（P2 SIG-RC-03 / SIG-RC-04）**：LLM 全失效/全幻觉代码/漏 `selected_stocks` 键时，`_select_stage_stocks` **静默回退确定性兜底排序**并把候选落入可交易 watchlist（带龙头打分的 `tradable_flag`），而非进入“当日无候选”安全分支；`selection_reason` 标“兜底”但不阻断下单，stage_quality 仍显示 OK 掩盖问题。建议对三种兜底来源在落表层显式降级（`action='放弃观察'`/`tradable_flag='BLOCKED'`）并点亮降级标识。
4. **先验概率错挂标的（P2 SIG-RP-04 / SIG-PA-05）**：FOCUS `stock_priors` 的 `ts_code` 不与入选集校验对齐（不同于 SELECTION 阶段的严格映射）。LLM 串号/幻觉代码会使 `continuation_prob/next_day_premium_prob` 落到错误标的或静默丢失，全程无日志。建议落表前用入选集 `ts_code` 过滤校验。

- **SIG-leader-scoring**：`score_stocks` 的 `context_rows` 默认回退 `rows`“重新引入自参照分位偏差” —— 生产路径显式传 `full_pool`（全涨停池），回退仅为防御性提示。
- **SIG-watchlist**：跨层重复 ts_code 唯一键冲突“savepoint 回滚致整日 watchlist 全空” —— 实际收口路径不会因单票冲突清空整组。
- **SIG-feeds**：`total_mv` 单位口径“万元 vs 元错配” —— 阈值/缩放与单位一致，不成立。
- **XCUT-contract**：执行侧落库无条件用 today 覆盖 `target_trade_date`“两侧日历不同步会错配买入日” —— `prefetch` 用 `prev_open(today)` 反推信号 T 拉取、落库强制 `target_trade_date=today` 是设计口径，不成立。
- **XCUT-contract**：ingest 信封 `account_id/trade_date` 可选为 None“静默串到错误账户” —— 与 SIG-QMT-03 重叠，但所述具体串栈路径在现有取数链上不成立（SIG-QMT-03 已以 P2 留存真正的“缺省即放行”脆弱性）。
- **报告上下文（rerun）**：1 条因与已留项重复/描述不成立被驳回。

## 7. 上线硬门槛建议（按依赖顺序，不含工时）

> 按“先堵资金安全洞 → 再补数据正确性 → 最后补观测与韧性”的依赖顺序组织；每项给出验收口径。

**门槛 A — 实盘前必须清零（P0 + 资金安全 P1）**
- A1 修复断线重连接线（P0-1/P0-2 + EXEC-risk-05/sched-09/10）：断线回调经 `guard.on_disconnected→reconnect` 换新 session，`_trade_conn_ok` 绑定 `guard.ready`，主循环退避重连 + 告警。**验收**：fake 注入“盘中断线”后，进程自动换 session 重连、补采用新句柄成功、`_trade_conn_ok` 恢复 True、卖出不被永久冻结（单测覆盖）。
- A2 发单前落盘失败 fail-closed（P0-3）：`flush_pending` 返 False 时拒发单 + 触发存储 fail-closed。**验收**：注入写线程死亡/commit 失败后，`place`/`place_sell` 不再调 `order_stock` 且停开新仓告警（单测）。
- A3 持仓字段并发安全（P1 EXEC-position-01/03、EXEC-DW-04）：单元字段读改纳入锁、`get_unit` 返只读快照、在途量冻结。**验收**：并发回调 + 调度巡检压测下 `volume/can_use_volume/counted_trade_ids` 无丢更新；REDUCE 部成后不重复挂在途量（单测）。
- A4 成交/委托唯一键与对账健壮（P1 EXEC-DW-01/02/05）：缺 `traded_id`/`order_id` 不落 `"None"`/不撞 NOT NULL；`reconcile` `int(... or 0)` 兜底 + 子项 try/except。**验收**：注入缺字段/NULL 量的脏成交后，无成交折叠、对账不崩、失败留痕不丢（单测）。
- A5 标的归一锚点化（P1 F1）：后缀只认末尾/开头、矛盾即判脏返 None。**验收**：`SH000001` 等矛盾脏串返 None 或正确交易所（单测）。

**门槛 B — watchlist/报告产出正确（实盘前必须清零，信号侧第一优先簇）**
- B1 选股不丢（P1 SIG-RP-01）：READY 早退分支补落选股。**验收**：构造“READY 但无选股行”后，定时主入口自动补落、`/internal/watchlist` 非空（单测）。
- B2 核心因子正确（P1 SIG-RC-01/SIG-LS-01/SIG-LS-02）：`winner_rate` 取嵌套键、cyq 注入打分行、题材按单题材归并。**验收**：落表行 `winner_rate` 非空、`chip_imputed` 在有 cyq 时为 False、单题材龙头 `theme_top` 可大于自身板高（单测）。
- B3 LLM 失效不伪装成可交易名单（P2 SIG-RC-03/04）：三种兜底来源落表层显式降级 + stage_quality 点亮。**验收**：注入 LLM 全失效/漏键后，落表行 `tradable_flag=BLOCKED` 或不落 selected（单测）。
- B4 先验不错挂、降级标记正确（P2 SIG-RP-04/SIG-PA-05/SIG-RP-02/SIG-PA-01）：FOCUS `ts_code` 与入选集校验、`advice_degraded` 回填或改视图字段。**验收**：越界 prior 被丢弃并记日志；建议成功后行级 `advice_degraded=False`（单测）。

**门槛 C — go/no-go 闸门可信（实盘放量前）**
- C1 回测统计口径（P2 SIG-backtest-01/02）：t 检验改用独立交易日数 + 样本标准差；`v_exec.0` 不得产出 GO。**验收**：相关样本下 t 不再按 √全样本计；`v_exec.0` summary 进 `evaluate_go_no_go` 被拒（单测）。

**门槛 D — 复盘数据与鉴权（上线即生效，可与放量并行）**
- D1 回流/归因正确（P1 SIG-QMT-02 + P2 SIG-QMT-01/03/04/05）：闭环 join 带 `(account_id,trade_date,order_id)`；信封必填强一致；净值剔出入金；COALESCE 收窄。
- D2 内网鉴权对称（P1 XCUT-01 + P2 SIG-WL-02/SIG-QMT-06）：两接口 token 策略两侧对齐；watchlist 导出补 IP 白名单；回流写 token 独立。

**门槛 E — 韧性与观测（放量前补齐）**
- E1 存储 fail-closed 可观测（P1 EXEC-storage-03）：磁盘满/commit 失败计数触发 `on_failure`、`is_healthy` 纳入最近写成功、队列设上限。
- E2 调度/卖出窗口完整（P2 EXEC-sched-05、EXEC-risk-04）：竞价段卖出可触发、盘中行情健康有更新来源、`KILL_SWITCH` 能撤在途。

> 其余 P2 建议在对应模块迭代中分批消化；同一根因（如持仓字段并发、内网鉴权不对称、`advice_degraded` 口径、题材/筹码因子缺数）应**合并修复**避免反复触碰同一文件。
