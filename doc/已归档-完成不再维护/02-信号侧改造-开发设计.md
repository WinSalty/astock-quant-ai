## 一、概述、范围与实施次序

### 1.1 文档定位

本文档（文档 A）只覆盖**信号侧**改造，即在 Linux 现有项目 `stock-ah-premium-ai`（FastAPI 后端 + React 前端 + MySQL `stock_ah_ai`）内部**新增/改造**的逻辑，最终产出每个信号日 T 的 **watchlist 信号**（落地为新表 `limit_up_selected_stock`，一股一行：信号日 `trade_date=T`、买入日 `target_trade_date=T+1`，附龙头强度分 / 角色 / 战法 / 情绪周期 `market_state` / 可成交性 `tradable_flag` / 连板与隔日溢价先验等）。

信号侧产出后，由执行侧 **QMT（Windows VPS）独立消费**：QMT 读 `limit_up_selected_stock` 决策下单，并把实盘成交/委托/持仓/资产回流到 `qmt_*` 表。两侧解耦、只通过 MySQL / 只读接口通信——**信号侧只写信号表 + 只读 QMT 表，QMT 侧只写 `qmt_*` + 只读信号表**。本文档**不涉及** QMT 下单逻辑、不涉及 `qmt_*` 表的采集与落库实现（那是执行侧职责，见下方引用），只负责"把信号算准、落表、可被回测验证、可被 QMT 消费"。

文档 B（QMT 侧改造）单独成文，与本文档的衔接点：本文档第二节定义的 `universe_filter` 纯函数，QMT 侧在下单前需复用同一份口径（详见文档 B 相关章节）。

边界约束（沿用既定口径，本文档不推翻）：

- 只做**主板 + 创业板**，排除 ST / 科创（688/689）/ 北交（8xx/4xx/.BJ）。
- A 股 **T+1**：买入当日不可卖，最早 T+2 卖；主板 ±10%、创业板 ±20%；不复权。
- 回测口径：`trade_date=T` 买入须在 T+1，经 `a_trade_calendar` 映射；一字 / 秒封买不进不计收益。
- 时间口径遵循项目 `AGENTS.md`：业务时间 `UTC naive` 入库（`datetime.now(UTC).replace(tzinfo=None)`），`created_at`/`updated_at` 走 `server_default=func.now()`（东八区本地理解）；前端统一 `formatEast8DateTime` 展示，禁止手工 ±8h；`DATE` 交易日字段本身即东八区交易日，不做时区换算。
- 所有新增代码 / SQL / 迁移必须补中文注释，写清业务意图、边界条件、幂等/重跑口径。

### 1.4 模块依赖顺序与 go/no-go 闸门

总原则：**先把信号落表跑通并验证有增量价值，再做精细化**。先落表 → 建回测引擎 → 验证信号相对基线有增量 → 再投入龙头增强 / 先验 / 空仓闸门的精细化调参。

```text
M1 universe 过滤（纯函数 + a_stock_st 同步）   —— 一切的地基，被 M5/M6/文档 B 复用
        │
        ▼
M5 limit_up_selected_stock 落表 + 导出接口     —— 先有信号物理产出，QMT 才能消费
        │
        ├─────────────▼
        │         M6 回测引擎                    —— 用历史样本验证"信号 vs 基线"是否有增量
        │              │
        │   ┌──── go/no-go 闸门 ①：信号增量验证 ────┐
        │   │ 回测显示 watchlist 相对"全涨停池等权基线" │
        │   │ 隔日收益/胜率有显著正增量 → 继续；否则停 │
        │   │ 下来回炉信号口径，不急于精细化           │
        │   └──────────────────────────────────────┘
        │              │（通过）
        ▼              ▼
M4 FOCUS JSON-mode + 连板/隔日溢价先验
        │
        ▼
M2 龙头增强层（强度分 / 角色 / 战法）
        │
        ▼
M3 空仓闸门（market_state 决策 + 空仓日落候选行）
        │
   go/no-go 闸门 ②：先验校准 + 空仓闸门有效性
   （上线后用闭环归因校准曲线 ECE、空仓闸门胜率回看，
    详见 closed-loop-attribution-design.md 第 3/5 节）
```

依赖与闸门要点：

- **M1 必须最先做**：`universe_filter` 是 M5 落表、M6 回测样本、文档 B 下单前过滤的共同地基；其依赖的 `a_stock_st` 表同步也在 M1 内完成。
- **M5 先于 M6**：先有 `limit_up_selected_stock` 物理产出，回测才能对照"信号 vs 实际/基线"。但 M5 第一版可只落"未经龙头增强 / 先验的基础信号"（强度分 / 先验字段先留空或给确定性占位），不阻塞落表跑通。
- **go/no-go 闸门 ①（信号增量验证）**：M6 回测产出后设硬闸门——若 watchlist 相对基线无显著增量，**先停止 M4/M2/M3 精细化投入**，回炉 universe / 选股口径；通过才继续。
- **M4 → M2 → M3**：先有 JSON 结构化先验（M4），龙头增强（M2）才有结构化字段可填，空仓闸门（M3）才有完整 `market_state` + 候选行落地；M3 的空仓日候选行是闭环归因"空仓反事实"的样本前提。
- **go/no-go 闸门 ②（精细化验收）**：M2/M3/M4 上线后，用闭环归因的先验校准（ECE、reliability diagram）与空仓闸门胜率回看判断是否达标；不达标则按归因结论调参，而非继续加新模块。

## 二、股票池过滤 universe_filter 与 a_stock_st 同步

### 2.1 目标与三层口径

`universe_filter` 是一个**纯函数**：给定证券代码、名称与"判定基准日"`as_of_date`，返回该票在该日是否进入可交易股票池，以及落选原因与板块归类。三处复用同一函数保证口径一致：

1. **信号侧落表前**（M5）：`limit_up_selected_stock` 落表前过滤候选，落选不入信号表。
2. **回测样本构建**（M6）：历史涨停池逐票按"信号日 T"过滤，剔除当时已 ST / 次新 / 非白名单的票，避免幸存者/口径偏差。
3. **QMT 下单前**（文档 B 复用）：执行侧下单前再过滤一道，防止信号表里混入边界票（详见文档 B）。

三层过滤口径（任一层不通过即落选，`passed=False` 并给 `reason`）：

| 层 | 规则 | 数据来源 | 边界 |
| --- | --- | --- | --- |
| L1 前缀白名单 | `ts_code` 前 3 位 ∈ {`600`,`601`,`603`,`605`,`000`,`001`,`002`,`003`,`300`,`301`}；其余（688/689 科创、8xx/4xx 北交、`.BJ`、920 等）一律落选 | 代码本身（无需查库） | 同时确定 `board`：`600/601/603/605/000/001/002/003`→`MAIN`（主板），`300/301`→`GEM`（创业板） |
| L2 ST 过滤 | 信号日 T（`as_of_date`）当日**是否 ST/\*ST/退市整理**：查新表 `a_stock_st`，命中即落选 | `a_stock_st`（M1 新建，按 `trade_date` 取每日 ST 名单） | **必须按 T 当日判定**，不能用"当前是否 ST"——回测历史样本尤其要用 T 当日 ST 状态，否则未来函数 |
| L3 次新过滤 | 上市未满 6 个**交易日**（含 T 当日不足 6 个开市日）落选 | `a_stock_basic.list_date` + `a_trade_calendar`（数交易日，非自然日） | 用 `a_trade_calendar.is_open=1` 计 `list_date` 到 `as_of_date` 之间的交易日数，`< 6` 落选；`list_date` 缺失视为不可判定，按落选处理并在 `reason` 标注 |

> 名称兜底：L1/L2/L3 不依赖名称做硬过滤，但 `name` 入参用于：① ST 名单按 `ts_code` 主匹配、名称含"ST"做兜底校验（防 `a_stock_st` 同步滞后）；② 落选 `reason` 文案可读。名称含 "ST"/"\*ST"/"退" 且 `a_stock_st` 未命中时，按"疑似 ST"落选并在 `reason` 标注（保守口径，宁可错杀边界票）。

### 2.2 函数放置与签名

放置文件：`backend/app/services/universe_filter.py`（新建，纯函数模块，不持有 DB session，所有外部数据通过入参注入，保证可单测、可被三处复用）。

数据访问分两层，避免纯函数耦合 DB：

- **纯判定核心** `evaluate_universe(...)`：入参为已查好的轻量数据（ST 命中集合、`list_date`、`as_of_date` 对应的交易日序列），无 IO，纯函数，单测友好。
- **便捷封装** `filter_for_trade_date(db, ts_code, name, as_of_date)`：负责查 `a_stock_st` / `a_stock_basic` / `a_trade_calendar`，再调核心；供落表/下单单票调用。批量场景（回测）另给 `build_universe_context(db, as_of_date)` 一次性预取当日 ST 集合与交易日历，避免逐票查库。

签名（伪，类型示意）：

```python
from dataclasses import dataclass
from datetime import date

# 板块枚举：MAIN 主板（±10%），GEM 创业板（±20%）
Board = Literal["MAIN", "GEM"]

@dataclass(frozen=True)
class UniverseVerdict:
    passed: bool          # 是否进入可交易股票池
    reason: str | None    # 落选原因码：OK / NOT_WHITELIST / ST / NEW_LISTING / NO_LIST_DATE / SUSPECT_ST
    board: Board | None   # 板块归类；NOT_WHITELIST 时为 None

# 纯判定核心：无 IO，全部数据由入参注入，便于单测与回测批量复用
def evaluate_universe(
    ts_code: str,
    name: str | None,
    as_of_date: date,
    *,
    is_st_on_date: bool,              # 由调用方按 as_of_date 查 a_stock_st 得到
    list_date: date | None,          # a_stock_basic.list_date
    trade_days_since_list: int | None,  # list_date→as_of_date 的开市交易日数（含边界口径固定）
    new_listing_min_days: int = 6,   # 次新阈值（交易日），默认 6
) -> UniverseVerdict: ...

# 单票便捷封装：内部查库后调 evaluate_universe（落表/下单前用）
def filter_for_trade_date(db, ts_code: str, name: str | None, as_of_date: date,
                          new_listing_min_days: int = 6) -> UniverseVerdict: ...

# 批量上下文：回测一次性预取当日 ST 集合 + 交易日历，逐票判定不再查库
def build_universe_context(db, as_of_date: date) -> "UniverseContext": ...
```

判定核心伪逻辑（顺序短路，任一层落选即返回）：

```text
1. prefix = ts_code[:3]
   if prefix not in WHITELIST: return UniverseVerdict(False, "NOT_WHITELIST", None)
   board = "GEM" if prefix in {"300","301"} else "MAIN"
2. if is_st_on_date: return UniverseVerdict(False, "ST", board)
   # 名称兜底：a_stock_st 未命中但名称含 ST/退 → SUSPECT_ST 落选（防同步滞后）
   if name and looks_like_st(name): return UniverseVerdict(False, "SUSPECT_ST", board)
3. if list_date is None or trade_days_since_list is None:
       return UniverseVerdict(False, "NO_LIST_DATE", board)
   if trade_days_since_list < new_listing_min_days:
       return UniverseVerdict(False, "NEW_LISTING", board)
4. return UniverseVerdict(True, "OK", board)
```

> 配置项：次新阈值 6、新增 `LIMIT_UP_UNIVERSE_NEW_LISTING_MIN_DAYS` 到 `app/core/config.py`，照既有风格 `Field(default=6, alias="LIMIT_UP_UNIVERSE_NEW_LISTING_MIN_DAYS")`（与 `limit_up_push_*` 同区）。白名单前缀集合作为模块常量，不入配置（口径稳定，改动需评审）。

### 2.3 涨跌停按 board 两档取值口径

`universe_filter` 产出的 `board` 是后续涨跌停价计算的唯一依据，统一两档口径（供 M4 先验、M6 回测、闭环归因滑点 `P_signal_limit` 复用，对齐 `qmt-trade-review-closed-loop-attribution-design.md` 第 4 节）：

- `board=MAIN`：涨停价 = `round(pre_close × 1.10, 2)`，跌停价 = `round(pre_close × 0.90, 2)`，四舍五入到分。
- `board=GEM`：涨停价 = `round(pre_close × 1.20, 2)`，跌停价 = `round(pre_close × 0.80, 2)`，四舍五入到分。

固化点：① 用前一交易日收盘 `pre_close`（不复权，取 `tencent_unadjusted_daily_quote` 或日线 `pre_close`）；② 四舍五入到分（与交易所一致，避免一字判定 `low==high==涨停价` 因精度错配）；③ ST 已在 L2 被剔除，本项目不出现 ±5% 档；④ 该口径建议落为 `universe_filter.py` 中的 `limit_up_price(pre_close, board)` / `limit_down_price(...)` 辅助函数，与过滤同模块，三处复用同一实现。

口径与边界：

- **接口须核对**：`api_name` 与字段以官方 `stock_st` 为准；若该接口按 `trade_date` 返回当日 ST 名单则直接 `split_by_trade_date=True` 逐日同步；若官方仅提供 `namechange`（更名流水），则需在同步后处理层把"改名为含 ST 的区间"展开成逐日名单（这一步在 sync 层加确定性转换，DatasetSpec 注释里固化该口径）。**本节标记为待核对决策点，落地前以官方文档/实测为准**。
- **历史回填**：回测样本需要历史每日 ST 状态，`full_start_date` 对齐既有历史起点；首次全量回填 + 日常增量（`split_by_trade_date` 断点续传）。
- **幂等**：`upsert_many` 按 `(ts_code, trade_date)` 唯一键覆盖，重跑/补采不产生重复行。
- **同步缺口降级**：若某 `trade_date` 同步失败/缺数，L2 判定时该日无 ST 记录 → 该层默认"非 ST"，此时由名称兜底（`SUSPECT_ST`）兜一道，但应在 `data_quality` 标注缺口，避免静默漏判。

### 2.6 单测要点

`universe_filter`（纯函数核心，无需 DB，重点覆盖）：

## 三、龙头战法增强层

### 3.0 定位与边界

本层是**确定性后处理层（非 LLM）**：在多阶段 LLM 流水线产出 `selected_*_stocks` 之后、报告状态收口为 READY 之前，对每只入选股做一次纯规则计算，把"龙头强度数值分 + 一组枚举标签"写回信号侧底表 `limit_up_selected_stock`，供后续 QMT 闭环复盘按强度/角色/战法分组归因。

为什么独立成"确定性层"而不是塞进 LLM 提示词：

1. **可回测、可校准**：打分卡每个维度的方向、归一化口径、权重都要能被历史数据反复跑、反复校准；LLM 输出不可复现，无法做 ECE/单调性这类先验校准（先验校准口径详见 `resources/doc/qmt-trade-review-closed-loop-attribution-design.md` 第 3 节）。
2. **可解释、可审计**：每只票的分值能拆回各维度子分，能回答"为什么这只判成总龙头"。
3. **不增加 LLM 调用与 token**：复用流水线已经算好的字段，零额外模型成本。

与既有体系的衔接（**引用不重写**）：

- 输入来源 `context["pipeline"].selected_first_board_stocks / selected_chain_stocks / selected_high_board_stocks` 的结构与生成口径，详见 `resources/doc/limit-up-multi-stage-analysis-refactor-plan.md` 第 6 节及本文档第二章。
- 输出落库表 `limit_up_selected_stock` 的**整表字段清单**已在 `resources/doc/qmt-trade-review-closed-loop-attribution-design.md` 第 1.1 节、`resources/doc/qmt-trade-review-design.md` 第 2.5 节定稿；本层只负责其中 `leader_strength_score`、`leader_strength_pctl`、`role`、`role_confidence`、`strategy_family`、`setup`、`action`、`tradable_flag` 这一**子集**的产出。`continuation_prob`（连板先验）、`next_day_premium_prob`（隔日溢价先验）、`market_state`（情绪周期）、`signal_close` / `limit_up_price` / `reasonable_open_high_low/high`（合理高开区间）由其他章节/文档负责，本层只读不写（详见第 3.7 节字段产出表的"归属"列）。

### 3.2 龙头强度打分卡（核心）

#### 3.2.1 总体设计原则

- **维度与方向**：六个维度，五正一负（位置筹码为负向，越高位、上方套牢越重、扣分越多）。
- **按 `limit_type` 分形态归一**：10cm（主板）与 20cm（创业板）在封流比、首封时间、换手率、振幅上分布差异巨大，**绝不能用同一根固定阈值线**。每个维度在归一化时按 `limit_type` 取对应分组的分布（分位映射），保证两类股的子分可比。本文档限定标的为主板 + 创业板，`limit_type` 取值只用 `10cm` / `20cm`（`_limit_type` 还会返回 `30cm`，但北交所已被项目约束排除，落表前过滤掉）。
- **权重 TBD，由回测校准**：本节给出的权重是初始占位，**最终权重必须由历史回测校准**（用 `qmt_signal_attribution_daily` 的实际隔日兑现做监督信号，详见 `resources/doc/qmt-trade-review-closed-loop-attribution-design.md` 第 6.4 节的分组与校准查询）。校准前后版本通过 `limit_up_leader_scoring_version` 区分。
- **建议分桶 / 树而非固定线性权重**：六维线性加权是 v1 的可解释起点；但封板质量、高度、资金之间存在强非线性与交互（如"高位 + 弱封"急剧恶化、"题材龙头 + 强资金"显著加成），**v2 建议改用单调约束的梯度提升树（GBDT，对各正向维度施加单调递增约束、对位置筹码施加单调递减约束）或人工分桶规则树**，用回测样本拟合，避免线性权重把交互项抹平。打分卡接口（见 3.5）刻意设计成"先算各维度归一化子分 → 再做聚合"两段，聚合函数可插拔（线性 / 树 / 分桶），换聚合器不动子分计算。

#### 3.2.2 六维子分定义（用哪些现有字段、方向、归一口径）

下表的"现有字段"均来自 `selected_*_stocks` 行（即 `_compact_stock_row` 产出，见 service 第 1679-1724 行；以及 `_stock_compact` 内的 `technical` 子字典 `amount_ratio_5d` / `circ_mv` / `turnover_rate`，见 service 第 1494-1499 行）与 `selection`（LLM 选股回写，含 `theme_role` / `leader_role` / `score_detail` 强中弱、`priority` / `risk_level`）。

| 维度 | 方向 | 用到的现有字段 | 归一口径（按 `limit_type` 分形态） |
| --- | --- | --- | --- |
| ① 封板质量 | 正 | `seal_ratio_pct`（**实为封流比** = 封单额/自由流通市值，由 `_safe_pct_ratio(limit_order, free_float)` 算得，见 service 第 1689-1693、1710 行）、`first_limit_time`（首封时间，越早越强）、`open_times`（开板次数，越少越强）、`last_limit_time` | 封流比按 `limit_type` 分组取分位映射到 [0,1]；首封时间越早分越高（用当日分布分位，缺失见 3.4）；开板次数 0 次满分、随次数衰减；三项加权成"封板质量子分" |
| ② 题材梯队 | 正 | `theme`、`selection.theme_role`（板块前排/龙头/跟风）、`selection.leader_role`（空间板/题材龙头/高辨识度），结合市场情绪上下文中该题材的梯队结构 | LLM 给的角色枚举映射成基准分（龙头>前排>跟风）；叠加该票所属题材的整体强度（题材内连板梯队完整度、`_theme_summary` 的 `ladder` 口径，见 service 第 1726-1769 行）。注意题材强度信息此处只读用于打分，不重算题材聚类 |
| ③ 高度 | 正 | `board_level`（连板高度，由 `_board_level` 从 KPL 状态文本解析"N天M板"/"N连"/"首"，见 service 第 1605-1629 行） | 高度本身按当日全市场最高板归一（相对高度），而非绝对板数；首板=低基准、空间板/最高板=高基准。**高度过高同时会触发位置筹码维度的负向**，二者共同建模"高位风险" |
| ④ 资金 | 正 | `technical.amount_ratio_5d`（5 日量比，见 service 第 1494 行）、`turnover_rate`（换手率）、`limit_order` / `max_limit_order`（封单额）、`selection` 中若带龙虎榜净买信号则叠加 | 量比、换手按 `limit_type` 分组分位归一；封单额用封流比口径已在①体现，此处资金子分侧重"承接活跃度"（量能放大 + 换手健康区间），换手过低（缩量一字）和过高（天量分歧）都不给满分，呈倒 U 形 |
| ⑤ 位置筹码 | **负** | `cyq_summary.upper_chip_pressure_pct`（上方筹码压力，来自筹码补数 `_build_selected_stock_supplements` → `_stock_supplement` 的 `cyq_summary`）、`cyq_summary.next_day_premium_bias`、`board_level`（高度越高位置越危险）、相对前高位置 | 上方套牢盘越重、位置越高，**子分越低（负向贡献越大）**。本维度是唯一减分项，用于压制"高位无量套牢票"的总分。筹码字段缺失时按"中性偏保守"处理（见 3.4） |
| ⑥ 辨识度 | 正 | `selection.theme_role` / `leader_role` / `score_detail`（强/中/弱）、`name`（是否市场公认人气标的）、是否当日唯一空间板/总龙头 | 辨识度衡量"市场记忆点 + 资金共识"，主要承接 LLM 选股阶段给出的角色与强中弱评级，叠加"是否当日题材唯一最高板"等确定性信号 |

#### 3.2.3 归一化与聚合伪逻辑

```text
function leader_strength_score(stock, market_ctx):
    lt = normalize_limit_type(stock.limit_type)   # 只保留 10cm / 20cm，30cm(北交所) 已被排除
    dist = market_ctx.dist_by_limit_type[lt]       # 当日按 limit_type 分组的字段分布(分位表)

    # --- 六维子分，全部归一到 [0,1] ---
    s_seal   = seal_quality_subscore(stock, dist)          # ① 封板质量(封流比/首封时间/开板次数)
    s_theme  = theme_ladder_subscore(stock, market_ctx)    # ② 题材梯队
    s_height = height_subscore(stock, market_ctx)          # ③ 相对高度(对当日最高板归一)
    s_money  = money_subscore(stock, dist)                 # ④ 资金(量比/换手倒U/封单)
    s_pos    = position_chip_subscore(stock, market_ctx)   # ⑤ 位置筹码(负向; 越高位/套牢越重 → 子分越低)
    s_recog  = recognition_subscore(stock)                 # ⑥ 辨识度(承接 LLM 角色+强中弱)

    subscores = {seal, theme, height, money, position: s_pos, recognition}

    # --- 聚合：v1 线性占位(权重 TBD, 由回测校准); v2 可换成单调约束树/分桶 ---
    raw = aggregate(subscores, weights=WEIGHTS_V1)   # 见 3.2.4 权重表
    score = round(scale_to_0_100(raw), 2)            # 落库取 0-100 数值分

    return score, subscores   # 子分同时落 score_detail_json, 供审计与回测拆解
```

> 关键约束：`aggregate` 与 `WEIGHTS_V1` 是可插拔的。回测校准产物（新权重 / 树模型 / 分桶规则）只替换 `aggregate` 实现并 bump `limit_up_leader_scoring_version`，六维子分计算不动。

### 3.3 角色多标签（role + 置信）

每只票判定一个**主角色** `role`（枚举）加一个**置信度** `role_confidence`（0-1）。判定规则与置信如下：

| 角色枚举值 `role` | 判定规则（确定性优先 + LLM 辅证） | 置信度口径 |
| --- | --- | --- |
| `MAIN_LEADER`（总龙头） | 当日全市场**最高板** 且 唯一/并列最高；或最高板同高时按 `leader_strength_score` 取头名。需 `selection.leader_role` 含"空间板/题材龙头"佐证 | 确定性条件 + LLM 佐证齐全→高置信；仅强度分领先无 LLM 佐证→中置信 |
| `SECTOR_LEADER`（板块龙头） | 在其 `theme` 内 `board_level` 最高 且 `leader_strength_pctl` 题材内头名；`selection.theme_role` ∈ {龙头, 板块前排} | 题材内高度+强度双第一→高；仅其一→中 |
| `MID_ARMY`（中军，**枚举标识待定**） | 题材内**非最高板但封板质量/资金靠前的中坚票**（高度次于龙头、但封流比与量能位于题材前列）。**该枚举的最终字面值待定**（候选 `MID_ARMY` / `BACKBONE`），需与前端展示文案、QMT 归因分组口径统一后定稿，定稿前 ORM 用 `MID_ARMY` 占位并在【文档B】DDL 注释标注"枚举字面待统一" | 高度+资金双中坚→中高；边界模糊→中 |
| `ASSIST`（卡位助攻） | 题材内卡位补涨、助攻龙头梯队的票；判定较依赖语境，**保留 LLM 文本**：`role_text` 字段直接存 `selection.selection_reason` / `theme_role` 的卡位描述原文，不强行枚举化 | 规则命中弱→低-中置信，以 LLM 文本为准 |
| `STRAGGLER`（杂毛） | 不满足以上任一、且 `leader_strength_pctl` 处于当日同层后段、`selection` 未给正向角色的跟风/弱封票 | 强度分位明确靠后→高置信(确信是杂毛) |

实现要点：

- **角色互斥取主标签**：一只票只落一个 `role`（取置信最高者）；但保留 `role_text`（LLM 卡位/助攻原文）与 `role_candidates_json`（命中过的候选角色 + 各自置信），供前端展示"主角色 + 备注"和回测重判。
- **中军枚举字面待定**：见上表，DDL 用占位值，文案统一后再 bump version 改写历史行（一次性数据迁移，幂等覆盖）。
- **卡位助攻保留 LLM 文本**：这是刻意设计——卡位关系是题材语境强相关的软判断，纯规则易误杀，用 LLM 文本兜住可读性，置信度标低提示下游谨慎使用。

`role_confidence` 统一口径：`确定性硬条件全中 = 0.9`，`确定性条件 + LLM 佐证 = 0.75`，`仅 LLM 文本/单一软条件 = 0.5`，`边界/兜底 = 0.3`（具体分档在 `limit_up_leader_scoring_version` 内固化，回测可调）。

### 3.4 缺失值处理（重要边界）

务必区分"真实缺失"与"零值/合理空"，三类字段的缺失语义不同：

1. **`open_times`（开板次数）真实为 `None`**：KPL 当日未回该字段时 `row.get("open_times")` 为 `None`，**不能当 0（0 = 全天未开板=强势）处理**。`None` 视为"封板质量该子项缺测"，按当日同层中位数填充（中性），并在 `score_detail_json` 标 `open_times_imputed=true`，避免把缺测股误抬成"零开板满分票"。
2. **`bid_*`（`bid_amount` / `bid_change` / `bid_turnover` / `bid_pct_chg`）真实为 `None`**：竞价相关字段缺失时，凡用到竞价信号的子分（主要影响④资金的承接活跃度判断）退化为不使用该信号、用其余可用信号计算，并标 `bid_imputed=true`。**禁止把 `None` 当 0** 参与比值/排序。
3. **`seal_ratio_pct` 语义澄清**：该字段名易误读为"封板时间占比/封成率"，**实为封流比**（封单额 ÷ 自由流通市值，见 service 第 1689-1693 行 `_safe_pct_ratio(limit_order, free_float)`）。本层文档、注释、字段说明一律按"封流比"解释，单测断言也按封流比口径。若 `limit_order` 或 `free_float` 任一缺失，`seal_ratio_pct` 为 `None`，①封板质量子分对应分项退化并标 `seal_ratio_imputed=true`。
4. **筹码 `cyq_summary` 缺失**（筹码补数失败/未补）：⑤位置筹码维度按"中性偏保守"给分（不奖励也不重罚），标 `chip_imputed=true`，避免缺筹码就给高分。

统一原则：**任何 imputed 字段都在 `score_detail_json` 留痕**，可成交性与角色判定不依赖被 impute 的高风险字段做"利好"推断；缺测只会让分更保守，不会凭空抬分。

### 3.5 战法适配标签（strategy_family + setup + action）

每只票产出三个战法相关标签，**`action` 复用既有"重点/谨慎/放弃"三档**（对齐 `resources/doc/limit-up-multi-stage-analysis-refactor-plan.md` 第 6.3 节高连板分层、`resources/doc/limit-up-push-investment-advice-refactor-plan.md` 第 3.2.2 节候选分层口径，不新造一套）：

| 标签 | 枚举取值 | 确定性判定依据 |
| --- | --- | --- |
| `strategy_family`（战法族） | `DABAN`(打板) / `BANLU`(半路) / `DIXI`(低吸) | 由 封板状态 + 首封时间 + 当前是否封死 推断：早盘强封死→打板族；盘中拉升未/刚封、回封→半路；高位/分歧票次日竞价回踩→低吸。结合 `limit_type` 分形态（20cm 半路与低吸空间更大） |
| `setup`（具体形态） | 如 `首板打板` / `连板打板` / `空间板半路` / `中军低吸` / `卡位低吸` 等组合标签 | `strategy_family` × `role` × `board_level` 组合派生；标签集合在 version 内固化 |
| `action`（操作建议档） | `FOCUS`(重点) / `CAUTION`(谨慎) / `GIVE_UP`(放弃) | 综合 `leader_strength_pctl` + `role` + ⑤位置筹码 + `tradable_flag` 推断：强度分位高且角色硬且非高位套牢→重点；高位/弱封/筹码压力大→谨慎；杂毛/可成交性差/高位无量→放弃。**与 LLM 建议分层枚举同名同义**，便于报告/建议/信号三处口径一致 |

> `action` 是确定性档位，与 LLM 投资建议正文里的分层是"同口径但独立产生"：信号侧 `action` 用于 QMT 归因的可执行性分组，LLM 正文用于人读。两者枚举值一致（重点/谨慎/放弃），便于交叉核对"机器判档 vs 模型判档"的一致率。

### 3.6 可成交性标注 `tradable_flag`

`tradable_flag` 标注"次日这只票按打板/半路逻辑**到底买不买得进**"，直接服务回测与闭环的"一字/秒封买不进不计收益"硬约束（对齐项目总约束与 `qmt-trade-review-board-design.md` 第 226 行决策价口径）。

取值与判定（确定性）：

| `tradable_flag` | 含义 | 判定规则 |
| --- | --- | --- |
| `0`（不可成交 / 买不进） | 一字板 / 秒板（开盘即封死、全天未开板） | 满足任一：`open_times` 真实为 0 且首封时间 = 开盘瞬时（一字）；或 `first_limit_time` 极早且 `open_times`=0（秒封一字）。**注意**：此处 `open_times`=0 必须是"真实为 0"，`None`（缺测）不得判成一字（见 3.4 第 1 条），缺测时 `tradable_flag` 给保守的"未知不可成交"或单独 `null`/`2` 档，口径在 version 固化 |
| `1`（可成交） | 盘中有开板 / 非一字封，次日有合理买点 | 不满足上述一字/秒封条件，且存在可成交窗口（开板次数≥1 或属半路/低吸形态） |

实现要点：

- `tradable_flag=0` 的票，闭环归因走"买不进反事实组"（用理论涨停价/合理买入价算反事实收益，详见 `qmt-trade-review-closed-loop-attribution-design.md` 第 3.1 节盯市口径"买不进反事实组"），**不计真实成交收益**。
- 一字/秒封判定**强依赖** `open_times` / `first_limit_time` / `last_limit_time` 的真实性；这三项任一缺测，宁可判"不确定不可成交"也不要乐观判成可成交（防止把买不进的一字票误算进可成交收益）。

```text
         + pipeline.selected_chain_stocks
```

### 3.8 关键方法签名（新 service）

```python
# backend/app/services/limit_up_leader_scoring_service.py

class LeaderScoringService:
    """龙头战法增强层：确定性后处理，给每只入选股算强度分+角色/战法/可成交性标签。"""

    def score_batch(
        self,
        rows: list[dict[str, Any]],          # selected_*_stocks 合并去重后的 compact 行
        market_ctx: "MarketScoringContext",  # 情绪周期 + 当日分 limit_type 分布 + 当日最高板
        trade_date: date,                    # 信号日 T
        target_trade_date: date,             # T+1 买入日(已经 a_trade_calendar 映射)
    ) -> list["LeaderSignal"]: ...

    def score_one(self, row, market_ctx) -> "LeaderSignal": ...

    # 六维子分(均返回 0-1, 按 limit_type 分形态归一; imputed 写回 detail)
    def _seal_quality_subscore(self, row, dist) -> tuple[float, dict]: ...
    def _theme_ladder_subscore(self, row, market_ctx) -> tuple[float, dict]: ...
    def _height_subscore(self, row, market_ctx) -> tuple[float, dict]: ...
    def _money_subscore(self, row, dist) -> tuple[float, dict]: ...
    def _position_chip_subscore(self, row, market_ctx) -> tuple[float, dict]: ...  # 负向
    def _recognition_subscore(self, row) -> tuple[float, dict]: ...

    def _aggregate(self, subscores: dict[str, float]) -> float: ...   # 可插拔: 线性/树/分桶
    def _classify_role(self, row, market_ctx, score, pctl) -> tuple[str, float, str, list]: ...
    def _classify_strategy(self, row, role, score) -> tuple[str, str, str]: ...  # family/setup/action

```

`LeaderSignal` 为落库前的纯数据对象（dataclass / dict），字段对齐 3.7 写列。`MarketScoringContext` 封装"当日按 `limit_type` 分组的字段分位表 + 当日最高板 + 题材梯队 + market_state（只读）"，由 `build_market_ctx(context)` 从 `context["market_context"]` / `_emotion_cycle_metrics`（service 第 1856-1908 行）/ `selected_*` 派生。

## 四、信号侧空仓闸门

> 衔接说明：本节定义的 `sentiment_cycle`（六档情绪周期）与 `market_state`（三档可参与状态）是信号侧 `limit_up_selected_stock` 顶层与每行的固化字段。该表的下游消费（复盘看板分组、QMT 成交回流 join、信号-执行-结果闭环归因、空仓日反事实回看）已在 `resources/doc/qmt-trade-review-design.md`（第 2.5 节「与 limit_up_selected_stock 的关联口径」）、`resources/doc/qmt-trade-review-board-design.md`（第 「信号回挂/漏单复盘」「只读视图 v_qmt_fill_funnel」段）、`resources/doc/qmt-trade-review-closed-loop-attribution-design.md`（第 1 节「数据来源与关联键」、第 5 节「空仓闸门有效性回看（行情反事实）」）三篇定稿。本节只产出 `sentiment_cycle`/`market_state`/`action` 三个字段及其确定性判定逻辑，不重复展开复盘看板、回流与归因。

### 4.1 设计意图与边界

空仓闸门是"在错误的情绪周期里强制不开新仓"的确定性风控前置。它不依赖 LLM 判断，而是用盘后已算好的 `_market_emotion`/`_emotion_cycle_metrics` 数值，经一张固定阈值表，先映射出全市场 `sentiment_cycle`（六档：冰点/启动/发酵/高潮/分歧/退潮），再映射出 `market_state`（三档：参与/谨慎参与/空仓）。该结果落入 `limit_up_selected_stock` 顶层一行（当日全市场判定）与每个候选行（每行携带当日市场态，便于复盘看板按行 join 时无需再回查顶层）。

强约束：

- **确定性、可复算**：同一份 `_emotion_cycle_metrics` 输入永远得到同一档位，禁止引入随机/LLM；这是闭环归因第 5 节「空仓闸门反事实」能成立的前提（反事实必须能离线重放当时的闸门决策）。
- **输入字段全部为 T 日盘后已知量**：闸门在 T 日盘后随报告生成时计算，决定 T+1（`target_trade_date`）是否开新仓；不得使用任何 T+1 盘中/盘后才可得的数据，否则构成未来函数。
- **阈值版本固化**：每次判定写入 `threshold_version`（如 `gate-v1`），阈值表改一次版本号进一次；复盘归因据此区分"是闸门口径变了还是行情变了"。
- **只做主板 + 创业板**（排除 ST/科创/北交），与全局口径一致。
- **空仓日仍落表留痕**：`market_state=空仓` 当日，候选行照常落 `limit_up_selected_stock`（候选集照常产出，只是 `action=放弃`、`tradable_flag` 关闭下单），不得不落行——否则闭环归因第 5 节的反事实组合会退化成"取当日全市场涨停池"的次优口径。

### 4.2 放置位置（文件路径与新增/改动点）

| 类型 | 位置 | 说明 |

### 4.3 输入字段（全部 T 日盘后）

闸门只消费下列已存在的盘后指标（来源：`_market_emotion` service:1810-1841 与其内嵌 `emotion_cycle` = `_emotion_cycle_metrics` service:1856-1908），不新增数据采集：

| 输入字段 | 来源路径（context.market_emotion.*） | 含义 | 方向 |
| --- | --- | --- | --- |
| `limit_up_count` | `limit_up_count` | T 日涨停家数 | 越多越热 |
| `broken_board_rate_pct` | `emotion_cycle.broken_board_rate_pct` | 炸板率%（炸板池剔除回封后 / 涨停∪炸板并集，service:1903-1904） | 越高越弱 |
| `limit_down_count` | `limit_down_count` | T 日跌停家数 | 越多越弱 |
| `highest_chain` | `highest_chain` | T 日最高连板高度 | 越高情绪越强（高潮信号） |
| `highest_chain_change` | `emotion_cycle.highest_chain_change` | 最高板较昨日变化（service:1907/1964） | 升=进攻，降=退潮 |
| `prev_limit_up_premium.avg_pct_chg` | `emotion_cycle.prev_limit_up_premium.avg_pct_chg` | 昨日涨停今日平均涨幅%（隔日溢价，service:1958） | >0 赚钱效应，<0 亏钱效应 |
| `advancement["1进2"].rate_pct` | `emotion_cycle.advancement["1进2"].rate_pct` | 首板晋级二板率%（service:1878-1884） | 高=接力顺畅 |
| `advancement["2进3"].rate_pct` | `emotion_cycle.advancement["2进3"].rate_pct` | 二板晋级三板率% | 高=中位接力顺畅 |
| `advancement["3进4"].rate_pct` | `emotion_cycle.advancement["3进4"].rate_pct` | 三板晋级四板率% | 高=高位接力顺畅 |

边界与缺测口径：

- 任一晋级率分母为 0（昨日无对应板高度的票）时，`_emotion_cycle_metrics` 已返回 `rate_pct=0`（service:1881-1883 经 `_safe_pct_ratio`）；闸门将"分母为 0 的 0 值"与"真实 0% 晋级"区分对待——分母为 0 视为该维度无信息，不作为弱市硬否决证据（避免新股日/数据缺失误判空仓）。
- `prev_limit_up_premium.avg_pct_chg` 为 `None`（无昨日涨停样本或行情缺测）时，赚钱效应维度按"中性"处理，不参与否决，并在判定产物 `gate_reasons` 中标注 `premium_unknown`。
- `highest_chain` 为 `None`（当日无可识别连板）时按 0 处理，等价于"无连板梯队"，是冰点/退潮的支持信号之一。

### 4.4 硬否决条件表（主板先验 + 创业板单列，阈值 TBD）

闸门分两步：**先硬否决判空仓**，未触发否决再进六档分类。硬否决用于捕捉"无论六档如何都不该开新仓"的极端弱市/混沌态。主板与创业板因涨跌幅制度不同（主板 ±10%、创业板 ±20%，创业板情绪波动更剧烈、炸板/亏钱效应阈值需更宽容），创业板阈值**单列**，不复用主板先验。

> 阈值 TBD：下表数值为占位先验，须在回测对账（用闭环归因第 5 节空仓日反事实组合收益验证"闸门误杀率/避损率"）后定稿，定稿即固化 `threshold_version=gate-v1`。任一阈值调整必须进版本号。

| 否决项 | 主板阈值（先验，TBD） | 创业板阈值（单列，TBD） | 触发即 | 理由 |
| --- | --- | --- | --- | --- |
| 炸板率过高 | `broken_board_rate_pct ≥ 45` | `broken_board_rate_pct ≥ 55` | 空仓 | 封板失败率过半，次日溢价极不确定 |
| 跌停家数过多 | `limit_down_count ≥ 30` | `limit_down_count ≥ 20`（合计口径） | 空仓 | 系统性亏钱效应，恐慌主导 |
| 隔日溢价深度为负 | `prev_limit_up_premium.avg_pct_chg ≤ -3` | `avg_pct_chg ≤ -4` | 空仓 | 昨日涨停今日大面，接力即亏 |
| 涨停家数枯竭 | `limit_up_count ≤ 15`（全市场口径） | — | 空仓 | 无赚钱效应、无承接，冰点 |
| 接力链断裂 | `1进2 率 ≤ 10` 且分母 ≥ 阈值（如 ≥10）| `1进2 率 ≤ 8` 且分母 ≥ 阈值 | 空仓 | 首板无法晋级，断板潮 |

否决口径说明：

- 否决项 OR 关系——命中任意一条即 `market_state=空仓`，并记录命中项到判定产物 `gate_reasons`（如 `["broken_board_rate_high"]`）。
- 涨停/跌停家数为全市场口径（主板+创业板合计），不按板分拆；创业板单列仅作用于"率"类阈值（炸板率、溢价、晋级率）与"创业板内部跌停"维度，反映创业板更高波动容忍度。
- 晋级率类否决必须叠加"分母 ≥ 阈值"前置（见 4.3 边界），防止小样本误杀。
- 阈值统一以 `Field(alias=...)` 暴露（如 `LIMIT_UP_GATE_BROKEN_RATE_VETO_MAIN`、`..._GEM`），默认值即上表，配置只调数值不改逻辑。

### 4.5 六档 sentiment_cycle 与 market_state 映射（单调一致 + 一致性断言）

未触发硬否决后，按"市场强度综合序"映射六档。六档语义沿用 `limit-up-push-investment-advice-refactor-plan.md`（第 「周期约束」段：退潮/分歧下调接力评级、冰点只输出观察清单）与既有 `_stage_system_prompt` 注入的情绪周期定位，保持口径一致。

六档定义与判定主导维度（阈值 TBD，与 4.4 同版本固化）：

| sentiment_cycle | 语义 | 主导判定维度（合成序） |
| --- | --- | --- |
| 冰点 | 情绪冰封、无承接 | `limit_up_count` 极低 + `avg_pct_chg` ≤0 + `highest_chain` 低 |
| 启动 | 冰点后回暖、首板放量 | `limit_up_count` 回升 + `1进2 率` 改善 + `avg_pct_chg` 转正 |
| 发酵 | 梯队成型、接力顺畅 | `1进2/2进3 率` 高 + `avg_pct_chg` 明显为正 + `highest_chain_change` ≥0 |
| 高潮 | 高度板兑现、情绪顶 | `highest_chain` 高 + `highest_chain_change` ≥0 + `avg_pct_chg` 高 |
| 分歧 | 高位分歧、炸板抬头 | `broken_board_rate_pct` 抬升 + `avg_pct_chg` 走弱 + `highest_chain_change` ≤0 |
| 退潮 | 亏钱效应主导、退潮 | `broken_board_rate_pct` 高 + `avg_pct_chg` <0 + `limit_down_count` 抬升 |

六档 → market_state 映射（**单调一致**约束：可参与度沿"冰点→退潮的弱、启动→发酵→高潮的强"两条单调链分布，不得出现"弱档反而更可参与"）：

| sentiment_cycle | market_state | 顶层 action 缺省 | 每行 action 取值域 |
| --- | --- | --- | --- |
| 冰点 | 空仓 | 放弃 | 放弃 |
| 退潮 | 空仓 | 放弃 | 放弃 |
| 分歧 | 谨慎参与 | 谨慎观察 | 重点观察/谨慎观察/放弃观察（下调评级） |
| 启动 | 谨慎参与 | 谨慎观察 | 重点观察/谨慎观察/放弃观察 |
| 发酵 | 参与 | 重点观察 | 重点观察/谨慎观察/放弃观察 |
| 高潮 | 参与 | 重点观察 | 重点观察/谨慎观察/放弃观察 |

> `action` 取值域复用 `limit-up-multi-stage-analysis-refactor-plan.md`（第 6.3 节「重点观察/谨慎观察/放弃观察」分层）与 `limit-up-push-investment-advice-refactor-plan.md`（第 「候选标的分层」段）既有口径，不新造词。空仓档每行 `action` 强制为"放弃"，覆盖任何个股层的乐观分层（闸门优先级最高）。

**一致性断言**（实现为模块内 `assert` + 单测，违背即抛 `ValueError`，禁止静默放行）：

1. **映射单调性**：定义 `state_rank = {空仓:0, 谨慎参与:1, 参与:2}`，断言映射表中"强度更高的档位"其 `state_rank` 不低于"强度更低的档位"——具体以一张固定的 `cycle → state` 字典实现，启动时校验该字典与 4.5 表完全一致。
2. **空仓 ⇒ 放弃**：断言 `market_state==空仓` 时顶层 `action` 与所有行 `action` 必为"放弃"。
3. **硬否决优先**：断言任一硬否决命中时 `market_state` 必为空仓，且 `sentiment_cycle ∈ {冰点, 退潮}`（否决与档位不冲突）。
4. **取值域封闭**：断言 `sentiment_cycle ∈ 六档枚举`、`market_state ∈ 三档枚举`、`action ∈ 四枚举（放弃/重点观察/谨慎观察/放弃观察）`，否则抛错。

### 4.6 空仓日产物形态（落表留痕 / action=放弃 / QMT 不开新仓但仍卖出 / 幂等）

`market_state=空仓` 当日的产物口径：

1. **仍落表留痕**：候选集照常经多阶段 pipeline 产出（service:2172-2183 `pipeline.selected_*`），照常映射回 `limit_up_selected_stock` 每行。空仓不是"不落行"，而是"落行但关闸"。这是闭环归因第 5 节反事实组合"取当日本应入选候选"口径成立的硬前提（详见 `qmt-trade-review-closed-loop-attribution-design.md` 第 5 节第 1 点）。
2. **action = 放弃**：顶层行与每个候选行 `action` 均为"放弃"，`market_state=空仓`，`sentiment_cycle ∈ {冰点, 退潮}`，并写 `gate_reasons`、`threshold_version`。
3. **tradable_flag 关闭下单**：候选行 `tradable_flag` 标记为不可下单（与 `qmt-trade-review-design.md` 第 2.5 节漏斗口径一致），QMT 执行侧据此**不开新仓**（不挂买单）。
4. **QMT 不开新仓但仍卖出**：空仓闸门只关"开新仓"，**不关平仓**。已持仓（T+1 之前买入、当日可卖部分）仍按既有卖出策略正常卖出——闸门是"不再加仓的进攻闸门"，不是"清盘开关"；这与 `qmt-trade-review-board-design.md`（T+1 约束：当日买入计入 volume 但不可卖，已实现盈亏只来自当日卖出）的口径不冲突。该约束写入字段语义：`market_state=空仓` 仅约束 `action/tradable_flag` 的"买入侧"，对持仓卖出无效。
5. **幂等**：闸门落表以 `(trade_date, ts_code)`（每行）与 `(trade_date)`（顶层）为幂等键，重跑同一 T 日覆盖写同值（确定性保证同输入同输出）；落表放在 `ensure_analysis_for_trade_date` 的同一事务内（service:399-402 commit 前），与 `analysis` 状态机一起提交，避免"报告 READY 但闸门未落"的中间态。重跑/重试场景（`_reset_analysis_for_retry`）需一并清理同 trade_date 旧候选行后重写，口径详见【文档B】落表章节。

### 4.7 判定表（确定性映射，单表可复算）

下表为闸门完整决策表（先否决后分类），所有行可由 4.3 输入直接复算，阈值占位 TBD，定稿固化于 `threshold_version`：

| 步骤 | 判定条件（按序短路） | sentiment_cycle | market_state | action |
| --- | --- | --- | --- | --- |
| 否决-1 | 命中 4.4 任一硬否决（主板/创业板分列） | 冰点 或 退潮（按主导维度细分） | 空仓 | 放弃 |
| 分类-冰点 | `limit_up_count` 极低 且 `avg_pct_chg ≤0` 且 `highest_chain` 低 | 冰点 | 空仓 | 放弃 |
| 分类-退潮 | `broken_board_rate_pct` 高 且 `avg_pct_chg <0` 且 `limit_down_count` 抬升 | 退潮 | 空仓 | 放弃 |
| 分类-分歧 | `broken_board_rate_pct` 抬升 且 `highest_chain_change ≤0` 且 `avg_pct_chg` 走弱 | 分歧 | 谨慎参与 | 缺省谨慎观察（行级允许下调） |
| 分类-启动 | `limit_up_count` 回升 且 `1进2 率` 改善 且 `avg_pct_chg` 转正 | 启动 | 谨慎参与 | 缺省谨慎观察 |

| 分类-高潮 | `highest_chain` 高 且 `highest_chain_change ≥0` 且 `avg_pct_chg` 高 | 高潮 | 参与 | 缺省重点观察 |

### 4.9 单元测试要点

针对 `sentiment_gate.py`（纯函数，最易测）：

- **六档边界**：为每档构造刚好越过/未越过阈值的输入，断言落档正确（共 6 档 × 边界两侧）。
- **硬否决优先**：炸板率/跌停/溢价/涨停枯竭/接力断裂逐项命中，断言 `market_state=空仓 且 sentiment_cycle∈{冰点,退潮} 且 action=放弃`，且 `gate_reasons` 含对应键。
- **主板 vs 创业板分列**：同一组率类指标，主板阈值未触发、创业板阈值触发（或反之），断言按 `board_scope`/分列阈值给出不同结论。
- **单调一致断言**：构造一组覆盖六档的输入，断言 `state_rank` 沿强度序非递减；篡改 `_CYCLE_TO_STATE` 字典使其违背单调性，断言模块加载/调用抛 `ValueError`。
- **缺测边界**：晋级率分母为 0、`avg_pct_chg=None`、`highest_chain=None` 三种缺测，断言不误判空仓且 `gate_reasons` 标注 `*_unknown`。
- **空仓 ⇒ 放弃覆盖**：空仓档下即使个股层给"重点观察"，断言每行 `action` 被强制为"放弃"。
- **可复算/确定性**：同一输入连续调用 N 次返回完全相等（含 `gate_reasons` 顺序稳定）。
- **threshold_version 固化**：断言产物 `threshold_version == SENTIMENT_GATE_THRESHOLD_VERSION`。

针对接入点（service 层，用既有 `context` fixture）：

- **落表幂等**：同 trade_date 重跑两次，顶层 1 行、候选行数不翻倍、字段值一致。
- **空仓不丢行**：构造空仓输入，断言候选行仍落、`tradable_flag` 关闭、`action=放弃`。
- **同事务**：闸门落表异常时 `analysis` 不进 READY（回滚一致）。

### 4.10 验收标准

- `sentiment_gate.py` 为无副作用纯函数，六档/三档/四 action 取值域封闭，非法输入抛 `ValueError`。
- 硬否决表（主板先验 + 创业板单列）阈值以 `config.limit_up_gate_*` 暴露，默认值即固化表，改阈值只调配置不改逻辑；阈值定稿后 `threshold_version` 固化、每次调整进版本号。
- 六档 → market_state 映射满足单调一致，并由启动期断言 + 单测双重保证；空仓 ⇒ action=放弃在顶层与每行均成立。
- `limit_up_selected_stock` 顶层与每行均写入 `sentiment_cycle`/`market_state`/`action`/`threshold_version`；空仓日候选行仍落表、`tradable_flag` 关闭，QMT 据此不开新仓但持仓卖出不受闸门影响。
- 落表与 `analysis` 状态机同事务、按 `(trade_date)`/`(trade_date, ts_code)` 幂等，重跑结果一致。
- 闸门产物可被闭环归因第 5 节空仓日反事实回放直接消费（候选行齐全 + 可复算），无需退化为全市场涨停池口径。
- 全部单测（4.9）通过；阈值占位 TBD 项在文档中明确标注待回测对账定稿，未定稿不影响逻辑可运行与可测。

## 五、FOCUS 阶段 JSON-mode 改造与连板先验结构化

> 本节针对三个重点分析阶段（`FIRST_BOARD_FOCUS` / `CHAIN_FOCUS` / `HIGH_BOARD_FOCUS`）从「纯 HTML 文本阶段」改造为「JSON-mode 结构化阶段」。多阶段流水线整体编排、各阶段 I/O、`selected_*` 结构详见 `resources/doc/limit-up-multi-stage-analysis-refactor-plan.md` 第 6 节（多轮 LLM 节点）与第 8.2 节（阶段缓存表）；首板精选阶段（`FIRST_BOARD_SELECTION` / `FIRST_BOARD_FOCUS`）的新增背景详见 `resources/doc/limit-up-push-investment-advice-refactor-plan.md` 第 263–274 行。本节只做 JSON-mode 改造增量，不重述上述流水线。本节产出的结构化先验字段（`continuation_prob` 等）是后续闭环归因与先验校准的输入来源，对接口径详见 `resources/doc/qmt-trade-review-closed-loop-attribution-design.md`（`qmt_signal_attribution_daily` 先验校准/逆向选择小节），本节不展开归因侧实现。

### 5.1 改造目标与动机

当前三个 FOCUS 阶段只产出 HTML 片段（`html_fragment`），其中已隐含「晋级可能性、次日溢价可能性、触发条件、失败条件」等关键先验判断，但全部以自然语言混在 HTML 里，下游（报告渲染、闭环归因、先验校准）无法可靠抽取。改造目标：

1. **结构化先验落地**：在同一次 LLM 调用里，除 `html_fragment` 外，同步产出机器可读的先验字段（`continuation_prob` 档位、`next_day_premium_prob`、`boost_conditions[]`、`fail_conditions[]`、`suggested_hold_thesis`），供后续归因/校准消费。
2. **报告渲染零回退**：`html_fragment` 仍由 LLM 生成，最终合成阶段（`FINAL_REPORT`）的输入键 `first_board_focus_html` / `chain_focus_html` / `high_board_focus_html` 语义不变，报告外观不变。
3. **READY 率不回退**：现状 `_run_text_stage` 对这三个 FOCUS 阶段有「失败可降级、报告仍 READY」的容错（`limit_up_push_service.py:2269-2285`）。迁移到 JSON 路径后必须等价保留，否则 LLM 偶发失败会把整份报告打成 FAILED，READY 率下降。

### 5.2 改造前后对照

| 维度 | 改造前（现状） | 改造后（目标） |
| --- | --- | --- |
| 执行路径 | `_run_text_stage(2242)` | `_run_json_stage(2188)` |
| LLM 调用 | `_chat_completion_with_reasoning(prompt, system_prompt, phase=...)`（不带 `json_mode`） | `_chat_completion_with_reasoning(prompt, system_prompt, json_mode=True, phase=...)`，即 `response_format={"type":"json_object"}`（`limit_up_push_service.py:3137-3139`） |
| LLM 输出 | 纯 HTML 片段 | 单一 JSON 对象（含结构化先验 + `html_fragment`） |
| 阶段返回 payload | `{"content", "html_fragment"}` | `{"html_fragment", "continuation_prob", "next_day_premium_prob", "boost_conditions", "fail_conditions", "suggested_hold_thesis", "stock_priors":[...]}` |
| `final_input` 取值 | `first_board_focus.get("html_fragment")` 等（`:2161-2163`） | **保持不变**，仍取 `html_fragment` 键；JSON payload 里该键照常存在 |
| `context["pipeline"]` | `*_focus_html` 取 `html_fragment`（`:2179-2181`） | **保持不变**取 `html_fragment`；可额外新增 `*_focus_priors` 存结构化先验（见 5.6） |
| 失败容错 | text 阶段对三 FOCUS 走 `_fallback_text_stage` 降级、报告仍 READY | 必须等价：JSON 阶段对三 FOCUS 失败/解析失败时落含 `html_fragment` 的降级 payload、报告仍 READY（见 5.5） |
| 阶段缓存 `prompt_version` | 三 FOCUS 走默认后缀 `v3`（`_STAGE_PROMPT_VERSION_SUFFIXES` 未列）（`:3075-3087`） | 三 FOCUS 显式登记新后缀（如 `focus-json-v1`），与旧 HTML 缓存物理隔离 |

关键约束：**DeepSeek `json_object` 要求整条消息为单一 JSON 对象**，不能「HTML 片段 + 独立 JSON」并存。因此 `html_fragment` 必须作为 JSON 的一个字符串字段内嵌（HTML 文本被 JSON 转义存储），不能让模型先吐 HTML 再吐 JSON。

### 5.3 现有 `_run_json_stage` 复用分析（无需新建路径）

`_run_json_stage(2188-2240)` 已具备本次所需全部基建，FOCUS 改造**复用**而非新建：

- 缓存读取：`_stage_cache_payload`（命中记 `CACHE_HIT`）。
- 调用：`_chat_completion_with_reasoning(user_prompt, system_prompt, json_mode=True)`（`:2208-2210`）。
- 解析：`_extract_json_payload(content)`（`:3028-3046`，剥 ```` ```json ```` 围栏 + 取首个 `{` 到末个 `}`）。
- **解析失败兜底**：`payload is None` 时回退 `fallback_payload` 并打 `parse_fallback=True` + 质量项 `PARSE_FALLBACK`（`:2212-2219`）。
- **调用异常兜底**：`except` 分支落 `fallback_payload` + `error_fallback=True` + 质量项 `FAILED_FALLBACK` + `failed=True` 写缓存（`:2227-2240`）。
- 缓存写入：`_save_stage_cache(stage_key, stage_input, payload, payload.get("html_fragment"))`（`:2225`），即 `content_html` 取 payload 的 `html_fragment`，天然适配本次结构。

唯一缺口：`_run_json_stage` 当前所有 `fallback_payload` 由各阶段 `_fallback_*` 函数提供，但 FOCUS 的兜底逻辑（按入选名单生成降级观察表）只存在于 `_fallback_text_stage(2291)`。改造需提供一个产出「含 `html_fragment` 的降级 payload」的 FOCUS 专用兜底（见 5.5），作为 `fallback_payload` 传入既有 `_run_json_stage`，**不改动 `_run_json_stage` 本身**——这一点很重要，因为它保证「解析失败 → `PARSE_FALLBACK`、调用异常 → `FAILED_FALLBACK`、两种情况都仍返回带 `html_fragment` 的 payload、报告仍 READY」的容错与现有 text 路径等价。

### 5.4 新 JSON Schema

```jsonc
  "html_fragment": "<h3>两连三连重点接力</h3>...",

  // 阶段级整体先验（覆盖该 FOCUS 阶段入选股的总体判断，便于报告级聚合）
  "continuation_prob": "高",            // 连板/晋级延续概率档位，枚举：高 | 中 | 低 | 极低
  "next_day_premium_prob": "中",        // 次日竞价/盘中溢价概率档位，枚举：高 | 中 | 低 | 极低
  "boost_conditions": ["题材发酵升级", "竞价高开 3%~5% 且承接放量"],  // 加分/触发条件
  "fail_conditions": ["竞价弱于 -2%", "首封后多次开板"],              // 失败/止损条件
  "suggested_hold_thesis": "仅作打板试错，断板即出，不留隔夜底仓",     // 持有逻辑一句话

  // 个股级先验数组：每只入选股一项，ts_code 与 selected_* 行对齐，供逐股归因
  "stock_priors": [
    {
      "ts_code": "600000.SH",
      "name": "示例股",
      "continuation_prob": "中",         // 同档位枚举
      "next_day_premium_prob": "高",
      "boost_conditions": ["板块龙头卡位"],
      "fail_conditions": ["竞价弱于 -1%放弃"],
      "suggested_hold_thesis": "竞价不弱则打，盘中破板止损"
    }
  ]
}
```

档位枚举统一为 `高 / 中 / 低 / 极低` 四档（与既有选股阶段 `强/中/弱` 区分：那是质量评分，这是概率先验）。`boost_conditions` / `fail_conditions` 为字符串数组。`stock_priors[].ts_code` 必须能映射回 `selected_first_board_stocks` / `selected_chain_stocks` / `selected_high_board_stocks` 的行（由 `_select_stage_stocks(2695)` 产出），否则归因侧按缺失处理。

阶段差异（提示词约束，schema 形状一致）：

- `FIRST_BOARD_FOCUS`：先验侧重「晋级二板可能性」，须显著标注首板接力不确定性高于连板接力（沿用 `_first_board_focus_prompt:2520-2521` 既有口径）。
- `CHAIN_FOCUS`：先验侧重「晋级三板/四板可能性」，20cm 标的须在 `fail_conditions` 体现更深断板回撤（沿用 `:2587`）。
- `HIGH_BOARD_FOCUS`：允许「重点/谨慎/放弃」分层，`continuation_prob` 应整体偏保守，须用 `emotion_cycle` 约束高位接力口径（沿用 `:2611-2614`）。

### 5.5 降级兜底（READY 率等价迁移，最关键）

现状 text 路径容错语义（`limit_up_push_service.py:2269-2285`）：三 FOCUS 阶段 LLM 失败时调 `_fallback_text_stage` 生成静态观察表 HTML，记 `FAILED_FALLBACK`、`failed=True` 写缓存、**不抛异常**，报告继续；非 FOCUS 阶段（如 `FINAL_REPORT`）失败直接 `raise`，整份报告进 FAILED 重试。改造后必须保持完全等价，做法：

1. **新增 FOCUS 专用兜底构造器** `_fallback_focus_stage(stage_key, stage_input, error_message=None) -> dict`：复用 `_fallback_text_stage(2291-2351)` 已有的「按入选名单 + 筹码摘要生成观察表 HTML」逻辑，产出 payload 至少含：
   - `html_fragment`（降级观察表 HTML，标题按阶段区分，沿用 `:2307-2312`）；
   - 结构化先验字段全部给保守缺省值：`continuation_prob="极低"`、`next_day_premium_prob="极低"`、`boost_conditions=[]`、`fail_conditions=["LLM 先验缺失，仅作观察不参与"]`、`suggested_hold_thesis="先验降级，建议仅观察"`、`stock_priors=[]`；
   - 标志位由 `_run_json_stage` 自动追加（`parse_fallback` / `error_fallback`），无需在此设置。
2. **接入既有 JSON 路径**：在 `_generate_multi_stage_llm_report` 中把三处 `_run_text_stage(LIMIT_UP_STAGE_*_FOCUS, ...)`（`:2114-2149`）改为 `_run_json_stage(LIMIT_UP_STAGE_*_FOCUS, stage_input, system_prompt, json_prompt, self._fallback_focus_stage(...), stage_quality)`。这样：
   - LLM JSON 解析失败 → `_run_json_stage:2212-2219` 用该兜底 payload + `PARSE_FALLBACK`，payload 含 `html_fragment`，报告仍 READY；
   - LLM 调用异常 → `_run_json_stage:2227-2240` 用该兜底 payload + `FAILED_FALLBACK` + `failed=True` 写缓存，报告仍 READY。
3. **`final_input` / `pipeline` 取值不变**：因兜底 payload 永远含 `html_fragment`，`first_board_focus.get("html_fragment")`（`:2161-2163`、`:2179-2181`）在成功与降级两种情况下都拿得到非空 HTML，最终合成不会因 FOCUS 降级而缺段。
4. **不污染 `FINAL_REPORT` 容错**：`FINAL_REPORT` 仍走 `_run_text_stage(2165)` 不变，其失败仍 `raise`（`:2272-2277` 白名单不含 FINAL_REPORT），整份报告进 FAILED 重试的口径不被本次改造改变。

> 等价性判据：改造前后，「三 FOCUS 任一阶段 LLM 失败 → 报告仍 READY、`has_stage_fallback` 点亮、质量项含 `FAILED_FALLBACK`」三条结论必须一致。`has_stage_fallback` 由 `stage_quality` 中是否存在 `FAILED_FALLBACK`/`PARSE_FALLBACK` 推导（沿用既有列表派生逻辑）。

### 5.6 提示词调整要点

三个 FOCUS 提示词函数 `_first_board_focus_prompt(2495)` / `_chain_focus_prompt(2566)` / `_high_board_focus_prompt(2592)` 的改动：

- **输出契约改写**：把结尾「输出 HTML 片段，使用 h3、p、ul、table、strong，不要输出 html/body」改为「输出严格 JSON 单一对象，禁止在 JSON 之外输出任何内容（含 Markdown 围栏与 HTML），`html_fragment` 字段内嵌 HTML 片段（使用 h3/p/ul/table/strong，不含 html/body）」，并附上 5.4 的 JSON 形状模板（镜像选股阶段提示词内联 schema 的写法，如 `_chain_selection_prompt:2539-2542`）。
- **先验字段口径**：明确 `continuation_prob` / `next_day_premium_prob` 用 `高/中/低/极低` 四档；`boost_conditions` / `fail_conditions` 给可执行的竞价/承接条件（沿用现有「竞价弱于多少放弃、合理高开区间、过高开警惕点」要求，落入这两个数组）；`suggested_hold_thesis` 一句话持有逻辑。
- **保留既有风险口径**：首板不确定性提示（`:2520-2521`）、20cm 更深断板（`:2587`）、`emotion_cycle` 约束高位接力（`:2614`）逐条保留，迁入 JSON 提示词语境。
- **system prompt 不动**：`_stage_system_prompt(2430)`（注入情绪周期定位 + 退潮/冰点不得激进接力 + 风险提示）继续复用，无需改。
- **「不能 HTML+独立 JSON 并存」显式约束**：提示词须明确「整条回复必须是且仅是一个 JSON 对象」，规避模型在 JSON 前后追加解释文字导致 `json_object` 校验失败或 `_extract_json_payload` 误截。

### 5.7 阶段缓存版本隔离（避免与旧 HTML 缓存混用）

阶段缓存命中键为 `(trade_date, stage_key, model, prompt_version, input_hash, status=READY)`（`_stage_cache_payload:2364-2376`）。`stage_key` 不变（仍是 `FIRST_BOARD_FOCUS` 等），若 `prompt_version` 也不变，则旧的「纯 HTML payload」缓存行会被新 JSON 路径命中并直接返回——但旧 payload 缺结构化先验字段，造成混用。隔离做法：在 `_STAGE_PROMPT_VERSION_SUFFIXES`（`:3075-3077`，当前仅含 `INVESTMENT_ADVICE: "advice-v1"`）显式登记三 FOCUS 阶段的新后缀：

```python
_STAGE_PROMPT_VERSION_SUFFIXES: dict[str, str] = {
    LIMIT_UP_STAGE_INVESTMENT_ADVICE: "advice-v1",
    LIMIT_UP_STAGE_FIRST_BOARD_FOCUS: "focus-json-v1",  # 新增：JSON-mode 改造，与旧 HTML 缓存物理隔离
    LIMIT_UP_STAGE_CHAIN_FOCUS: "focus-json-v1",
    LIMIT_UP_STAGE_HIGH_BOARD_FOCUS: "focus-json-v1",
}
```

`_stage_prompt_version` 拼出 `{final_prompt_version}:{stage_key.lower()}:focus-json-v1`（`:3086-3087`），与旧 `...:v3` 不同，旧缓存自然不命中、不被复用，新旧物理隔离；旧 HTML 缓存行保留不删（无副作用，存量 READY 报告不受影响）。**优先用此机制而非 bump 全局 `LIMIT_UP_PUSH_FINAL_PROMPT_VERSION`**：全局 bump（`config.py:248-251`）会让全部阶段缓存失效并整体重算，本次只改三个阶段，按阶段后缀升版更精准、影响面最小。

### 5.8 单元测试要点

沿用既有 mock 模式（`FakeTushareClient` / `FakeNotificationService` / monkeypatch `_chat_completion_with_reasoning`）。重点用例：

1. **JSON 正常解析**：mock LLM 返回合法 JSON（含 `html_fragment` + 全部先验字段）→ payload 各字段正确解析；`final_input` 的 `*_focus_html` 等于该 `html_fragment`；质量项为 `OK`。
2. **JSON 解析失败兜底**：mock LLM 返回非 JSON（如纯 HTML 或截断文本）→ `_extract_json_payload` 返回 None → 走 `_fallback_focus_stage` payload，含 `html_fragment` 非空、先验为保守缺省值、`parse_fallback=True`、质量项 `PARSE_FALLBACK`；**报告仍 READY**。
3. **LLM 调用异常兜底**：mock `_chat_completion_with_reasoning` 抛异常 → 走 `except` 分支 → `error_fallback=True`、`FAILED_FALLBACK`、`failed=True` 写缓存；**报告仍 READY**、`has_stage_fallback=true`。
4. **READY 率等价回归**：构造「三 FOCUS 之一失败」场景，断言报告 `status=READY`（与改造前 text 路径同输入同结论），防回退。
5. **`json_mode` 透传**：断言 FOCUS 阶段调用 `_chat_completion_with_reasoning` 时 `json_mode=True`，即 payload 含 `response_format={"type":"json_object"}`（可对 `_chat_completion_with_reasoning` 入参做 spy）。
6. **缓存版本隔离**：预置一条旧 `prompt_version=...:chain_focus:v3` 的 HTML-only READY 缓存行 → 新路径查询用 `...:chain_focus:focus-json-v1` → **不命中旧行**，触发新 LLM 调用；同输入二次调用命中新行（幂等不重调）。
7. **`stock_priors` 映射**：返回的 `stock_priors[].ts_code` 与 `selected_chain_stocks` 行对齐时可被下游取到；缺失/多余 `ts_code` 不报错（按缺失处理）。

### 5.9 验收标准

- 三个 FOCUS 阶段全部经 `_run_json_stage` + `json_mode=True`（`response_format=json_object`）产出单一 JSON 对象，对象内同时含结构化先验字段与 `html_fragment`，无「HTML+独立 JSON 并存」。
- 新生成报告的渲染外观与改造前一致（`FINAL_REPORT` 输入的 `*_focus_html` 仍为有效 HTML 片段）。
- READY 率不回退：三 FOCUS 任一 LLM 失败（调用异常或 JSON 解析失败）时，报告仍 READY，质量项含 `FAILED_FALLBACK` / `PARSE_FALLBACK`，`has_stage_fallback=true`，且降级 payload 含非空 `html_fragment`。
- 阶段缓存版本隔离生效：旧 HTML-only 缓存不被新 JSON 路径命中；新路径同输入幂等不重调 LLM。
- `FINAL_REPORT` 失败仍触发整份报告 FAILED 重试的口径未被改变。
- 结构化先验字段可被下游归因侧消费（字段名/档位枚举与本节 5.4 一致），对接细节见 `resources/doc/qmt-trade-review-closed-loop-attribution-design.md`。
- 全部新增/改动代码补齐中文注释（业务意图、边界、JSON 解析失败与缓存版本隔离的重跑口径）。

### 6.1 表语义与口径

- **粒度**：一股一行。`trade_date = T`（信号日，即报告对应的涨停复盘日）；`target_trade_date = T+1`（计划买入日）。`T+1` 由 `a_trade_calendar` 映射得到，不得在应用层手工 `+1 天`（避免跨周末/节假日错位），与全局 T+1 口径一致。
- **唯一性**：`(trade_date, ts_code, prompt_version)` 唯一。同一信号日同一只票在同一 `prompt_version` 下只保留一行；换提示词版本（重算）会产生新维度行，不与旧版本互相覆盖。
- **来源**：写入收口从 `context["pipeline"]` 还原（`selected_first_board_stocks` / `selected_chain_stocks` / `selected_high_board_stocks` + `stock_supplements` + 建议分层结论），与报告 READY 状态原子提交。
- **代码形态**：`ts_code` 统一 `600000.SH` / `000001.SZ` 形态（pipeline 选股行已是该形态，必要时经 `stock_identity_resolver.resolve_by_code` 兜底），供 QMT 侧 `norm_code` 直接关联。
- **时间字段**：`trade_date` / `target_trade_date` 为东八区交易日 `DATE`，本身不做时区换算；`created_at` / `updated_at` 走 `TimestampMixin`（`server_default=func.now()`），前端 `formatEast8DateTime` 展示，禁手工 ±8h。
- **范围约束**：仅主板 + 创业板，排除 ST / 科创 / 北交（继承全局口径，由上游选股阶段保证，本表不再二次过滤）。

### 6.2 完整 DDL（`resources/sql/03_full_schema_with_comments.sql` 同步版）

字段分组：①主键与关联键；②板块/连板维度；③龙头强度分及各维度分；④角色/战法/形态/动作；⑤情绪周期与可成交性；⑥先验概率；⑦晋级/失败条件与持有逻辑；⑧热字段（可成交性判定与展示快照）；⑨优先级/原始结构/入选理由；⑩审计与版本。

```sql
-- 打板信号计划落表：一股一行，trade_date=T 信号日、target_trade_date=T+1 买入日。
-- 由 limit_up_push_service._persist_selected_stocks 在报告 READY 收口处与报告状态原子提交；
-- 整组 delete-then-insert（latest-wins），写失败降级不阻断报告交付。
-- 唯一键 (trade_date, ts_code, prompt_version)；外键 source_analysis_id -> limit_up_analysis_cache.id。
CREATE TABLE `limit_up_selected_stock` (
  `id`                     BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
  -- ① 主键与关联键（与 QMT 闭环归因 join：ts_code + target_trade_date）
  `trade_date`             DATE         NOT NULL COMMENT '信号日 T（东八区交易日，对应报告复盘日）',
  `target_trade_date`      DATE         NOT NULL COMMENT '计划买入日 T+1（由 a_trade_calendar 映射，禁手工+1天）',
  `ts_code`                VARCHAR(16)  NOT NULL COMMENT '标准代码 600000.SH/000001.SZ 形态，供 norm_code 关联',
  `name`                   VARCHAR(64)      NULL COMMENT '股票名称快照',
  -- ② 板块/连板维度
  `board`                  VARCHAR(16)      NULL COMMENT '所属市场板：MAIN 主板 / GEM 创业板',
  `tier`                   VARCHAR(16)  NOT NULL COMMENT '入选分层来源：FIRST_BOARD/CHAIN/HIGH_BOARD',
```

### 6.5 写入收口 `_persist_selected_stocks(analysis, context)`

**切入点**：`ensure_analysis_for_trade_date` 的 READY 收口段（`limit_up_push_service.py:399-402`：`analysis.status=READY; generated_at; error_message=None; commit`）。落表逻辑插在 `analysis.status = ANALYSIS_STATUS_READY`（行 399）之后、`self.db.commit()`（行 402）之前——使本表插入与报告 READY 在**同一事务原子提交**：要么报告 READY 且选股行落库，要么一起回滚。

**已有 READY 早退守卫**：`ensure_analysis_for_trade_date` 在命中已有 READY 时直接 `return existing`（行 345-346），不会走到收口段。为保证"切到 ADVICE 首日/历史报告补落选股行"也能补齐本表，需在该早退分支加守卫：命中 READY 且本表无该 `(trade_date, prompt_version)` 行时，补调一次 `_persist_selected_stocks(existing, context)`（独立提交，try/except 降级，不影响早退返回）。这样既不重复对已落库报告重算，又能为存量 READY 报告回填选股行。

**幂等：整组 delete-then-insert（latest-wins）**。同一 `(trade_date, prompt_version)` 先删后插，保证"最新一次写入即全集"，避免 pipeline 入选名单收缩时残留旧行。

**写失败降级不阻断**：本表写入全程 try/except 包裹。报告本体（`limit_up_analysis_cache`）是主交付物，选股落表是附加产物——落表失败不得让报告 READY 失败或回滚。但因为要"原子提交"，实现上分两层：在收口段内只做"构造行 + add 到 session"（异常即捕获并清掉本表待写对象，不影响 analysis 字段），随后 `commit` 一并落库；若 `commit` 因本表约束失败，回退为"报告单独提交 + 选股行旁路单独事务重试一次，再失败仅记日志"，保证报告交付。

伪逻辑：

```
def _persist_selected_stocks(self, analysis, context) -> None:
    """把 pipeline 选股 + 建议分层结论结构化落入 limit_up_selected_stock。

    业务意图：信号计划结构化镜像，供 QMT 闭环归因与只读导出。
    幂等：整组 delete-then-insert（latest-wins），按 (trade_date, prompt_version) 替换。
    异常口径：构造/写入失败不抛出，降级记日志；调用方在 READY 收口处包裹，落表失败不阻断报告。
    """
    pipeline = (context or {}).get("pipeline")
    if not pipeline:
        # 旧报告（多阶段改造前）无 pipeline 结构 -> 本表本次不落，记一条降级日志即返回
        return
    trade_date = analysis.trade_date
    prompt_version = analysis.prompt_version
    schema_version = LIMIT_UP_WATCHLIST_SCHEMA_VERSION   # 见 6.6
    model = self.settings.limit_up_push_model
    # T+1：用 a_trade_calendar 映射，禁手工 +1 天；映射不到（数据缺口）则记降级、target=NULL 允许稍后回填
    target_trade_date = self._next_trade_date(trade_date)   # 复用既有交易日历映射
    advice_degraded = self._advice_is_degraded(analysis, pipeline)  # advice_status!=READY 视为降级
    market_state = self._resolve_market_state(context)     # 取自 _market_emotion 当日结论，全行同值

    # 1) 把三个 selected_* 来源 + 建议分层结论合并成"一股一结构化行"的中间结构
    rows = []
    seen = set()                       # 同票多分层时去重（保留更高分层/更强角色，口径固化）
    for tier, items in (("HIGH_BOARD", pipeline.get("selected_high_board_stocks")),
                        ("CHAIN",       pipeline.get("selected_chain_stocks")),
                        ("FIRST_BOARD", pipeline.get("selected_first_board_stocks"))):
        for item in (items or []):
            code = normalize_ts_code(item.get("ts_code"))   # stock_identity_resolver 兜底
            if not code or code in seen:
                continue
            seen.add(code)
            advice = self._advice_for_code(pipeline, code)  # 建议分层结论：action/boost/fail/hold_thesis
            rows.append(self._build_selected_row(
                item=item, advice=advice, tier=tier, trade_date=trade_date,
                target_trade_date=target_trade_date, market_state=market_state,
                source_analysis_id=analysis.id, model=model,
                prompt_version=prompt_version, schema_version=schema_version,
                advice_degraded=advice_degraded,
            ))

    # 2) 整组 delete-then-insert（latest-wins），在调用方事务内执行
    self.db.query(LimitUpSelectedStock).filter(
        LimitUpSelectedStock.trade_date == trade_date,
        LimitUpSelectedStock.prompt_version == prompt_version,
    ).delete(synchronize_session=False)
    self.db.add_all(rows)
    # 注意：本函数不 commit；commit 由 READY 收口段统一执行，保证与 status=READY 原子提交
```

收口段改造（`399-402` 段）：

```
analysis.status = ANALYSIS_STATUS_READY
analysis.generated_at = self._now_naive()
analysis.error_message = None
try:
    self._persist_selected_stocks(analysis, context)   # 入同一 session，未 commit
except Exception:
    self.db.rollback() 之外的对象状态清理；report 字段保持已设值
    logger.warning("打板选股落表构造失败，降级不阻断报告 trade_date=%s", trade_date, exc_info=True)
self.db.commit()         # 报告 READY 与选股行原子提交
self.db.refresh(analysis)
```

> 设计取舍：把 `_persist_selected_stocks` 设计为"只 add、不 commit"，使其天然进入收口事务，达成"与 READY 原子提交"。`try/except` 仅包裹"构造 + add"，捕获后清理本表待写对象再继续 `commit`，保证报告本体不被选股侧异常拖垮。早退守卫分支（命中已有 READY 补落）因为脱离收口事务，需自带 `try/except + commit/rollback`。

`_build_selected_row` 字段映射要点（业务口径）：

- `board`：由 `ts_code` 后缀 + 板段判定（`.SH/.SZ` 主板、`30` 开头创业板）。`board_level`：取 `selection.board_status` 解析数值（"2连板"→2，首板→1，无法识别→`None`）。
- `leader_strength_score` / `strength_dim_json` / `role_tags` / `strategy_family` / `setup`：来自选股阶段 `item["selection"]`（`theme_role`/`leader_role`/`selection_reason` 等）与（若有）强度子分；缺失置 `None`。
- `action` / `boost_conditions` / `fail_conditions` / `suggested_hold_thesis`：来自建议分层结论（重点/谨慎/放弃观察 + 竞价触发/失败条件 + 持有逻辑，详见 `limit-up-push-investment-advice-refactor-plan.md` 第 3.3 节）；建议降级时置 `None` 且 `advice_degraded=1`。
- `seal_ratio_pct` / `limit_order` / `turnover_rate` / `close`：来自当日选股原始行（KPL/行情快照），`winner_rate` 来自 `stock_supplements`（cyq 补数）对应股摘要。
- `continuation_prob` / `next_day_premium_prob`：若 pipeline/建议给出先验则落库；当前若无显式先验，落 `None`（闭环归因侧先验校准对 `None` 跳过该行，详见 `qmt-trade-review-closed-loop-attribution-design.md` 第 4 节）。
- `item_json`：`{"selection": item, "advice": advice}` 完整快照（去掉超大字段），作回放/审计兜底。

### 6.8 单元测试要点（`backend/tests/`）

- **幂等（同输入二次写入无重复）**：同一 `(trade_date, prompt_version)` 连续两次跑 `ensure_analysis_for_trade_date`（或直接两次 `_persist_selected_stocks`），断言本表行数不翻倍、唯一键无冲突报错、内容为最新一组。
- **整组替换（latest-wins）**：第一次 pipeline 选 8 只、第二次（同版本重算）收缩为 5 只，断言落库为 5 只且旧 3 只被删除（delete-then-insert 生效），不残留。
- **降级（建议失败/旧报告无 pipeline）**：①建议 `advice_status=FAILED` 走降级整报时，断言选股行仍落库、`advice_degraded=1`、`action`/`boost_conditions`/`fail_conditions` 为 `None`、选股侧字段（tier/board_level/leader_strength_score）完整；②`context` 无 `pipeline`（旧报告）时，断言 `_persist_selected_stocks` 不抛、本表零行、报告仍 READY。
- **原子提交**：构造一行违反 NOT NULL（如缺 `schema_version`）触发 commit 异常，断言报告本体仍按降级旁路提交为 READY（报告不被选股侧异常回滚），并记降级日志。
- **已有 READY 早退补落**：预置一条 READY 报告但本表无对应行，再次 `ensure_analysis_for_trade_date` 命中早退分支，断言补落选股行成功且早退仍返回原 analysis。
- **T+1 映射**：断言 `target_trade_date` 由 `a_trade_calendar` 得到（跨周末时不是简单 +1 天）；映射缺口时 `target_trade_date=None` 且记降级、不抛。
- **导出接口**：①缺 token / 错 token → 401，未配置 token → 503；②正确 token + 有数据 → 200 且 `schema_version`/`stocks` 字段与库内一致；③无数据 → 200 空 `stocks`；④`prompt_version` 缺省取最新 READY 报告版本。

## 七、回测引擎

> 定位：本章设计**短线打板信号回测引擎**。项目当前唯一的回测能力是分红再投长持（`dividend_reinvestment_service.py` + `dividend_reinvestment_backtest_run/summary/yearly` 三表），其口径为「长期持有 + 现金分红再投」，与打板「T→T+1 隔日兑现、不复权、一字/秒封买不进不计」完全不同，不可复用，故本章从零新建。
>
> 回测引擎是「信号→执行→结果」闭环里的**理论基准提供者**：它产出每条信号在不复权 T+1 行情上「假设按规则成交」的预期收益，供实盘复盘做「实盘 − 回测 gap」对账。与实盘复盘的对账口径**详见 `resources/doc/qmt-trade-review-board-design.md` 第 4 章**（收益率/盈亏/滑点/FIFO 公式）及 `resources/doc/qmt-trade-review-closed-loop-attribution-design.md` 第 4 章（滑点 `P_backtest` 基准）、第 6 章（`backtest_expected_ret` 物化字段）；本章只保证回测假设买入价、一字/秒封剔除、T→T+1 交易日历映射三者与那两篇**逐字一致**，避免 gap 失真。
>
> 复盘看板 / QMT 数据回流 / 闭环归因 / `qmt_*` 四表 DDL 已在上述三篇 qmt-trade-review-* 文档定稿，本章只做衔接说明 + 引用，不重复展开。

### 7.1 数据源与对照组方案抉择（先决项）

回测三类数据全部走**本地表**，不在回测期再调 Tushare（回测要可复现、可批量重跑，禁止把限频外部接口放进热路径）。

| 标的基础 | `a_stock_basic`（已存在） | `market`/`list_date`/`delist_date`/`list_status` | 主板/创业板涨跌停幅度判定、退市过滤 |

#### 7.1.1 对照组「全市场涨停池」历史数据来源（必须给结论）

回测要算「信号相对全市场涨停池的增量」（IC/RankIC/超额/分层 spread），需要历史每日全市场涨停名单。**项目无 `limit_list_d` 等本地涨停表**：`limit_up_push_service` 每天直接调 Tushare 拿涨停/连板/题材，仅把当日全部涨停（约 88 只）压缩快照落进 `limit_up_analysis_cache.context_json.limit_up_stocks`（字段见 `_compact_stock_row`，service:1679-1724：含 `ts_code`/`name`/`board_level`/`limit_type`/`theme`/`seal_ratio_pct`/`pct_chg`/`turnover_rate` 等）。两条回填路径：

**方案 (a)：新增 `limit_list_d` 持久化表 + DatasetSpec 同步回填历史。**

- 做法：在 `sync_service.DATASET_SPECS` 增 `limit_list_d`（`api_name="limit_list_d"`、`supports_date_range=True`、`split_by_trade_date=True`、`full_start_date` 设回测起点，复用 `repository.upsert_many` 幂等 + `sync_checkpoint` 断点），新建 `LimitListDaily` 模型（`ts_code`/`trade_date`/`limit`(U/D/Z)/`fd_amount`/`first_time`/`last_time`/`open_times`/`limit_times`/连板数 等官方字段），逐交易日回填历史。
- 利：字段权威完整（封单额、开板次数、连板数官方口径），覆盖全市场（不限于 88 只压缩快照），可独立服务于回测之外（实盘可成交性判定、闭环归因均可读）。
- 弊：需消耗 Tushare 历史额度逐日回填（受限频与积分约束），且 `limit_list_d` 与项目当前「不落涨停表、只快照」的架构形成新的同步职责，需新增同步监控。

**方案 (b)：从历史 `limit_up_analysis_cache.context_json.limit_up_stocks` 抽取回填。**

- 做法：写一次性回填脚本，遍历 `limit_up_analysis_cache`（`status=READY`）逐日解析 `context_json.limit_up_stocks`，落进 `limit_up_market_pool`（本章为对照组新建的轻量表，见 §7.4.3）。
- 利：零外部调用、即时可得（数据已在库），与「信号侧当日实际看到的涨停池」**完全同源**（回测对照组 = 信号当日真实可见集合，无前视偏差）。
- 弊：只覆盖有报告的交易日（节假日/未生成日缺口）、上限受 `LIMIT_UP_CONTEXT_STOCK_LIMIT` 压缩（约 88 只，可能不是当日全部涨停）、字段是压缩衍生口径而非官方原值（如封单额为快照值）。

**结论（推荐方案 b，并保留 a 作为后续增强）：**

回测对照组的核心诉求是「信号 vs 当日**信号侧真实可见**的涨停集合」的相对增量——用方案 (b) 抽取 `limit_up_stocks` 恰好满足「同源、无前视、零外部依赖、即时可复现」，且打板报告每日生成已覆盖绝大多数交易日，缺口可接受（缺口日在回测里按「无对照组」记 0 留痕，不插补）。方案 (a) 的官方完整字段对**回测对照组**不是必要项（对照组只需 `ts_code` 集合 + 涨幅，用于算超额与分层），其完整封单/连板字段更适合服务实盘可成交性判定。故：

- **本章 §7 回测引擎默认走方案 (b)**：新建 `limit_up_market_pool` 轻量对照组表 + 一次性回填脚本（从 `limit_up_analysis_cache` 抽取）。
- **方案 (a) `limit_list_d` 列为后续增强**：当需要官方完整封板字段（实盘逆向选择、秒封封单阈值判定）时再补，届时对照组可平滑切换数据源（回测引擎对照组数据源做成可配置 `control_group_source` = `CACHE_POOL`(b) / `LIMIT_LIST_D`(a)），口径切换写入文档。

### 7.2 撮合引擎硬规则（与实盘/闭环归因逐字一致）

撮合是回测的灵魂。所有规则**版本化**（`v_exec.N` 可成交性、`v_cost.N` 成本），回测 run 落库时固化版本号，保证「同一信号在同一版本下结果可复现、跨版本可对比」。

#### 7.2.1 T→T+1→T+2 交易日映射

- 信号 `trade_date=T`，买入日 `B = target_trade_date`。`B` **不直接取 `T+1` 自然日**，而是 `a_trade_calendar` 中 `cal_date>T AND is_open=1` 的最小日（即 T 的下一交易日）；回测落库时 `target_trade_date` 已由信号侧写好，回测只校验其与日历一致。
- T+1 当日买入**不可卖**，最早 T+2 卖出。卖出日 `S = a_trade_calendar` 中 `cal_date>B AND is_open=1` 的最小日（B 的下一交易日，打板隔日卖典型 = T+2）。
- 持有窗口扩展（多窗口评估，见 §7.5.1）：T+2、T+3…T+N 卖出日均按交易日历逐级 `LEAD` 取，绝不自然日加减。

#### 7.2.2 涨跌停价与板别（不复权）

- 涨停价 `limit_up_price = round(pre_close × (1+幅度), 2)`（四舍五入到分，与闭环归因 §4 `P_signal_limit` 同口径）；幅度按 `board`：主板 ±10%、创业板 ±20%；**只做主板 + 创业板，排除 ST/科创/北交**（信号侧已过滤，回测再用 `a_stock_basic.market` + 代码前缀双重校验兜底）。
- `pre_close` 取 `tencent_unadjusted_daily_quote` 当日 `pre_close`（不复权，与实盘账单口径一致）；`board` 由 `ts_code` 后缀 + `market` 判定（创业板 `300*/301*.SZ` ±20%，其余主板 ±10%）。

#### 7.2.3 可成交性硬规则（`v_exec`，决定是否计入分子）

买入日 `B` 行情形态判定「买不进」并**剔除（不计收益，但计入分母留痕，见 §7.5.4）**：

1. **一字板**：`limit_type` 标记为「一字」**或** `B 日 high == low == limit_up_price`（全天封死涨停，买不进）。
2. **秒封/全封**：`B 日 open == high == low == limit_up_price`（开盘即一字，等价一字，买不进）。
3. **无量跌停顺延（卖出侧）**：卖出日 `S` 若 `high == low == 跌停价`（跌停封死无法卖出），卖出**顺延**到下一交易日，直至可卖或触及窗口上限；顺延次数与最终卖出日落库留痕。

> 这三条与闭环归因 `qmt-trade-review-closed-loop-attribution-design.md` §2.1（一字/秒封 miss_reason）、§5（一字/秒封反事实不计收益）、滑点 §4（`P_backtest`「一字/秒封不计、否则按 B 日开盘或合理价成交」）**逐字对齐**，对账才不失真。

#### 7.2.4 成交价口径（`v_cost`）

买入价 `P_buy`（剔除一字/秒封后）按版本：

- `v_cost.0`：B 日 `open`（隔日开盘买，最贴近打板隔日开盘出/入习惯，**默认**）。
- `v_cost.1`：B 日合理高开区间价（`reasonable_open_high_low/high` 的中值，若信号侧提供）。
- `v_cost.2`：B 日 `vwap = amount/volume`（成交均价，保守口径）。

卖出价 `P_sell`：默认卖出日 `open`（与闭环归因 §3.1 `P_sell` 默认隔日 `open` 一致），可配 `close`/vwap。

可成交性版本：

- `v_exec.0`：仅剔一字（规则 1）。
- `v_exec.1`：剔一字 + 秒封（规则 1+2，**默认**，与实盘/归因一致）。
- `v_exec.2`：在 v_exec.1 基础上加封单额/换手阈值过滤（需方案 a 的 `limit_list_d` 封单字段，列为增强）。

成本扣减（`v_cost` 同时承载费用口径）：佣金 + 印花税（卖出单边）+ 过户费，按可配费率；回测默认口径与实盘对账时**显式标注是否含费**，避免「回测净收益 vs 实盘毛收益」错配。

#### 7.2.5 撮合伪逻辑

```
function backtest_signal(signal, exec_ver, cost_ver, hold_windows):
    # 1) 交易日映射（全程 a_trade_calendar，禁自然日）
    B = signal.target_trade_date                      # 买入日 = T 的下一交易日
    qB = quote(signal.ts_code, B)                     # 不复权 B 日行情
    if qB is None: return mark(signal, status="NO_QUOTE")   # 停牌/缺行情 → 空仓留痕

    # 2) 涨停价与板别（不复权）
    board = board_of(signal.ts_code, qB.market)        # 主板±10% / 创业板±20%
    limit_up = round(qB.pre_close * (1 + rate(board)), 2)

    # 3) 可成交性硬规则（v_exec）
    if is_one_word(signal.limit_type, qB, limit_up):           # 一字：high==low==limit
        return mark(signal, tradable=False, miss="ONE_WORD")  # 不计收益，计入分母
    if exec_ver >= 1 and is_seconds_seal(qB, limit_up):       # 秒封：open==high==low==limit
        return mark(signal, tradable=False, miss="SECONDS_SEAL")

    # 4) 买入价（v_cost）
    P_buy = pick_buy_price(qB, cost_ver, signal)       # open / 合理价 / vwap

    # 5) 多窗口卖出（T+1 当日不可卖，最早 T+2；无量跌停顺延）
    results = {}
    for w in hold_windows:                             # 例 [1,2,3] 个交易日
        S = next_n_open_days(B, w)                     # B 起第 w 个交易日（LEAD over calendar）
        qS = quote_with_limit_down_rollover(signal.ts_code, S, board)  # 无量跌停顺延
        if qS is None: results[w] = mark_missing()      # 行情缺口留痕
        else:
            P_sell = pick_sell_price(qS, cost_ver)
            gross = (P_sell - P_buy) / P_buy
            net   = gross - cost_of(cost_ver, P_buy, P_sell)   # 含/不含费按版本
            results[w] = {gross, net, sell_date:S, rollover_days}
    return mark(signal, tradable=True, P_buy, results)
```

### 7.3 类设计（可插拔 EntryRule / ExitRule）

服务文件 `backend/app/services/limit_up_backtest_service.py`。核心拆分为「数据装载 → 撮合 → 评估 → 落库」四段，撮合的进出场规则做成**策略对象**便于扩展不同打板战法。

```
# backend/app/services/limit_up_backtest_service.py

@dataclass
class BacktestConfig:
    """回测口径配置（落 limit_up_backtest_run.params_json，保证可复现）。"""
    start_date: date; end_date: date
    exec_version: str = "v_exec.1"          # 可成交性版本
    cost_version: str = "v_cost.0"          # 买入价 + 费用版本
    hold_windows: tuple[int, ...] = (1, 2, 3)   # 卖出窗口（交易日数）
    sell_price_policy: str = "NEXT_OPEN"
    include_fees: bool = False
    control_group_source: str = "CACHE_POOL"    # 对照组数据源：CACHE_POOL(b)/LIMIT_LIST_D(a)

class EntryRule(Protocol):
    """进场规则：判定信号在买入日是否可成交、给出买入价。"""
    def is_tradable(self, signal, quote_B, limit_up_price) -> tuple[bool, str | None]: ...
    def buy_price(self, signal, quote_B, cfg: BacktestConfig) -> Decimal: ...

class ExitRule(Protocol):
    """出场规则：给定买入日与持有窗口，按交易日历+跌停顺延确定卖出日与卖出价。"""
    def resolve_sell(self, ts_code, buy_date, window, board, cfg) -> SellLeg: ...

class DefaultLimitUpEntryRule(EntryRule):
    """默认打板进场：一字/秒封剔除（v_exec），开盘价/合理价/vwap 买入（v_cost）。"""

class HoldWindowExitRule(ExitRule):
    """默认隔日出场：T+2 起按窗口卖，无量跌停顺延，卖出价按 sell_price_policy。"""

class LimitUpBacktestService:
    def __init__(self, db, calendar_repo, quote_repo, entry: EntryRule, exit: ExitRule): ...
    def run(self, cfg: BacktestConfig) -> int:
        """编排一次回测：建 run → 装载信号/行情/对照组 → 逐信号撮合 →
        写明细 limit_up_backtest_result → 算评估指标 → 标记 run 完成。返回 run_id。
        幂等：同 run_key(=口径哈希+区间) 重跑先清同 run_id 明细再写，避免重复行。"""
    def _load_signals(self, cfg) -> list[Signal]: ...     # 读 limit_up_selected_stock
    def _load_control_pool(self, cfg) -> dict[date, set]: ... # 读对照组（方案 b 默认）
    def _match_one(self, signal, cfg) -> ResultRow: ...   # 调 EntryRule/ExitRule
    def _evaluate(self, run_id, rows, control, cfg): ...  # IC/超额/分层/校准
```

可插拔点：换战法 = 实现新的 `EntryRule`/`ExitRule` 注入 `LimitUpBacktestService`，不改撮合主流程；可成交性/成本版本由 `BacktestConfig` 切换。

### 7.5 评估指标清单

#### 7.5.1 信号收益分布（多持有窗口）

按 `hold_window ∈ {1,2,3,…}` 分别统计可成交信号的 `net_return_pct`：均值/中位/标准差、胜率（`>0` 占比）、盈亏比、分位（P10/P25/P75/P90）、最大回撤型单笔最差。多窗口对照看「隔日卖 vs 持有 2~3 日」的收益衰减/增益曲线。

#### 7.5.2 相对对照组增量（信号 vs 全市场涨停池）

- **超额收益**：`avg(信号 net_return) − avg(对照组当日等权 next_day_return)`，按窗口与情绪周期分组。
- **IC / RankIC**：以 `leader_strength_score`（或 `continuation_prob`）为预测分、`next_return` 为实际，按交易日截面算 Pearson IC 与 Spearman RankIC，再取时序均值 + ICIR（IC 均值/IC 标准差），评信号区分度。
- **分层 spread 显著性**：按预测分分 N 层，Top 层 − Bottom 层收益差（spread），对其时序序列做 t 检验（给 t 值/p 值），判断分层是否显著单调。

#### 7.5.3 分组区分度（多维交叉，先可成交过滤再分桶）

**关键顺序：先按 `tradable_flag=1` 过滤掉一字/秒封，再分桶**（否则买不进的最强票污染分组收益，与闭环归因逆向选择口径一致）。分组维度交叉：`market_state`(情绪周期) × `leader_strength_pctl`(龙头分位桶) × `role`(角色) × `strategy`(战法)。每格输出：样本数、可成交率、平均净收益、胜率、盈亏比、相对对照组超额；高亮显著正/负偏离格。

#### 7.5.4 空仓日记 0 留痕 + 反事实

- **空仓日计入分母**：`market_state=空仓` 或当日无可成交信号的交易日，组合当日收益记 0（不是从分母剔除），落 `is_empty_day=1`，保证「年化/胜率/夏普」分母含空仓日，不虚高。
- **反事实**：空仓日仍取当日**本应入选**候选（或退化为当日对照组涨停池），按盯市口径算「若买了的反事实收益」（一字/秒封不计，与闭环归因 §5 一致），落 `limit_up_backtest_result` 标 `is_counterfactual`，供「空仓闸门是否救了我们」回看。反事实与真实回测收益**物理分区**，禁混入主收益汇总。

### 7.6 单元测试要点

| 用例 | 断言 |
| --- | --- |
| **日期映射** | T=周五 → B 跳过周末取下周一（`a_trade_calendar.is_open`）；T+1 当日不可卖，S 最早 T+2；构造含节假日的日历，T→B→S 全部命中交易日，绝不出现自然日 `+1` 误差 |
| **一字剔除** | `limit_type='一字'` 或 `high==low==limit_up_price` → `tradable_flag=0`、`miss_reason='ONE_WORD'`、`net_return_pct IS NULL`；`open==high==low==limit_up_price`（v_exec.1）→ `SECONDS_SEAL` |
| **秒封边界** | v_exec.0 下秒封**不**剔（仅剔一字）；v_exec.1 下秒封剔——同一信号两版本结果不同，验证版本化生效 |
| **无量跌停顺延** | 卖出日跌停封死（`high==low==跌停价`）→ 卖出顺延到下一可卖交易日，`limit_down_rollover_days` 正确累加 |
| **空仓日计入分母** | 含 K 个空仓日的区间，年化/胜率/夏普分母 = 交易日总数（含空仓日），空仓日收益贡献 0；移除空仓日后分母变小 → 断言两者不等，证明未被剔除 |
| **涨停价板别** | 创业板 `300*.SZ` 用 ±20%、主板 ±10%，`round` 到分；不复权用当日 `pre_close` |
| **幂等重跑** | 同 `run_key` 重跑：先清同 `run_id` 明细再写，结果行数不翻倍 |
| **对账口径** | 回测 `buy_price`（v_cost.0=B 日 open）与闭环归因 `P_backtest` 口径一致，可对账（`slippage_vs_backtest_bps` 可算出） |

#### 8.1.1 回测对照组历史数据来源（二选一，给出结论）

现状关键约束：涨停/连板（`limit_list_d`）等表**当前无本地物理表**，`limit_up_push_service` 每日直接调 Tushare 取数，仅在每份报告的 `limit_up_analysis_cache.context_json.limit_up_stocks`（当日全部涨停约 88 只）留快照。回测「全市场涨停池对照组」缺历史本地表，须二选一：

- **方案一（推荐，作为目标态）**：新增 `limit_list_d` 物理表 + Tushare 同步 DatasetSpec，按 `trade_date` 回填历史。优点：数据完整、字段规范（连板数、封单额、首次封板时间等可得）、与未来全市场策略可复用；缺点：需消耗 Tushare 额度回填且接口字段须核实。
- **方案二（过渡，零外部依赖）**：从历史 `limit_up_analysis_cache.context_json.limit_up_stocks` 抽取回填对照组。优点：无需额外同步、即取即用、覆盖已有报告区间；缺点：仅覆盖已生成报告的交易日、字段以快照口径为准（可能缺连板细分字段）、JSON 抽取需做归一与去重。

**结论**：采用**方案一为最终态**、**方案二为冷启动过渡**——先用方案二打通回测引擎与 go/no-go 验证（无需等额度回填），并行落 `limit_list_d` 同步；待 `limit_list_d` 历史回填校验通过后，回测对照组来源切换为 `limit_list_d`，`limit_up_backtest_run` 以「对照组来源」字段记录本批用的是哪一种，保证结果可追溯。

## 附：对抗核验修正（覆盖前文冲突项，以此为准）
