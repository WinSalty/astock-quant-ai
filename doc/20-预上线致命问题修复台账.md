## 三、逐问题修复台账

### C-2 — 竞价段实时封板续持（`position/sell_decider.py`）
- **分析**：`decide_auction` 各分支从不引用 `book.is_sealed`；竞价正封涨停（`is_sealed=True`、`open_pct>0`、未背离）
  的**弱先验**隔夜票会走兜底 `if not prior_strong → _reduce_or_clear(CLEAR)`，以现价≈涨停价挂卖砸开自家封板 /
  在涨停高点放弃强势仓，与分时 `decide_intraday` 的「秒板稳封→HOLD」自相矛盾。
- **修复**：在兜底（及背离/弱开判定）之前插入 `if book.is_sealed and self._is_seal_strong(book): return HOLD("竞价封板续持")`，
  与分时同口径（封流比未知/弱则不进续持分支、按原兜底处理，不过度保护）。
- **测试**：新增 4 例（封板+弱先验→HOLD、封板+无先验→HOLD、未封板弱开仍 CLEAR 防过度保护、封流比未知不续持）。
- **review 结论**：correct，分支位置与一致性正确，无回归。

### C-3 — 跨帧盘口字段接线扩展点（`position/sell_book_builder.py`、`app/main.py`）
- **分析**：`broke_board/below_support/volume_surge/near_close_weak/price_volume_diverge`/`open_times` 恒默认
  → 炸板/破位/烂板/尾盘了结四类硬扳机永不触发。**跨帧数值保真依赖真机 tick 帧历史（外部依赖）**，但**接线框架可离线落地**。
- **修复（离线部分）**：`build_order_book(..., cross_frame_builder=None)` 扩展点——为 None（T0.1）保守默认（值不变），
  注入时原地填充六个跨帧字段、异常降级为默认 + `CROSS_FRAME_ERR` 留痕；Engine 增 `_sell_cross_frame_builder`
  + `set_sell_cross_frame_builder` 注入点，`build_sell_books` 透传。**不改 decider 与下游**。
- **跳过并登记（真机）**：跨帧字段的真机计算器（固化 tick 键名/量纲 + 逐 code 帧历史）属 T1.2，已登记待办 §0/§A。
- **测试**：新增 4 例（无 builder 全默认且无 ERR、注入填充生效、注入抛错降级+留痕、端到端 `set_sell_cross_frame_builder` 生效）。
- **review 结论**：correct，默认行为未变、向后兼容、异常降级安全。

### H-1 — 乱序回报不回退已成交态（`order/local_ledger.py`）
- **分析**：`sync_status` 对非 CANCELLED/REJECTED 态无条件改写；成交回报先到置 TRADED 后，迟到的
  `on_stock_order(部成/已报)` 会把已全成单降级 → `on_ttl_expired` 误判在途、对已全成委托反复发撤单 + 对账态污染。
- **修复**：`sync_status` 加守卫——`e.state == TRADED` 时只记 `error_msg`、不改状态。
  **关键**：守卫用「已 TRADED 不回退」语义，**不**用 `OrderState.terminal()`（含 `PART_TRADED`，会放行迟到部成降级、等于没修）。
- **review 驱动必修**：发现**对称漏洞**——启动重建 `reconcile_fills_from_detail` 在崩溃窗口明细不全（`total_vol<plan_volume`）
  时仍会把已 TRADED 降级 PART_TRADED。补同口径守卫：`e.state == TRADED` 直接跳过重算、保留全成快照
  → 台账三处状态收口（`add_fill`/`sync_status`/`reconcile`）「已成不回退」口径全对齐。
- **测试**：乱序不降级（REPORTED/PART_TRADED）、TRADED 仍记 msg、未到 TRADED 正常推进、reconcile 已成不被不全明细下修 + 仍能向上收口。

### H-2 — 重连互斥防多 trader 泄漏（`connection/connection_guard.py`、`adapters/xt_real.py`）
- **分析**：`_lock` 只护短临界区，`trader_factory/start/connect/subscribe` 全在锁外；断线回调线程 / 主循环 supervise /
  调度 PREWARM 三路并发各造一个 trader、各 `start()` 起后台线程 → 句柄漂移 + 旧 trader 后台线程无人 stop 泄漏。
- **修复**：新增 `_conn_lock`（RLock）串行整个建连/重连全序（含 I/O）；`connect_and_subscribe`/`reconnect`
  非阻塞 try-acquire——抢不到即跳过返回 False（不并发再建）；`reconnect→connect_and_subscribe` 靠 RLock 同线程重入放行；
  替换前 `_stop_trader_quietly` best-effort 停旧 trader（`RealXtTrader.stop()` 转发，异常吞错）。`_conn_lock` 与 `_lock`
  解耦、I/O 不在 `_lock` 内执行 → 不与状态快照死锁。scheduler PREWARM 加 `guard.ready` 预检：已就绪不重建。
- **测试**：并发重连只一路真正建连（另一路跳过、不再造 trader）、重连替换前停旧 trader、PREWARM 已就绪不重建。
- **review 结论**：correct，RLock 重入与 try/finally 覆盖所有返回路径正确，无问题。

### H-3 — 风控两闸装配期 fail-closed（`config/settings.py`、`app/main.py`）

  对 None 整段放行（fail-open），且无启动期提醒，运维易误以为护栏生效。
- **修复**：`assert_safe_to_trade` 增两道与竞价择时/卖出链同级的 fail-closed 强校验——二者为 None 即拒启动；
  显式关停应配实际永不触发的大值（如 0.99，不用 0——0 是「零容忍」最严非关停），属有痕可审计的主动决策。

  补——guard 已配但 `ref` 缺失/非正 → 拒发（与账户回撤闸「配了闸却算不出观测值→禁开」`_open_blocked_by_risk` 同取向）。
- **review 误报澄清**：复核曾指「`risk._breached` 对 observed=None 容错放行与 fail-closed 矛盾」「日初权益基线失败仍放行」——

### H-4 — 调度相位可恢复（`app/scheduler.py`）
- **分析**：① PREWARM 过 9:15 即强制定稿（不论连接/装载是否成功）→ 基于未就绪 trader 重建持仓/抓权益基线、当日永不重做；
  ② RUN_AUCTION 先标 fired 再阻塞，入口/早期抛异常（被 run 顶层吞）即当日竞价整段放弃不重试。
- **修复**：① PREWARM 连接未就绪【不论时间】都不定稿、持续重试到连上（连接恢复后 `engine.prewarm` 重跑幂等补齐、
  `_capture_day_open_equity` 查询失败不缓存坏基线、下轮自愈）；watchlist 真失败仍仅开窗前重试；decide 仅 < 收盘才返回
  PREWARM → 重试天然以收盘为界、不会无限。② RUN_AUCTION 改成功跑完才标 fired，异常则下轮（仍 <9:30）decide 重新返回重试。
- **测试**：未就绪过 9:15 不定稿+连接恢复后定稿、RUN_AUCTION 异常不标 fired 且可重试、瞬时异常后重试成功才定稿、
  重试天然在收盘转 CLOSE_BATCH（非无限）、PREWARM 已就绪不重建。
- **review 结论**：correct（`guard.ready` 无锁读的竞态为设计接受、下轮自愈，属 low）。

### M-1 — 持久化原子性（`storage/sqlite_ledger.py`）
- **分析**：`add_fill` 先整行镜像（含累计 `filled_volume`）再单独 INSERT 明细、`purge_before` 删明细+删台账分两次
  独立 `submit`；叠加「重启以明细为权威重算」→ 明细丢失时把正确快照纠成偏小值 / 删一半留孤儿明细 + `load_from_db`
  全表读回致无界增长。
- **修复**：`write_queue` 每任务一次 `commit`（原子前提）→ `add_fill` 把「整行镜像 + 明细 INSERT」合并为单写任务（同事务，
  同成同败，丢失则快照回退与明细自洽、重算不再被纠错）；`purge_before` 双 DELETE 合并为单任务（绝不留孤儿）；
  `load_from_db` 窗口装载时明细按「台账行在窗口内」收敛（防孤儿无界读回）。
- **测试**：add_fill 单任务（非两次 submit）、去重命中仅镜像单任务、明细 INSERT 失败整任务回滚保持快照自洽、
  purge 单任务、窗口装载跳过孤儿明细。
- **review 结论**：correct，闭包捕获/去重分支/异常回滚/窗口收敛均正确，测试可在还原时失败。

### M-2 — 持仓查询失败单票上限 fail-closed（`order/order_executor.py`、`app/main.py`）
- **分析**：`_held_value` 查询失败保守返 0 → 被单票上限净额校验当成「该票无持仓」→ 跨日复买同一连板龙头漏计隔夜持仓、
  单票敞口可累计逼近 2×cap（查询失败 + 同票跨日复买 + 配了单票上限 三者同时）。
- **修复**：`_held_value` 改回传 `(市值, held_known)`；新增 `exposure_for_code_checked` 透传 `held_known`；
  两个消费点（`order_executor._plan_volume` 离线兜底 + `main._strength_budget_volume` 强度加权主链）在
  `not held_known` 时 fail-closed 拒单（返 0 + 强告警），绝不把查询失败当无持仓。旧 `exposure_for_code` 保留向后兼容
  （供 F13 单测/best-effort 读数），无第三消费点漏网。
- **测试**：离线兜底路径查询失败拒单 + 强告警、查询成功无持仓正常下单、未配上限不受影响、`exposure_for_code_checked`
  标志正确、**强度加权主链路径查询失败 fail-closed**（补齐第二消费点覆盖）。
- **review 结论**：correct，两消费点口径一致、与 F13 单计不重复口径自洽。
