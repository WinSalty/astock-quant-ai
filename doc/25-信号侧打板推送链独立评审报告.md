## 确认问题 · P1（污染执行侧据以下单的 watchlist / 闸门）

### S-1 [幂等] 缺省 `prompt_version` 回退取选股表 `desc(id)` 不校验报告 READY，同日换 pv 且新批零候选/未落表时返回旧 pv 陈旧候选

- 位置：`app/api/routes_watchlist_export.py:87-97`（缺省 pv 查询）；联动 `app/services/limit_up_push_service.py:535-541`（删旧仅按 `trade_date+prompt_version`）、`:671-674`（空候选短路只删新 pv、不插行）、`:379`（pv 取自可变配置）；约束 `app/db/models/notification.py:245-257`（唯一键含 prompt_version，允许两 pv 共存）。
- 触发：同一 `trade_date` 先后用不同 `prompt_version` 生成（`limit_up_push_prompt_version` 在两次同日生成间被改动），且新 pv 这批 `tiered` 为空（空仓/无涨停）或落表前失败。
- 影响：换 pv 后旧 pv 选股行不会被清；缺省 pv 查询取 `desc(id)` 命中旧 pv 行 → watchlist 全集返回旧 pv 的陈旧可交易标的，被执行侧当作当日清单下单。代码注释 `:90` 自认 force 重生成致「报告 pv 与选股行 pv 不一致」是预期场景。

### S-2 [取数口径] 本地日线缺信号日 T 行时，`_calculate_indicator` 静默用 T-1（或更早）日的 close/vol/return 冒充信号日因子落表

- 位置：`app/services/limit_up_push_service.py:1909-1918`（兜底 `if not daily_rows` 只看空非空）、`:1947-1955`（区间取数，非 T 精确匹配）、`:2061-2093`（`latest=rows[-1]`，无 trade_date 校验）、`:2402/2414`（透传 watchlist 契约）。
- 触发：某入选股本地 `a_daily_quote` 已同步到 T-1 但信号日 T 行未入库（同步延迟/新股缺口），且窗口内存在更早日线行。
- 影响：本地有 T-1 行时兜底不触发、T 行不补；`latest=rows[-1]` 为 T-1 行，使 `close / vol(=first_board_vol) / pct_chg / return_5d_pct / return_10d_pct` 全部回退 T-1，原样写入 `limit_up_selected_stock` 并经 watchlist 透传，污染执行侧据 close/量能比/空间高度的竞价与高位规避判断，无任何缺测标记。对照 `:1988-1992`：`daily_basic` 路径用 `trade_date == trade_date` 精确匹配——daily 行情路径独缺此守卫。

### S-3 [异常] 炸板池接口失败（FAILED）与真实零炸板被混为一谈，退潮/分歧/炸板硬否决全部失效

- 位置：`app/services/limit_up_push_service.py:1770-1777`（炸板池 `required=False`）、`:1822-1826`（FAILED 返回 `[]`）、`:2585-2595`（炸板率算成 0.0）；传导 `app/services/sentiment_gate.py:333`（直透无 FAILED 感知）、`:277-278`、`:209-213`（闸门把 0.0 当已知健康值）。
- 触发：信号日炸板 tag 的 `kpl_list` 查询失败/限频/权限波动。
- 影响：`broken_rows=[]` 使 `broken_board_rate_pct` 被算成 0.0（分母仍有 `today_codes`，可计算且非 None）；`resolve_gate` 视为零炸板健康市，硬否决 `broken_board_rate_high` 不触发、退潮/分歧分支永不命中，把「炸板数据缺失（未知）」静默当「市场无炸板」，可在真实高炸板退潮日放行开新仓。与项目自有纪律（评审 F1 已对 `top_list` 区分 FAILED/EMPTY）自相矛盾，炸板池未做同等处理。

### S-4 [空值] FOCUS 先验 `continuation_prob`/`next_day_premium_prob` 无枚举/范围校验，LLM 返回非法档位时静默落 NULL

- 位置：`app/services/limit_up_push_service.py:497-503`（`_prob_band_to_decimal` 末行 `table.get(...)` 静默返回 None）、`:791-796`（落表调用点）、`:702-714`（priors 索引循环仅对 ts_code 越界/重复留痕）、`:2926-2944`（stage_quality 仅判可解析即 OK）。
- 触发：FOCUS 阶段 LLM 在 `stock_priors[].continuation_prob/next_day_premium_prob` 返回不在 `{高,中,低,极低}` 内的值（如 high / 高位 / 0.7 / 带空格变体）。
- 影响：该票概率列直接落 NULL，无 warning/质量项留痕，`stage_quality` 仍 OK。执行侧无法区分「先验缺测」与「先验真为低」，复盘无从发现。与 SELECTION 阶段 ts_code 严格映射、FOCUS priors 越界丢弃都有日志的处理口径不一致——唯独档位值非法这条静默吞掉。

### S-5 [取数口径] FOCUS 降级 payload 的保守先验「极低」对落表列是死值，PARSE/FAILED 兜底时概率列实际落 NULL 而非保守值

- 位置：`app/services/limit_up_push_service.py:3104-3117`（兜底 payload 顶层放「极低」、`stock_priors=[]`）、`:2898-2900`（装配仅取 `stock_priors`）、`:696-714`（priors 仅来自 `*_focus_priors`）、`:737/791-796`（空 prior 传 None）、`:497-498`（`_prob_band_to_decimal(None)→None`）。
- 触发：三个 FOCUS 阶段任一发生 PARSE_FALLBACK 或 FAILED_FALLBACK。
- 影响：落表收口只从 `stock_priors` 按 ts_code 取每股 prior，从不读 payload 顶层概率。FOCUS 兜底时 `stock_priors` 为空 → 该 tier 全部票概率列落 NULL，而非注释承诺的「极低」保守值；NULL 行与有真实先验时落具体小数的行混在同一表，执行侧无法识别这批是降级缺测。

### S-6 [字段契约] 总龙头降级后 `role_confidence` 与 `role_tags` 不一致，污染执行侧按角色置信度的归因/加权

- 位置：`app/services/limit_up_leader_scoring_service.py:704-714`（降级只原地改 `role_tags`）、`:599-600`（`role_confidence` 早被冻结）；透传 `app/services/limit_up_push_service.py:758,783`。
- 触发：同一交易日并列最高板 ≥2 只被判 `MAIN_LEADER`（高位连板分歧日常见），非强度最高者被降级时。
- 影响：降级分支只改 `results[i].role_tags`（首位换 `SECTOR_LEADER`），但 `strength_dim_json['role_confidence']` 仍是 `MAIN_LEADER` 时算出的 0.9/0.6 未回写。两字段独立透传 watchlist 后，消费方拿到错配的（角色, 置信）二元组。

### S-7 [边界] 对照组均值与信号收益样本不等价，go/no-go 超额闸门在「仅有对照池的日子」上判定

- 位置：`app/services/limit_up_backtest_service.py:687-690`（缺对照日则 `control_excess_pct=None`）、`:743-746`（超额均值仅对非 None 行求均）、`:883`（闸门读该子集均值）；根因 `_load_signals(583-612)` 与 `_control_mean_by_date(442-461)` 来自两条独立管线、无同日齐备约束。
- 触发：某信号日有 `LimitUpSelectedStock` 行却无匹配 source 的对照池数据。
- 影响：该日全部信号被排除出 `control_excess_mean`，而 hit_rate/net_mean/cumulative 对全部信号取数；闸门 `control_excess_mean > _GONOGO_MIN_EXCESS` 的判定样本是收益样本的真子集，可在「仅部分日子」的超额上放行实盘。

## 确认问题 · P2

### S-8 [字段契约] 响应体 `schema_version` 用常量、单股 `item.schema_version` 取落库值，导出历史旧行时版本号分叉

- 位置：`app/api/routes_watchlist_export.py:146`（envelope 未传 schema_version 走默认）；`app/schemas/limit_up_watchlist.py:115,124`；`app/services/limit_up_push_service.py:834`（写入时点常量落库）。
- 影响：信封版本号恒为当前常量（1.2.0），item 经 `model_validate` 透传旧落库值；导出按旧契约落库的历史行时信封（新）与 item（旧）版本号不一致，消费方若按信封版本号决定解析规则可能用新规则解析旧结构。

### S-9 [取数口径] `_window_return` 按「非空 close 计数」而非「交易日数」回看，日线有缺口时 `return_5d/10d` 跨越多于 N 个交易日

- 位置：`app/services/limit_up_push_service.py:4701-4705`、`:2062-2066`（closes 上游已剔 None）、`:2086-2087`（调用）；落表 `:828-829` → `app/db/models/notification.py:332-333`；契约语义 `app/schemas/limit_up_watchlist.py:105-109`。
- 影响：停牌/缺口使有数据样本少于交易日数，`return_5d_pct/return_10d_pct`（空间高度/位置）被高估，无缺口标记；执行侧据此做高位规避会偏差。

### S-10 [边界] `_board_level` 未识别（返回 0）的票被三层分桶全部排除，N 天 M 板/异常 status 票静默漏出选股池

- 位置：`app/services/limit_up_push_service.py:2294-2298`（无法解析返回 0）、`:2261-2264`（`level<=0 continue`）、`:2216-2233`（三层池均经 `_stocks_by_board_level`）、`:3606-3639`（候选全集=过滤后池，无旁路）、`:2528`（仅聚合 `unrecognized_board_count`）。
- 影响：涨停行 status/tag 不匹配任一正则（KPL 文案变体/字段缺失）→ 该票不进 first/chain/high 任一源池、永不入选 watchlist，且无逐票留痕。上游 KPL 文案改版可成批涨停股静默缺席选股，执行侧整类信号丢失而无告警。

### S-11 [空值] 空仓闸门否决判定中 `highest_chain` 为 None 时被 `_num` 归零，`ice_dominant` 的 `hc<=2` 恒成立，冰点/退潮细分偏向冰点

- 位置：`app/services/sentiment_gate.py:158-159,267,298-305`；根因 `app/services/limit_up_push_service.py:2509,2527`（`valid_levels` 为空时 `highest_chain=None`）。
- 影响：`hc=0.0` 使 `hc<=2` 恒真；该枯竭日若实际仍有高板但 `highest_chain` 因上游缺失为 None，会被误判冰点而非退潮。冰点/退潮均映射空仓、顶层 action 一致，故对开仓无差异，但落 `sentiment_cycle` 列口径失真（执行侧/复盘据此区分周期）。

### S-12 [取数口径] role 角色判定用无 cyq 的 compact 强度分位、落表总分用 cyq 注入后强度，筹码维度口径分叉

- 位置：`app/services/limit_up_leader_scoring_service.py:665-699`（pctl 无 cyq）配合 `:596/:599`（score 含 cyq、role 吃无 cyq 的 pctl）、`:527-532`（pctl 分档）、`:363-370`（无 cyq 走中性 0.5）。
- 影响：落 pctl 兜底分支（MID_ARMY/STRAGGLER）的票，role 与 `leader_strength_score` 基于两套不同 position 取值，可现「总分因高位套牢被压低、角色仍判中军靠前」或反向矛盾；执行侧同时消费 role 与 score 会得到不一致的强弱信号。

### S-13 [收益统计] go/no-go t 检验对含零交易日的组合日均值统一扣往返成本，对未成交日重复扣费

- 位置：`app/services/limit_up_backtest_service.py:872-877`（净日均值扣费）、`:779-781`、`:788`（含零交易日的 `mean_daily` 构造）、`:814`（`_GONOGO_ROUND_TRIP_FEE`）。
- 影响：`net_day_mean = mean_daily − 0.0026`，而 `mean_daily` 含空仓/全买不进日（贡献 0.0）。零交易日无买卖却被扣往返费，系统性低估净日均值与 t 统计量；方向偏保守但 go/no-go 阈值校准失真。

### S-14 [取数口径] 对照组隔日收益不做停牌/封死顺延，与信号侧 seal-rollover 口径不一致，超额比较失真

- 位置：`app/services/limit_up_backtest_service.py:525-546`（对照侧无顺延/min(0)）、`:378-391`（信号侧顺延）、`:409`（信号侧卖不出 min(0)）、`:690/883`（超额进闸门）。
- 影响：对照池对封死/停牌票仍按当日 open/close 原始价（可能无法成交）记成交，无顺延、无 UNSOLD 保守处理；同一封死场景信号侧记 min(0)/顺延价、对照侧记原始价，`control_excess_pct` 的两侧成交可行性口径不对等，超额闸门判定偏差。文档声称「与信号回测同口径」与实现不符。

### S-15 / S-16 [异常] `X-Internal-Token` 含非 ASCII 字节时 `hmac.compare_digest` 抛 TypeError → 500 而非约定 401（watchlist 导出 / qmt 回流）

- 位置：`app/api/routes_watchlist_export.py:54`、`app/api/routes_qmt_ingest.py:50`（函数体无 try/except，全 app 无 exception_handler，main 仅 CORSMiddleware）。
- 触发：调用方在 `X-Internal-Token` 头传入含 `0x80-0xFF` 字节的值（Starlette 按 latin-1 解码为非 ASCII str）。
- 影响：`hmac.compare_digest` 对非 ASCII str 抛 TypeError，未捕获 → 500。已用 venv 端到端复现（裸 `\xff\x80` 字节 → 500，ascii 错 token → 401）。500/401 可区分构成 token 字符集探测旁路，违背 docstring 声明的「缺失/不符一律 401（常量时间比较防时序侧信道）」。

### S-17 [取数口径] 题材内最高板归一含自身，`SECTOR_LEADER` 对「独苗题材」（题材内仅此一只连板）恒成立

- 位置：`app/services/limit_up_leader_scoring_service.py:257-259`（自身 bl 也登记进 `theme_max_board`）、`:54-59`（`_theme_top_for_row` 取 max 含自身）、`:508,516-518`（`SECTOR_LEADER` 判定 `themes and bl >= theme_top and bl >= 2`）。
- 触发：入选票所属题材在全涨停池中仅有该票一只（小众/独苗题材）且 `bl>=2`。
- 影响：`theme_top >= 自身 bl` 恒成立，独苗题材时 `theme_top == bl`，使 `bl >= theme_top` 恒为真，任一 `bl>=2` 的独苗题材票被标 `SECTOR_LEADER`；`role_hint` 含「龙头/前排/高辨识度」时置信进一步给到 0.9。题材里只有一只连板的杂票被冠以板块龙头高置信，经 watchlist 透传后执行侧据角色加权可能误配名额。（经确认为缺陷，非预期口径。）

## 待确认（代码事实成立，但定级依赖本仓库无法证实的外部假设）

- **U-1** watchlist 导出 `order_by(tier 字符串字典序, priority 可空)`（`routes_watchlist_export.py:120`）：tier 字典序 `CHAIN<FIRST_BOARD<HIGH_BOARD` 与业务档位序不符、priority NULL 排前。是否构成交易影响取决于执行侧是否依赖 items 数组顺序——已在联合评审核实（执行侧按字段重排，见 `26`）。
- **U-3** FOCUS 先验越界过滤只按 ts_code 不校验 tier（`limit_up_push_service.py:695-714`）：同码跨 tier 时先验贴错行。三层 board_level 分区互斥 + `code not in by_code` 已闸断软件路径，残留依赖 KPL 是否对同股返回跨板级双行。
- **U-5** ingest 信封一致性仅校验 `account_id/trade_date`，不校验 `record.table` 与 `data` 列语义匹配（`qmt_ingest_service.py:172-203`）。5 表两两错配在本仓库被 NOT NULL/唯一键缺列（422/回滚）兜底，但属附带效果非显式校验，依赖执行侧 mappers 不串栈。
- **U-6** `get_settings` 用 `@lru_cache` 缓存 Settings，token env 直配/IP 白名单/token 文件路径改动**不重启不生效**（503 滞留），`core/config.py:698`；`_FILE` 内容轮换可热生效。取决于运维是否热改不重启（本机上线流程含 `systemctl restart`）。
- **U-8** go/no-go `net_mean` 用 buyable 横截面均值、t 统计量用 portfolio 日均值，两套不同总体（`limit_up_backtest_service.py:867-887`）。合取闸门下只会更严不会错误放行，残留为 `net_mean_tstat` 命名/语义不自洽。

## 共性结论（系统性口径风险）

S-2 / S-3 / S-9 / S-10 同源指向一个共性：pipeline 取数对**数据缺口/接口 FAILED** 普遍采用「静默退化为 0/沿用旧值」而非「标记缺测」，这些 0/旧值一路落 `limit_up_selected_stock` 并经 watchlist 透传执行侧，执行侧无从感知降级。这是信号侧最值得关注的系统性口径风险。
